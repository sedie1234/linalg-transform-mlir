in-tree pass 3종 — convert-linalg-to-loops / convert-linalg-to-affine-loops / convert-linalg-to-parallel-loops (mlir/lib/Dialect/Linalg/Transforms/Loops.cpp, LLVM 19.1.7) 를 해부(A)·재현(B)·관찰(C) 한다. 추후 개인 컴파일러 이식을 위해 내부 코드 구성(호출 체인)과 IR 전후 변화를 파일:라인 단위로 파악하는 것이 목적.

이 trio 는 linalg 학습의 핵심 변환이다: buffer semantics 의 linalg op 이 갖는 계산 의미 3요소 — iterator_types(loop 종류) · indexing_maps(메모리 접근) · region(scalar 계산) — 를 명시적 loop nest + load/compute/store 로 푸는 lowering 이다. high-level 의미가 어디서 어떻게 소멸하는지를 보는 자리이기도 하다.

이 cycle 의 특이 구도: pass 3개 = template 기계 1개

세 pass 는 모두 linalgOpToLoopsImpl<LoopTy> (Loops.cpp:208-254) 의 instantiation 차이일 뿐이다 — scf::ForOp / affine::AffineForOp / scf::ParallelOp. 그래서 본인 pass 는 my-linalg-to-loops-trio 1개로 통합하고 mode=scf|affine|parallel 옵션으로 LoopTy 를 고른 뒤, byte-diff 는 모드별로 각각 대응 in-tree pass 와 비교한다.

5 결론

학습 정점 1 — linalg 의미 3요소가 loop 구조로 1:1 사상된다

iterator_types → loop 종류 (parallel 묶음/scf.for), indexing_maps → load/store index (affine.apply 전개 후 fold), region → innermost body (clone-inline). lowering 후에는 이 3요소가 IR 에서 구조로만 남고 선언적 메타데이터는 소멸 — named op 복원(#0002 specialize)이 어려워지는 이유를 코드로 확인.

학습 정점 2 — affine baseline 과의 대조: 합법성을 "믿는다 vs 증명한다"

parallel 모드는 generateParallelLoopNest 가 linalg op 에 이미 선언된 iterator_types 를 그대로 믿고 scf.parallel 을 만든다. affine 세계에서 같은 결론(이 loop 는 parallel)을 얻으려면 dependence 분석(isLoopParallel)으로 증명해야 한다 (affine-mlir #0006/#0013 경험과 대조). linalg 의 반자동 철학 — 합법성은 op 정의 시점에 고정, lowering 은 기계적 — 의 표본.

이식 체크리스트 (개인 컴파일러)

① populate* 가 없으므로 가져올 것은 linalgOpTo{Loops,AffineLoops,ParallelLoops} (Transforms.h:769-778) 3개 + 얇은 pattern 골격 + FoldAffineOp 25줄 verbatim. ② affine 모드는 load/store 까지 affine dialect 로 바뀐다 — 이후 affine 분석 패스를 이어 붙일 거면 affine 모드 출력이 진입점. ③ tensor semantics 입력은 조용히 비발화하므로 파이프라인에서 bufferization 선행을 스스로 보장해야 한다. ④ link: MLIRLinalgTransforms + MLIRSCFDialect(+MLIRArithDialect).

byte-diff 13/13 identical — 모드 3종 모두 이식 검증 완료. 후속 아이디어: (a) parallel 모드 출력의 scf.parallel 을 convert-parallel-loops-to-gpu 류로 잇는 분배 실험, (b) affine 모드 출력에 affine-mlir 의 dependence/tile 패스를 적용해 "linalg lowering → affine 최적화" 합류점 검증, (c) GenerateLoopNest 의 ProcInfo/distribution 훅 해부.
