1 무엇을 하고자 하는지

in-tree pass linalg-specialize-generic-ops (mlir/lib/Dialect/Linalg/Transforms/Specialize.cpp, LLVM 19.1.7)를 해부(A) → out-of-tree 재현(B) → IR 전후 관찰(C)의 3단계로 학습한다. 목적은 추후 개인 컴파일러에 이 pass를 이식할 수 있도록 내부 코드 구성 (pass class → driver → pattern → 핵심 함수 체인)과 변환이 만드는 IR 차이를 파일:라인 단위로 파악하는 것.

이 pass는 linalg.generic을 named op로 되돌리는 변환이다 — #0001에서 해부한 linalg-generalize-named-ops(named → generic)의 정확한 역방향. 따라서 본 cycle의 비교 질문은: generalize가 "named op가 이미 가진 정보를 노출"하는 무조건 발화 변환이었다면, specialize는 generic의 구조를 어떻게 판정해서 어떤 부분집합에만 발화하는가? 특히 idiom 인식 분기(copy/fill/elemwise/contraction)가 어디서 어떤 순서로 검사되는지가 해부의 특이 포인트.

특이 포인트 — 분기 순서가 곧 우선순위

5개 idiom 검사는 독점적 사다리다: copy → fill → unary → binary → contraction 순으로 검사하고 먼저 걸린 것이 이긴다. copy(yield 단독 + identity maps)는 fill(scalar 입력, map이 ()라 identity 검사 탈락)과 구조적으로 배타라 순서 충돌은 없지만, 검사 비용이 싼 것부터 배치된 구조. 옵션 플래그는 0개이므로 분기로 흐르는 옵션도 없다.

3 예상되는 결과

  • 같은 populate 함수 + 같은 greedy driver를 호출하므로 my-specialize-generic-ops와 linalg-specialize-generic-ops의 출력은 3개 입력 모두 byte-identical일 것.
  • positive 입력 10개 함수는 모두 대응 named op로 바뀌고 linalg.generic이 0개가 될 것. @sub_swapped는 areBinOpsSwapped=true를 타서 named op의 ins operand가 교환되어 나올 것.
  • negative 입력 4개 함수는 전부 generic으로 잔류하고 pass는 성공(재인쇄만)할 것.
  • round-trip(--my-generalize-named-ops --my-specialize-generic-ops): #0001 입력의 matmul/add는 복원되지만 transpose는 specializeGenericOp에 인식 분기가 없어 generic으로 남을 것.

함정 — specialize는 generalize의 완전한 역함수가 아니다

generalize ∘ specialize ≠ identity. linalg.transpose·linalg.broadcast· maximumf elemwise 등 인식 분기가 없는 named op는 generic으로 내려간 뒤 되돌아오지 못한다. 이식 시 "어느 idiom까지 복원해야 하는가"를 화이트리스트 (:278 exp 1종, :287-302 binary 4종, contraction 페어 3종)로 직접 결정해야 한다.

5 결론

  • 이식 검증 완료: in-tree와 동일 절차(populate + greedy)만으로 출력 byte-identical 3/3. 가져올 함수는 단 1개 — linalg::populateLinalgGenericOpsSpecializationPatterns(RewritePatternSet&) (lib MLIRLinalgTransforms). 더 세밀한 제어가 필요하면 그 아래 linalg::specializeGenericOp(RewriterBase&, GenericOp)를 직접 호출해도 된다 (transform dialect의 structured.specialize도 같은 함수를 쓴다).
  • 구조 패턴: #0001과 완전히 동일한 "최소 pattern pass" 골격 (옵션 0, pattern 1, greedy driver). 차이는 pattern 내부 — generalize는 무조건 변환, specialize는 idiom 인식 사다리(copy → fill → unary → binary → contraction, Specialize.cpp:262-309)로 인식 가능한 부분집합에만 발화.
  • 인식의 두 층위: (1) 구조 판정은 dialect 라이브러리의 linalg::isaCopyOpInterface·linalg::isaContractionOpInterface 등 isa*OpInterface 함수들(LinalgInterfaces.cpp)이 담당 — maps의 identity/projected-permutation 여부, body op 개수, linalg::inferContractionDims를 통한 contraction dims 추론. (2) op 종류 매칭은 pass 쪽 화이트리스트(exp / add·sub·mul·div / mul+add 페어)가 담당. 이식 시 (1)은 재사용, (2)는 자기 컴파일러의 named op 집합에 맞게 확장하는 구도가 자연스럽다.
  • swap 처리가 비가환 연산(sub/div)의 의미 보존 핵심: body가 op(%b, %a)면 named op의 ins를 교환해 생성 (:32-37, :58-69). IR로 직접 확인됨 (ins(%arg1, %arg0)).
  • round-trip 비대칭: generalize ∘ specialize는 matmul/add 같은 인식 idiom만 복원. 후속 후보 — transform dialect structured.specialize와의 대조, 그리고 elementwise 인식이 더 넓은 최신 LLVM의 linalg.elementwise 통합 named op 방향 조사.
