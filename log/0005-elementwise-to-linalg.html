1 무엇을 하고자 하는지

in-tree pass convert-elementwise-to-linalg 를 (A) 해부 — pass class → driver → pattern → 핵심 함수의 호출 체인을 파일:라인 단위로 추적, (B) 재현 — 같은 populate 함수 (populateElementwiseToLinalgConversionPatterns) + 같은 driver 를 out-of-tree 에서 호출하는 my-elementwise-to-linalg 를 만들어 byte-diff 로 이식 가능성 검증, (C) 관찰 — 발화/비발화 입력으로 IR 전후 변화를 코드 단계와 매핑한다. 추후 개인 컴파일러에 이 pass 를 이식하기 위한 사전 학습.

pass 의 기능: ElementwiseMappable trait 을 가진 op 이 ranked tensor operand 위에서 동작하면 linalg.generic (identity indexing map × all-parallel) 으로 치환. #0001~#0004 가 linalg 내부의 named↔generic 정규화였다면, 이 pass 는 linalg 바깥 (arith/math 등) 의 tensor-level 연산을 linalg 세계로 끌어들이는 진입 관문이다.

이번 cycle 의 해부 질문

① driver 가 greedy 인가 dialect-conversion 인가 — 그리고 ConversionTarget 은 무엇으로 구성되는가? ② isElementwiseMappableOp 판정의 정확한 내용은 무엇이며, 변환 대상 op "목록" 은 어디에 있는가?

해부 질문 ① 의 답 — 본 cycle 첫 dialect-conversion pass

#0001~#0004 는 모두 applyPatternsAndFoldGreedily 였는데 이 pass 는 ConversionTarget + applyPartialConversion (:131, :139). target 은 named legality 등록 없이 markUnknownOpDynamicallyLegal 단 한 줄 (:135-137) — "모든 unknown op 은 elementwise-mappable-on-ranked-tensors 가 아닐 때만 legal". 즉 illegal 집합 = 변환 대상 집합. partial conversion 은 illegal op 만 root 로 잡아 pattern 을 적용하고, 끝났는데 illegal 이 남으면 pass 가 실패한다. 이 pass 에선 pattern 의 match 전제(:81)와 legality(:135)가 정확히 같은 술어의 부정 관계라 pattern 이 성공하는 한 잔존 illegal 은 구조적으로 불가능 — "변환 대상이 0개가 될 때까지" 라는 종료 상태의 선언이 공짜 안전망으로 따라온다.

3 예상되는 결과

  • 빌드 — 보일러플레이트 구조는 #0001~#0004 와 동일하나 driver 가 DialectConversion 으로 바뀜. applyPartialConversion 은 MLIRTransformUtils (기존 MLIRTransforms 의 public 의존) 에 있으므로 MLIRMemRefDialect 만 선제 추가하면 1회 green 예상.
  • byte-diff 0 — runOnOperation 절차가 in-tree 와 한 줄 한 줄 동일 (같은 pattern 1개, 같은 target 구성, 같은 partial conversion, 옵션 없음). 유일한 리스크는 직접 재구성한 legality 술어 2줄인데 원본과 줄 단위 동일이므로 세 입력 모두 byte-identical 예상.
  • IR 변화 — static: tensor op 5개가 각각 linalg.generic 으로, cmpf 만 tensor.empty 동반. dynamic: cmpf 의 empty 에 tensor.dim×2 + arith.constant×2 추가. negative 3종 불변.
  • 불확실 포인트 — ① addf 처럼 result type == operand type 인 op 의 DPS init 이 정말 입력 operand 그대로 outs 에 등장하는지 (그럼 SSA 상 입력이 outs 에 재등장하는 특이한 모양) ② dialect-conversion driver 가 greedy 와 달리 후처리 fold 를 안 하므로 출력에 잔여 정리가 없는 그대로의 모양일지.

C. negative — 불변 확인

output.negative.mlir 에서 ① scalar arith.addf (f32 operand — all_of 실패) ② vector arith.addf (vector<4xf32> — Vectorizable trait 이 있어도 ranked tensor 가 아님) ③ arith.select %cond_i1_scalar, %t, %t (operand 하나라도 scalar 면 탈락 — 소스 :28-29 의 TODO "any_of 일반화는 미구현" 케이스) 모두 op 구조 그대로. 세 케이스가 술어 :24-31 의 두 가드 (trait, all_of) 와 1:1 대응.

함정 — "변환 대상 목록" 을 찾으려 하지 말 것

이 pass 에는 op allowlist 가 없다. MatchAnyOpTypeTag pattern + trait 판정이라, 어떤 dialect 든 ElementwiseMappable.traits (= Elementwise + Scalarizable + Vectorizable + Tensorizable, Operation.cpp:1393-1396) 를 단 op 이면 전부 대상이다 (arith·math·complex 등). 개인 컴파일러 이식 시 커스텀 dialect op 도 이 trait 4개만 달면 코드 수정 없이 자동으로 linalg 로 진입한다 — 반대로 trait 를 빠뜨리면 조용히 변환에서 빠진다.

5 결론

  • 해부 요약: 옵션 0개·pattern 1개 (ConvertAnyElementwiseMappableOpOnRankedTensors, ~35줄)·헬퍼 2개 (술어 2줄 + outputs 채우기 ~27줄) 의 소형 pass. 본질은 "trait 4개 + 전 operand ranked tensor → identity map × parallel 의 linalg.generic 으로 감싸고 body 에 같은 이름의 scalar op 재생성".
  • driver 답안 (해부 질문 ①): 본 cycle 첫 dialect-conversion pass — ConversionTarget 을 markUnknownOpDynamicallyLegal 한 줄 (술어의 부정) 로 구성하고 applyPartialConversion (:139). pattern 전제와 legality 가 같은 술어를 공유해 "illegal = 변환 대상 = 변환 가능" 이 일치 — 종료 상태 선언 + 미달 시 pass 실패라는 안전망이 greedy 대비 추가로 따라온다. 후처리 fold 가 없어 출력이 날 것 그대로인 점도 greedy 와의 관찰 가능한 차이.
  • 판정 답안 (해부 질문 ②): op 목록이 아니라 trait — OpTrait::hasElementwiseMappableTraits (Operation.cpp:1393-1396, 4 trait conjunction) + operand 전부 RankedTensorType (all_of). scalar 혼합 broadcast 는 :28-29 TODO 로 명시적 미지원.
  • DPS init 채우기의 두 분기: result type 과 같은 operand 가 있으면 그 operand 를 init 으로 재사용 (입력이 outs 에 재등장하는 pure init), 없으면 (cmpf 의 i1) tensor.empty + dynamic dim 은 첫 operand 에서 tensor.dim 으로 추출 (tensor::getMixedSizes, :69).
  • 이식 검증: 같은 populate + 같은 target 구성 + 같은 driver 호출로 3/3 입력 byte-identical. file-static 이라 export 안 되는 legality 술어 2줄만 exported building block 으로 동일 구성 — 이식 시에도 pattern·헬퍼 2개·술어를 한 세트로 가져가고, pattern 전제를 바꾸면 target legality 도 같이 바꿔야 한다.
  • 후속 아이디어: ① linalg-fuse-elementwise-ops 해부 — 이 pass 가 만든 generic 5개 사슬이 어떻게 1개로 합쳐지는지 (자연스러운 다음 단계 파이프라인) ② one-shot-bufferize 와 연결해 pure init (outs 에 재등장한 입력) 이 buffer 세계에서 in-place 로 처리되는지 관찰.
