1 무엇을 하고자 하는지

in-tree pass linalg-named-op-conversion 을 (A) 해부 — pass class → driver → pattern → 핵심 함수의 호출 체인을 파일:라인 단위로 추적, (B) 재현 — 같은 populate 함수 + 같은 driver 를 out-of-tree 에서 호출하는 my-named-op-conversion 을 만들어 byte-diff 로 이식 가능성 검증, (C) 관찰 — 발화/비발화 입력으로 IR 전후 변화를 코드 단계와 매핑한다. 추후 개인 컴파일러에 이 pass 를 이식하기 위한 사전 학습.

pass 의 기능: named op → named op 정규화. #0001(named→generic)·#0002(generic→named)와 달리 추상화 수준을 바꾸지 않고, 더 일반적인 named op 를 같은 의미의 더 좁은 named op 로 바꾼다. LLVM 19.1.7 시점의 실체는 depthwise conv 2D 한 가족 — channel multiplier 차원(M)이 정적으로 1 인 depthwise_conv_2d_nhwc_hwcm(_q) 를 depthwise_conv_2d_nhwc_hwc(_q) 로 좁히는 것.

이번 cycle 의 해부 질문

"named → named 변환" 의 대상 op 목록은 무엇이고, rank 가 고정된 named op 사이에서 차원 수가 다른 변환(5-D 출력 → 4-D 출력)을 어떻게 합법적으로 성립시키는가?

핵심 답 — rank 고정 named op 사이의 변환은 "reshape 샌드위치"

named op 는 layout 별 rank 가 ODS/YAML 에 고정돼 있어 (hwcm: kernel 4-D·출력 5-D / hwc: kernel 3-D·출력 4-D) op 교체만으로는 타입이 안 맞는다. 그래서 입력측은 tensor.collapse_shape 로 M=1 차원을 흡수해 들어가고, 결과측은 tensor.expand_shape 로 원래 5-D 타입을 복원해 나온다 — 변환 전후 함수 시그니처·사용처 타입이 전혀 안 바뀌는 국소 정규화. M=1 일 때만 collapse 가 무손실이므로 getDimSize(3) == 1 이 유일한 발화 조건 코어다.

함정 — pattern 코드에 없는 op 가 IR 에 나온다: ExpandShapeOp builder 의 산물

NamedOpConversions.cpp 어디에도 tensor.dim·arith.constant 생성 코드가 없는데 dynamic 케이스의 after IR 에는 4쌍이 등장한다. :98 의 replaceOpWithNewOp<tensor::ExpandShapeOp>(op, resultTy, src, reassociation) 가 쓰는 reassociation-only builder (TensorOps.cpp:1687-1701) 가 getMixedSizes → inferOutputShape (:1663-1674) 로 output_shape 를 채우면서 dynamic 차원마다 tensor.dim(+index 상수) 을 materialize 하기 때문. #0003 의 "상수 hoist 는 driver 의 몫" 과 짝이 되는 교훈 — op builder 도 IR 을 만든다. 이식 시 pattern 코드만 읽고 출력 IR 을 예단하지 말 것.

5 결론

  • 해부 요약: 옵션 0개·pattern 2개·greedy driver 1줄의 최소 pattern pass. 본질은 static 함수 matchAndReplaceDepthwiseConv 약 65줄 — "가드 3개(tensor semantics·ranked·M==1) → kernel/init collapse → TypeSwitch 로 좁은 named op 생성 + discardable attr 보존 → expand 로 원 타입 복원".
  • 대상 op 목록: 거창한 이름과 달리 19.1.7 에선 정확히 2종 — depthwise_conv_2d_nhwc_hwcm→..._hwc, ..._hwcm_q→..._hwc_q. multiplier=1 전용이며 판정은 정적 shape 기준.
  • rank 불일치 해법: named op 의 고정 rank 때문에 op 교체만으로는 불가 — collapse_shape(입력측)/expand_shape(결과측) 샌드위치로 외부 타입을 보존한 채 내부만 좁힌다. dependentDialects 에 tensor 가 명시된 이유.
  • 이식 검증: 같은 populate + 같은 driver 호출만으로 3/3 입력 byte-identical. 개인 컴파일러에는 핵심 함수 1개(~65줄) + 얇은 pattern 2개와 그 의존 (named op 4종 builder, tensor::CollapseShapeOp/ExpandShapeOp, getPrunedAttributeList) 만 가져가면 된다.
  • builder 의 몫: dynamic 케이스의 tensor.dim/arith.constant 는 pattern 이 아니라 ExpandShapeOp 의 reassociation-only builder 가 만든 것 (TensorOps.cpp:1687-1701). driver(#0003)·builder(#0004) 모두 pattern 코드 밖에서 IR 을 만든다 — 재현·이식 시 한 세트로 가져가야 byte-identical 이 나온다.
  • 후속 아이디어: ① linalg-fold-unit-extent-dims 해부 — 같은 "size-1 차원 제거" 를 generic 수준에서 일반화한 인접 pass 와 대조 ② shape inference 류 pass 를 앞단에 둬 dynamic M 을 정적 1 로 굳혔을 때 발화 범위가 넓어지는지 파이프라인 실험.
