1 무엇을 하고자 하는지

in-tree pass linalg-inline-scalar-operands 를 (A) 해부 — pass class → driver → pattern → 핵심 함수의 호출 체인을 파일:라인 단위로 추적, (B) 재현 — 같은 populate 함수 + 같은 driver 를 out-of-tree 에서 호출하는 my-inline-scalar-operands 를 만들어 byte-diff 로 이식 가능성 검증, (C) 관찰 — 발화/비발화 입력으로 IR 전후 변화를 코드 단계와 매핑한다. 추후 개인 컴파일러에 이 pass 를 이식하기 위한 사전 학습.

pass 의 기능: linalg.generic 의 DPS input 중 indexing map 이 상수인 operand("scalar operand")를 op 의 operand/indexing_maps 목록에서 제거하고, body 안 tensor.extract 로 끌어들인다(inline). loop iteration 과 무관한 read 를 인터페이스에서 치워 generic 을 단순화하는 변환이다.

해부의 중심 API 두 개: 변환 진입점인 linalg::populateInlineConstantOperandsPatterns (단일 pattern InlineScalarOperands 를 등록) 와, 'scalar' 판정의 실제 술어인 AffineMap::isConstant. inline 시 인덱스를 뽑는 AffineMap::getConstantResults, tensor semantics 게이트인 linalg::GenericOp::hasPureTensorSemantics 가 보조한다. greedy driver 는 applyPatternsAndFoldGreedily.

이번 cycle 의 해부 질문

어떤 operand 가 'scalar' 로 인정되는가 — rank-0 인가, size-1 인가, 아니면 다른 기준인가? 그리고 indexing_maps 와 body 는 정확히 어떤 절차로 재작성되는가?

핵심 답 — 'scalar' 판정은 rank-0 도 size-1 도 아닌 "map 의 상수성"

판정 함수는 AffineMap::isConstant (AffineMap.cpp:377-379) = llvm::all_of(getResults(), llvm::IsaPred<AffineConstantExpr>). 모든 result 가 상수 affine expr 이면 scalar. 두 부류가 걸린다:

(a) rank-0 tensor tensor<f32> — map (d0) -> () 는 result 0개. empty range 의 all_of 는 vacuous true → scalar 인정, tensor.extract %t[] (인덱스 0개) 로 inline.
(b) 상수 인덱스 접근 — map (d0,d1) -> (0,1) 처럼 iteration 변수와 무관한 고정 위치 읽기. getConstantResults() 가 [0,1] 을 주고 tensor.extract %t[%c0, %c1] 로 inline.

즉 tensor 의 shape(rank/size-1 여부)이 아니라 접근 패턴(indexing map) 이 기준이다. size-1 tensor 라도 map 이 (d0) -> (d0) 면 비후보, 큰 tensor 라도 map 이 (d0) -> (2) 면 후보.

함정 — pattern 산출물 ≠ pass 산출물: 상수의 최종 위치는 driver 가 정한다

pattern 코드는 setInsertionPointToStart(body) 후 arith::ConstantIndexOp 를 만들므로(:71, :78-80) 상수가 generic body 안에 생긴다. 그러나 최종 IR 에서 %c0/%c1 은 func entry 에 있다 — applyPatternsAndFoldGreedily 의 fold/constant materialization 단계가 상수를 가장 가까운 isolated-from-above 영역 선두로 끌어올리기 때문. pattern 을 단독 적용(예: transform dialect 의 apply_patterns 외 직접 호출)했을 때와 pass 결과가 다를 수 있는 표본. 이식 시 driver 까지 한 세트로 가져가야 in-tree 와 동일한 IR 이 나온다.

5 결론

  • 해부 요약: 옵션 0개·pattern 1개·greedy driver 1줄의 최소 pattern pass. 본질은 InlineScalarOperands::matchAndRewrite 약 55줄 — "상수 map input 골라내기 → 그것 뺀 새 GenericOp → body 복제 → scalar 를 tensor.extract 로 대체 + block arg 삭제(역순)".
  • 'scalar' 의 정의: tensor shape 이 아니라 AffineMap::isConstant — 모든 map result 가 상수 (rank-0 의 empty map 포함, vacuous true). DPS init 은 제외, tensor semantics 전용.
  • 이식 검증: 같은 linalg::populateInlineConstantOperandsPatterns + 같은 driver 호출만으로 3/3 입력 byte-identical. 개인 컴파일러에는 pattern 1개(자기완결 ~60줄)와 그 의존(GenericOp DPS API, AffineMap::isConstant/AffineMap::getConstantResults, tensor::ExtractOp, arith::ConstantIndexOp)만 가져가면 된다.
  • driver 의 몫: 상수 hoist 는 pattern 이 아니라 greedy driver 의 fold 단계 산물 — pass 와 동일한 결과를 원하면 pattern 만이 아니라 driver 동작까지 묶어 이식해야 한다 (#0001·#0002 와 일관된 교훈).
  • 후속 아이디어: ① linalg-fold-unit-extent-dims 해부 — size-1 차원을 다루는 인접 pass 와 '상수 map' 기준의 차이 대조 ② elementwise fusion 과 연계해 inline 후 fusion 이 쉬워지는 사례 관찰.
