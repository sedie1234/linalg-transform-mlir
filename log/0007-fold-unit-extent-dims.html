1 무엇을 하고자 하는지

in-tree pass linalg-fold-unit-extent-dims (구현: mlir/lib/Dialect/Linalg/Transforms/DropUnitDims.cpp, LLVM 19.1.7) 를 해부(A)·재현(B)·관찰(C) 한다. 추후 개인 컴파일러 이식을 위해 (1) pass class → driver → populate → pattern → 핵심 함수의 호출 체인을 파일:라인 단위로 추적하고, (2) in-tree 가 export 하는 populate 함수들을 out-of-tree 에서 그대로 가져와 호출하는 my-fold-unit-extent-dims 를 만들어 byte-diff 로 이식 정확성을 검증하고, (3) 대표 입력으로 "코드 단계 ↔ IR 변화" 매핑을 기록한다.

이 pass 의 기능: broadcasting 표현용 unit-extent dimension (size 1) 을 linalg op 의 operand 와 iteration space 에서 제거한다. 예컨대 tensor<1x5xf32> 를 상수-0 으로 접근하는 generic 을 tensor<5xf32> 직접 접근으로 정규화한다. 특이 포인트로 옵션 use-rank-reducing-slices 가 RankReductionStrategy 를 갈라 collapse/expand 의 op 종류와 패턴 목록이 어떻게 달라지는지 추적한다.

학습 정점 1 — unit dim 판정은 indexing map 의 역사상

affine 처럼 trip count 를 분석하는 게 아니라, inversePermutation(concatAffineMaps(maps)) 로 "iteration dim d 가 어느 operand 의 어느 자리에 닿는가"를 대수적으로 얻고 그 자리의 static size 가 1 이면 d 는 one-trip loop 로 판정한다. 제거도 대수적 — d 를 AffineConstantExpr(0) 으로 치환하면 map 재작성·operand collapse 가 기계적으로 따라온다. affine #0007(unroll)·#0013(tile) 의 도메인 분석과 대비되는 linalg 의 "구조가 곧 분석" 철학.

학습 정점 2 — RankReductionStrategy 는 같은 수학, 다른 표현

옵션 use-rank-reducing-slices 가 가르는 유일한 분기는 runOnOperation :827-830 의 options.rankReductionStrategy 설정. 이후 (1) populate 선택(:798-808)과 (2) collapseValue/expandValue 내부의 op 종류가 전부 여기서 갈린다. Reshape = collapse_shape/expand_shape (메타데이터 변환, copy 없음) ↔ Slice = extract_slice/insert_slice (복사 의미론, bufferization 친화). 또 Slice 모드 populate(:782-796) 에는 RankReducedExtractSliceOp/RankReducedInsertSliceOp 3종과 CollapseShape/ExpandShape canonicalization 이 빠진다.

함정 — 이 파일의 모든 패턴이 이 pass 에 들어가는 건 아니다

같은 DropUnitDims.cpp 끝의 populateContractionOpRankReducingPatterns (:1067-1097, matmul→vecmat / batch_matmul→matmul 등 named contraction 의 unit dim 강하 — RankReduceMatmul/RankReduceToUnBatched) 는 runOnOperation 에 포함되지 않는다 (transform dialect 등 다른 경로 전용). 이식 시 pass 가 실제로 부르는 populate 2개만 가져가면 된다.

negative.mlir (unit dim·상수0 접근·pad·slice 없음) 은 pass 없이 my-mlir-opt 만 통과시킨 출력과 완전 동일 (diff 0). 코드 경로: dropUnitDims 는 newIndexingMaps == indexingMaps → failure(:489-493), MoveInitOperandsToInput 은 body 가 %out 을 안 읽어 candidates 공집합 → failure(:100-101).

5 결론

  • 이식 검증 완료: populate 2개(populateFoldUnitExtentDimsPatterns, populateMoveInitOperandsToInputPattern) + greedy driver 만 동일 순서로 호출하면 8/8 byte-identical. 개인 컴파일러 이식 시 가져갈 것은 이 둘과 ControlDropUnitDims 옵션 구조체뿐이다 (같은 파일의 contraction rank-reducing 패턴 묶음은 이 pass 소관이 아님).
  • 본체는 단일 함수: 패턴 DropUnitDims 는 linalg::dropUnitDims(:389-550) 로의 위임 한 줄이고, 핵심 아이디어는 indexing map 역사상으로 one-trip dim 을 찾아 AffineConstantExpr(0) 치환 + collapse/expand 샌드위치로 제거하는 것. controlFn 훅으로 어느 dim 을 떨굴지 호출자가 선별 가능 (기본 = 전부).
  • use-rank-reducing-slices 의 실체: RankReductionStrategy enum 하나가 (a) collapse/expand 의 op 종류 (reshape ↔ slice)와 (b) populate 되는 패턴 집합(RankReduced*SliceOp 3종 유무) 을 동시에 가른다. 기본 reshape 모드가 metadata-only 라 보통 우월하고, slice 모드는 reshape 를 못 견디는 백엔드/bufferization 경로용.
  • 본 cycle 첫 옵션 pass 의 PassWrapper 레시피: Option<bool> 직접 선언 + base-forwarding copy ctor — tablegen PassBase 없이도 CLI --my-pass=opt=val 문법이 그대로 동작함을 확인.
  • 후속 아이디어: (i) linalg-detensorize 해부 — rank-0 까지 떨어진 tensor 의 scalar 화와 본 pass 의 접점, (ii) #0006 fusion 과 연계 — unit-reduction 케이스가 elementwise 로 변한 뒤 실제로 fuse 되는지 파이프라인 실험, (iii) populateContractionOpRankReducingPatterns 를 쓰는 transform dialect 경로 추적.
