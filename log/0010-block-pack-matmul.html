1 무엇을 하고자 하는지

in-tree pass linalg-block-pack-matmul (mlir/lib/Dialect/Linalg/Transforms/BlockPackMatmul.cpp, LLVM 19.1.7) 를 해부(A)·재현(B)·관찰(C) 한다. 추후 개인 컴파일러 이식을 위해 내부 코드 구성(호출 체인)과 IR 전후 변화를 파일:라인 단위로 파악하는 것이 목적.

이 pass 는 본 cycle 첫 data-layout 변환 pass 다. 지금까지의 pass 들이 op 의 표현(generalize/specialize)이나 iteration 공간(fold-unit-dims, fuse) 을 바꿨다면, 이 pass 는 matmul 의 피연산자 메모리 배치 자체를 2-level blocked layout 으로 재배열한다: 2D MxK 행렬이 4D [MB][KB][mb][kb] 가 되고 (major 블록 × minor 블록), 계산은 기본 옵션에서 linalg.mmt4d 와 같은 접근 패턴 [MB][NB][mb][nb] += [MB][KB][mb][kb] * [NB][KB][nb][kb] 의 packed generic 이 된다.

특이 포인트 2개를 코드로 확인한다

① 옵션 8개(block-factors 등)가 ControlBlockPackMatmulFn closure 를 거쳐 linalg::blockPackMatmul 까지 흐르는 경로. ② 이전 관찰에서 "block-factors 미지정이면 no-op" 이었던 것의 정확한 코드 근거 (어느 라인의 어느 검사가 전체를 끄는가).

3 예상되는 결과

  • byte-diff: 같은 populate 함수 + 같은 greedy driver + 같은 옵션 기본값이므로 8개 combo 전부 byte-identical 예상.
  • bf.matmul: tensor.pack ×3 + 6D linalg.generic (par,par,red,par,par,red) + tensor.unpack ×1. RHS pack 에 outer_dims_perm=[1,0] inner_dims_pos=[1,0] (기본 rhs-transpose 효과), LHS/출력은 identity. named linalg.matmul 은 generic 으로 소멸 (packMatmulGreedily 의 generalizeNamedOp).
  • bf16.pad-matmul: 30 % 16 ≠ 0 → 모든 pack 에 padding_value(0.0) + 2x2x16x16 (30→32 올림).
  • no-op 3종 (noopts / nopad / negative-memref): 출력 = 입력 roundtrip, tensor.pack 0개.
  • generic-transpose-b: B 가 이미 [N][K] 라 자연 packing 이 곧 목표 layout → packTranspose 가 identity perm 으로 끝나 재transpose 없음.

구조의 정수: "큰 transform 함수 1개 + 정책 closure 1개"

pattern 7개는 전부 op-종류 필터일 뿐이고 (named 6종은 무조건 위임, generic 만 contraction + 3-map 검사), 실제 결정 — 블록 크기·padding 허용· transpose 여부 — 은 ControlBlockPackMatmulFn 이 op 마다 돌려주는 BlockPackMatmulOptions 가 전담한다. 이식 시 controlFn 만 갈아끼우면 op 별 휴리스틱(예: 행렬 크기에 따라 다른 block factor, 특정 op 는 nullopt 로 제외)을 pass 재컴파일 없이 줄 수 있는 설계다.

  • 변환은 기존 빌딩블록 3개의 합성: linalg::packMatmulGreedily (= inferContractionDims + generalizeNamedOp + interchangeGenericOp + linalg::pack) → inferContractionDims → transposePackedMatmul×2 (= linalg::packTranspose). BlockPackMatmul.cpp 자체에는 새 알고리즘이 거의 없고 Transforms.cpp 의 범용 pack 기계를 matmul 용으로 조립한 것.
  • transpose 옵션의 의미는 "절대 layout 목표": 코드는 operandMap 에서 현재 layout 의 transposed 여부를 판정(:107-110)해 옵션과 다를 때만 perm={1,0} 을 적용한다. 그래서 이미 [N][K] 인 matmul_transpose_b 형 입력은 기본 옵션에서 재transpose 없이 통과 — 옵션은 "뒤집어라"가 아니라 "최종적으로 이 layout 이 되게 하라"다.
  • byte-diff 8/8 identical — populateBlockPackMatmulPatterns + applyPatternsAndFoldGreedily driver + 옵션 8개(이름·기본값 동일) 재현만으로 이식 완료. link lib 추가 0개 (MLIRLinalgTransforms + MLIRTensorDialect 기존분으로 충분).
  • affine 대비: affine #0013 의 tile 은 iteration 공간 분할(loop nest 재구성)이고 data 는 그대로다. 이 pass 는 반대로 iteration 공간 분할(6D generic)과 data 재배치(tensor.pack)를 동시에 IR 로 명시한다 — polyhedral 파이프라인에는 대응물이 없는, linalg 의 "layout 을 1급 시민으로" 접근. 후속 lowering 에서 pack/unpack 이 실제 복사가 될지(materialize) 폴딩될지는 별도 pass 의 몫.
  • 후속 아이디어: ① linalg.mmt4d named op 와의 관계 — packed generic 을 specialize 하면 mmt4d 가 되는지 (#0002 연계). ② tensor.pack/unpack 의 lowering (linalg::lowerPack, Transforms.h:1126) 해부. ③ batch matmul 입력으로 batch dim 보존 경로(:123-128 offsetPerms) 관찰.
