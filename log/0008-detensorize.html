1 무엇을 하고자 하는지

in-tree pass linalg-detensorize (mlir/lib/Dialect/Linalg/Transforms/Detensorize.cpp, LLVM 19.1.7) 를 해부(A)·재현(B)·관찰(C) 한다. 추후 개인 컴파일러 이식을 위해 내부 코드 구성(호출 체인)과 IR 전후 변화를 파일:라인 단위로 파악하는 것이 목적.

Detensoring 이란: tensor 기반 프런트엔드(TOSA/HLO 류)는 loop counter·분기 조건 같은 제어용 스칼라까지 0-d tensor (tensor<i32>)에 박제해 내려보낸다. 이 pass 는 그런 0-d tensor 값과 그 위의 linalg.generic 을 primitive scalar(i32)와 scalar 연산으로 끌어내리고, 그 값을 나르는 CFG(block argument / branch operand)의 type 까지 함께 강하한다.

본 cycle(#0001~#0007 누적) 관점에서 이 pass 가 새로 보여주는 것:

  • 유일한 InterfacePass — anchor 가 op 이름이 아니라 FunctionOpInterface. pass 스케줄링이 어떻게 달라지는가.
  • DialectConversion 기계 전체 세트 — #0005 의 applyPartialConversion 을 넘어, TypeConverter + source/target/argument materialization + applyFullConversion + signature conversion 까지.
  • cost model 의 분리 — "어떻게 바꾸나"(pattern)와 "무엇을 바꾸나"(cost model 객체)가 코드 레벨에서 분리된 첫 사례. 기본 ControlFlowDetectionModel vs 옵션 aggressive-mode 의 AggressiveDetensoringModel.
  • populate* 함수가 없는 pass 의 이식 — 핵심 pattern 이 전부 file-local 일 때 out-of-tree 재현을 어떻게 구성하나.

2.4 A — 특이점 (이식 시 주의)

학습 정점 1 — entry block 보호 트릭 (:484-490 ↔ :570-573)

dialect conversion 의 signature 변환이 entry block 을 건드리면 함수 type 이 깨진다. 그래서 pass 시작 시 splitBlock(entryBlock, begin()) + cf.BranchOp 생성으로 본문 전체를 dummy non-entry block 으로 밀어내고, legality (:514-523) 도 drop_begin(body, 1) 로 entry 를 제외하며, 끝에 eraseOp(branch) + mergeBlocks 로 복원한다. 함수 경계(외부 호출 규약)는 절대 detensor 하지 않는다는 보수적 설계가 코드 구조로 강제된 것. detensorize_entry_block.mlir 이 이 트릭의 회귀 테스트.

함정 1 — dependentDialects = [] 인데 cf.br 을 생성한다

Passes.td:104 는 빈 리스트지만 runOnOperation 은 cf::BranchOp(:490), tensor::ExtractOp/FromElementsOp(materialization)를 생성한다. mlir-opt 류에서 안 깨지는 이유: registerAllExtensions → func::registerInlinerExtension 의 extension 이 FuncDialect 로드 시 cf::ControlFlowDialect 를 같이 로드해 주기 때문 (InlinerExtension.cpp:83-89). 개인 컴파일러 이식 시 이 우연에 기대면 미로드 dialect 의 op 생성으로 fatal error — 이식판 MyDetensorize.cpp 는 getDependentDialects 에 cf/tensor 를 정직하게 선언했다 (출력 IR 영향 없음, 로드 시점만 보장).

함정 2 — InterfacePass 는 module 위에 직접 못 올린다

my-mlir-opt --linalg-detensorize input.mlir 은 unable to schedule pass 'LinalgDetensorizePass' on a PassManager intended to run on 'builtin.module' 에러. 반드시 -pass-pipeline="builtin.module(func.func(linalg-detensorize))" 로 anchoring 해야 한다 (in-tree 테스트들의 RUN 라인도 전부 이 형태).

2.5 B — out-of-tree 재현 (MyDetensorize.cpp)

이 pass 는 #0001~#0007 과 달리 자기 전용 populate* 함수가 없다 — 핵심 pattern/cost model 이 전부 Detensorize.cpp 의 익명 namespace(file-local) class 라 export 되지 않는다. 따라서 재현을 두 부분으로 구성:

  • (a) export 된 building block 은 #include 로 가져와 호출: populateBranchOpInterfaceTypeConversionPattern / isNotBranchOpInterfaceOrReturnLikeOp / isLegalForReturnOpTypeConversionPattern (FuncConversions.h:43-70, lib MLIRFuncTransforms), driver 인 applyFullConversion + applyPatternsAndFoldGreedily, tensor::FromElementsOp::getCanonicalizationPatterns, TypeConverter/OpConversionPattern 인프라.
  • (b) export 안 된 file-local 조각은 줄 단위 동일 이식(verbatim port, 알고리즘 재작성 없음): sourceMaterializationCallback, canBeDetensored, shouldBeDetensored, DetensorizeGenericOp, FunctionNonEntryBlockConversion, DetensorizeTypeConverter, CostModel 2종, runOnOperation 본문 = Detensorize.cpp:31-574.

pass class 는 PassWrapper<MyDetensorizePass, InterfacePass<FunctionOpInterface>> — in-tree 와 같은 interface anchor. 옵션 aggressive-mode 도 같은 이름·기본값(false)으로 노출. 4-edit 통합:

3 예상되는 결과

  • byte-diff = 0 예상: runOnOperation 절차를 줄 단위로 동일하게 구성했으므로 모든 입력·모드에서 in-tree 와 출력이 byte-identical 할 것. 단 file-local 코드의 이식 누락(예: cost model 의 사후 blockArg 제외 :412-445)이 있으면 blockArg 집합이 달라져 분기 type 이 어긋날 위험.
  • while.mlir: blockArg 와 branch operand 가 i32 로 강하되고 generic 2개가 arith.cmpi/arith.addi 로 inline, 경계에만 tensor.extract(입구)/tensor.from_elements(출구)가 남을 것.
  • pure-compute.mlir: 기본 모드 무변화(seed 없음 → opsToDetensor 비음), aggressive 에서만 scalar 화 — 두 cost model 의 차이가 IR 로 직접 드러나는 대조쌍.
  • rank1-negative.mlir: rank 1 → canBeDetensored false → 양 모드 무변화.
  • entry block 트릭([1]→[8])이 비발화 입력에서도 항상 수행되므로, "비발화 = 입력 그대로" 가 깨지지 않는지(완전 복원되는지) 관찰 필요.

4.3 IR 전후 변화 — pure-compute.mlir (cost model 대조쌍)

기본 모드 출력 = 입력 그대로 (entry 트릭 [1]→[8] 완전 복원 확인). aggressive 모드만 발화:

before = 기본 모드 after (비발화 — seed 없음)
func.func @detensorable_but_no_cf(
    %arg0: tensor<f32>, %arg1: tensor<f32>) -> tensor<f32> {
  %0 = tensor.empty() : tensor<f32>
  %1 = linalg.generic {…, iterator_types = []}
    ins(%arg0, %arg1 : tensor<f32>, tensor<f32>)

aggressive-mode after — 경계에만 tensor 잔존
func.func @detensorable_but_no_cf(
    %arg0: tensor<f32>, %arg1: tensor<f32>) -> tensor<f32> {
  %extracted = tensor.extract %arg1[] : tensor<f32>
  %extracted_0 = tensor.extract %arg0[] : tensor<f32>

기본 모드 비발화의 코드 근거: ControlFlowDetectionModel 의 seed 는 cf.cond_br/cf.br operand (:262-268) — 이 함수에 branch 가 없어 workList 가 빈 채 시작 → opsToDetensor = {} → target 기준 모든 op legal → 변환 0.

4.4 rank1-negative.mlir — "비발화 ≠ 출력 무변화" 관찰

rank 1 → canBeDetensored :51-53 false → 양 모드에서 generic/branch type 불변. 단 출력이 입력과 완전히 같지는 않았다: ^bb1(%x: tensor<4xf32>) 의 인자가 사라지고 %arg0 직사용으로 바뀜. 이는 detensorize 의 변환이 아니라 [7] applyPatternsAndFoldGreedily 에 기본 탑재된 region simplification (단일 값만 받는 redundant blockArg 제거)의 효과 — 이 pass 는 비발화 시에도 끝에 greedy 정리 단계를 항상 돌린다. (in-tree 와 동일하게 발생하므로 byte-diff 는 여전히 0.)

학습 정점 2 — materialization 은 "경계 보정용 임시 op"

변환 중간에는 tensor↔scalar 가 어긋나는 모든 자리에 tensor.extract(target)/tensor.from_elements(source/argument)가 꽂히지만, [7] greedy 의 ExtractOp::fold(extract(from_elements(x))→x, TensorOps.cpp:1147-1165)가 내부 쌍을 전부 상쇄해 최종 IR 에는 함수 경계(입구 extract / 출구 from_elements)에만 남는다. cost model 의 탐색(:390-398)도 from_elements 를 "나중에 fold 될 것"으로 전제하고 통과시킨다 — pass 의 두 단계([6] conversion, [7] cleanup)가 서로를 가정하는 설계.

5 결론

  • 이식 검증 완료: my-detensorize 가 22/22 케이스에서 in-tree 와 byte-identical. populate 함수가 없는 pass 는 (a) export 된 building block 호출 + (b) file-local 조각의 verbatim port 조합으로 재현 가능함을 확인 — 개인 컴파일러 이식 시 Detensorize.cpp:31-574 를 통째로 가져가되 FuncConversions/driver 는 라이브러리 함수로 쓰면 된다.
  • pass 의 골격: ① entry block 보호(splitBlock+cf.br) → ② cost model 로 "무엇을 풀지" 결정 → ③ 결정을 그대로 legality 로 번역한 ConversionTarget + pattern 3종(연산/blockArg/branch operand 의 3층 분담) → ④ applyFullConversion → ⑤ greedy 로 materialization 쌍 상쇄 → ⑥ entry 복원. "결정(cost model)과 실행(conversion)의 분리 + full conversion 으로 결정의 완전 이행 강제"가 설계의 핵심.
  • DialectConversion 기계의 완전판 학습: TypeConverter 의 type 규칙 + 3종 materialization, SignatureConversion 에 의한 blockArg 재작성, 동적 legality 의 3계층(특정 op / interface / unknown) 구성까지 — #0005 partial conversion 의 상위 호환 사례.
  • 이식 체크리스트에 추가: (1) InterfacePass 는 pipeline anchoring 필요 (func.func(…)); (2) dependentDialects=[] 는 in-tree 의 함정 — cf/tensor 를 직접 선언할 것; (3) 비발화 입력도 region simplification 으로 출력이 변할 수 있음(greedy 후처리가 무조건 돌므로).
  • 후속 아이디어: scf.if/scf.while 위의 detensoring 은 이 pass 가 못 한다 (cf 전용 seed :262-268, pred 가 BranchOpInterface 아니면 전면 포기 :339-345) — scf→cf 강하 후 적용하는 파이프라인 위치 제약을 별도 work 으로 관찰할 만함. 또 0-d 제약(:51-53)을 1-element tensor 로 일반화하는 TODO (:48)도 코드에 남아 있다.
