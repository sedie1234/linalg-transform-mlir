1 무엇을 하고자 하는지

in-tree pass linalg-generalize-named-ops (구현: mlir/lib/Dialect/Linalg/Transforms/Generalization.cpp, LLVM 19.1.7) 를 세 단계로 학습한다 — 추후 개인 컴파일러에 이 pass 를 이식할 수 있도록 내부 코드 구성과 IR 변화를 파악하는 것이 목적이다.

  • A. 해부 — pass class → driver → populate → pattern → 핵심 함수의 호출 체인을 파일:라인 단위로 추적.
  • B. 재현 — out-of-tree pass my-generalize-named-ops 가 in-tree 와 같은 populate 함수 + 같은 driver 를 가져와 호출 (알고리즘 재구현 금지), 출력 byte-diff 로 이식 정확성 검증.
  • C. 관찰 — 발화/비발화 입력으로 before/after IR 을 캡처하고 "코드 단계 ↔ IR 변화" 를 매핑.

이 pass 가 첫 표본인 이유

linalg-generalize-named-ops 는 가장 단순한 pattern pass 다 — 옵션 0개, pattern 1개, driver 호출 한 줄. 앞으로 해부할 모든 pattern pass (runOnOperation → populate*Patterns → applyPatterns* 골격)의 기준 표본이 된다.

LinalgGeneralizationPattern 과 generalizeNamedOp 의 관계

pattern (Transforms.h:1408-1421) 은 얇은 어댑터다 — matchAndRewrite 는 linalg::generalizeNamedOp 한 줄 호출이 전부. 변환 로직은 standalone 함수에 있고, pattern 은 그것을 greedy driver 에 끼우는 포장이다 (transform dialect 의 structured.generalize 도 같은 함수를 부른다). 또 OpRewritePattern<matmul> 식의 op 별 pattern 이 아니라 OpInterfaceRewritePattern<LinalgOp> — 인터페이스 match 라서 matmul/add/transpose/fill/conv 등 모든 named op 를 pattern 1개로 처리한다.

populate 정의 위치 함정

linalg::populateLinalgNamedOpsGeneralizationPatterns 의 선언은 Transforms/Transforms.h:1588 이지만 정의는 별도 Patterns 파일이 아니라 pass 구현 파일 자신 (Generalization.cpp:95-98) 에 있다. 링크 시 MLIRLinalgTransforms 가 필요한 이유.

핵심 통찰 — generalization 은 "노출"이지 "계산"이 아니다

named op 는 이미 generic 과 동일한 정보(indexing_maps · iterator_types · payload region)를 내부에 갖고 있다. 이 pass 는 그 정보를 새로 만들지 않고 getter 로 꺼내 속성으로 펼칠 뿐이며 region 조차 inlineRegionBefore 로 이동만 한다. affine 의 polyhedral 모델이 IR 에서 iteration domain 을 역추출해야 했던 것과 정반대 — linalg 는 정보를 구조 자체에 보존한다.

5 결론

  • pattern pass 의 최소 골격 확립 — RewritePatternSet → populate*(patterns) → applyPatternsAndFoldGreedily 3줄. 이후 cycle 의 모든 pattern pass 해부에서 이 골격 대비 무엇이 추가되는지만 보면 된다.
  • 이식 체크리스트 (개인 컴파일러 기준): ① Transforms/Transforms.h include, ② MLIRLinalgTransforms + MLIRTransforms 링크, ③ dependentDialects 에 linalg::LinalgDialect, ④ anchor 는 any-op (특정 func/module 에 묶을 필요 없음). pattern 을 거치지 않고 generalizeNamedOp(rewriter, op) 를 직접 불러도 된다 (transform dialect 가 그렇게 쓴다).
  • byte-diff 3/3 identical — "같은 populate + 같은 driver" 재현 전략이 이 pass 계열에서 유효함을 확인. 이 검증 패턴을 cycle 표준으로 유지.
  • 주의점 2개: linalg.map 은 named op 이지만 generalize 대상이 아니다 (precondition bail — region arg 구성이 다름). populate 함수의 정의가 Patterns 전용 파일이 아니라 pass .cpp 안에 있을 수 있다.
  • 후속 아이디어: 역방향 pass linalg-specialize-generic-ops (Passes.td:97) 해부 — generalize 가 "노출"이라면 specialize 는 generic 에서 named op 를 인식해야 하므로 비대칭적으로 어려울 것. 대조 표본으로 적합.
