Linalg in-tree pass 해부 카탈로그

LLVM 19.1.7 mlir/include/mlir/Dialect/Linalg/Passes.td의 pass 12종 전수 — pass class → driver → pattern → 핵심 함수의 호출 체인(파일:라인), 옵션이 흐르는 분기, IR 전후 변화, 그리고 개인 컴파일러에 이식할 때 무엇을 #include·호출·링크해야 하는가. 모든 pass는 out-of-tree my-mlir-opt에서 같은 함수를 호출해 재현했고, in-tree 출력과의 byte-diff(합산 69/69 identical)가 이식 가능성의 증명이다.

이 카탈로그를 읽는 법

각 pass 섹션은 같은 골격이다: ① 호출 체인 — runOnOperation()에서 핵심 함수까지 파일:라인 단위, ② 옵션 흐름 — CLI 옵션이 코드 어느 분기를 가르는지, ③ 핵심 함수 시그니처 — out-of-tree에서 가져다 쓸 수 있는 export 표면, ④ IR 전후 — 실측 before/after 발췌와 "어느 코드가 어느 변화를 만들었나" 주석, ⑤ 이식 가이드 — #include·호출 절차·link lib·함정. 급할 때는 ⑤만 읽어도 pass를 옮길 수 있도록 ⑤는 자기완결로 썼다.

#0001 linalg-generalize-named-ops — named op → generic (정보 노출)

pass class LinalgGeneralizeNamedOpsPass — Generalization.cpp:79-85 def Passes.td:92-95 · anchor 없음 · dependentDialects=[linalg] 옵션 0개 greedy byte-diff 3/3

모든 named op(matmul/add/transpose/fill…)를 linalg.generic으로 푼다. 핵심 통찰: generalization은 정보를 계산하지 않는다 — named op가 이미 내부에 보유한 indexing_maps·iterator_types·payload region을 노출할 뿐이다 (region조차 rewriter.inlineRegionBefore로 이동, 재생성 아님). 진입점은 linalg::populateLinalgNamedOpsGeneralizationPatterns로 패턴을 채우고 applyPatternsAndFoldGreedily로 고정점까지 돌리는 것, 실제 변환은 linalg::generalizeNamedOp 한 함수가 담당한다. pattern pass의 최소 해부 표본.

카탈로그 횡단 인사이트

  • IR은 pattern 코드 밖에서도 생긴다 — driver의 fold/hoist(#0003), op builder의 부수 생성(#0004 ExpandShapeOp의 tensor.dim×4), driver의 region simplification(#0008 negative). byte-identical 재현 = pattern + driver + builder를 한 세트로 가져갈 때만 성립.
  • 합법성 / 비용 / 변환의 3분리가 linalg transform 설계의 골격: areElementwiseOpsFusable(합법) vs ControlFusionFn(비용) vs fuseElementwiseOps(변환). polyhedral(완전 자동, ILP)과 달리 비용 결정이 명시적 훅으로 노출된다.
  • linalg op는 자기 의미를 선언으로 보유 — generalize는 노출(#0001), lowering은 선언 신뢰(#0009 parallel이 iterator_types를 분석 없이 믿음). affine의 의존성 분석 기반 접근(Allen-Kennedy)과의 본질 차이.
  • negative 케이스의 bail 지점(파일:라인)을 알아야 "왜 안 변하지"를 즉답할 수 있다 — 각 섹션의 negative 행이 그 목록.
