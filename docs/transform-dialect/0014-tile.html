T04 — tile (scf.for vs scf.forall)

같은 linalg.matmul 한 줄을 두 가지 tile op으로 쪼갠다 — tile_using_for는 순차 scf.for nest, tile_using_forall는 병렬 scf.forall. 그리고 num_threads vs tile_sizes는 같은 payload를 낳는다.

Q이 실험은 뭘 보여주나?

tiling은 큰 연산을 작은 타일 단위 반복으로 쪼개는, 모든 고성능 커널의 기본 변환이다. transform dialect에는 tiling op이 두 종류 있고, 이 실험은 그 둘이 만드는 루프 형태가 어떻게 다른지를 한 payload로 대조한다.

tile_using_for

tiling 축마다 순차 scf.for를 만든다. reduction 축(K)도 타일링 가능 → 3중 nest. iter_args로 부분합을 누적.

tile_using_forall

tiling 축을 병렬 scf.forall 하나로 만든다. shared_outs + parallel_insert_slice로 충돌 없는 동시 쓰기. parallel 축만 타일링.

num_threads vs tile_sizes

tile_using_forall의 두 입력 방식. 한쪽은 thread 수를, 다른 쪽은 타일 크기를 준다. 이 실험은 둘이 같은 결과를 내는 케이스를 잡았다.

payload는 공통이다 — (128×256) × (256×64) → (128×64) matmul 한 줄. tiling 차원은 M=128, N=64, K=256. 타일 크기 32를 쓰면 M/32=4, N/32=2, K/64=4로 떨어진다(완전 분할).

func.func @matmul(%A: tensor<128x256xf32>, %B: tensor<256x64xf32>,
                  %C: tensor<128x64xf32>) -> tensor<128x64xf32> {
  %0 = linalg.matmul ins(%A, %B : tensor<128x256xf32>, tensor<256x64xf32>)
                     outs(%C : tensor<128x64xf32>) -> tensor<128x64xf32>
  return %0 : tensor<128x64xf32>
}
Q어떻게 했나? (schedule)

세 schedule 모두 흐름은 같다 — match로 linalg.matmul handle을 잡고 → tile op을 적용 → yield. 차이는 어떤 tile op을·어떤 인자로 호출하느냐다. 핵심은 op이 돌려주는 result arity(handle 개수)가 다르다는 점.

A — tile_using_for, tile_sizes [32, 32, 64]. 세 축(M, N, K) 모두 타일링 → 루프 handle 3개(%loops:3). result = (tiled_op, loop_M, loop_N, loop_K).

transform.named_sequence @__transform_main(%arg0: !transform.any_op {transform.readonly}) {
  %matmul = transform.structured.match ops{["linalg.matmul"]} in %arg0
    : (!transform.any_op) -> !transform.any_op
  %tiled, %loops:3 = transform.structured.tile_using_for %matmul tile_sizes [32, 32, 64]
    : (!transform.any_op) -> (!transform.any_op, !transform.any_op, !transform.any_op, !transform.any_op)
  transform.yield
}

B — tile_using_forall, num_threads [4, 2]. M축 4-way, N축 2-way로 분할. result = (forall_op, tiled_op) — 루프 handle은 scf.forall 하나뿐이다. K(reduction)축은 타일링하지 않는다 — forall은 parallel 축만 쪼갠다.

transform.named_sequence @__transform_main(%arg0: !transform.any_op {transform.readonly}) {
  %matmul = transform.structured.match ops{["linalg.matmul"]} in %arg0
    : (!transform.any_op) -> !transform.any_op
  %forall, %tiled = transform.structured.tile_using_forall %matmul num_threads [4, 2]
    : (!transform.any_op) -> (!transform.any_op, !transform.any_op)
  transform.yield
}

C — tile_using_forall, tile_sizes [32, 32]. 같은 forall op이지만 thread 수 대신 타일 크기를 준다. thread 수는 ceildiv(dim, tile)로 역산: M = ceildiv(128,32) = 4, N = ceildiv(64,32) = 2. 즉 B와 같은 (4, 2) iteration space가 나온다.

transform.named_sequence @__transform_main(%arg0: !transform.any_op {transform.readonly}) {
  %matmul = transform.structured.match ops{["linalg.matmul"]} in %arg0
    : (!transform.any_op) -> !transform.any_op
  %forall, %tiled = transform.structured.tile_using_forall %matmul tile_sizes [32, 32]
    : (!transform.any_op) -> (!transform.any_op, !transform.any_op)
  transform.yield
}

실행: mlir-opt --transform-interpreter --canonicalize sched-*.mlir 후 transform.named_sequence 블록을 제거해 payload-only 결과를 남긴다.

arity가 schedule 의도를 드러낸다. A의 %loops:3은 "축 3개를 순차 루프로", B/C의 단일 %forall은 "parallel 축들을 forall 하나로". 후속 schedule에서 이 handle을 다시 받아 unroll/distribute 등을 걸 수 있으므로, 몇 개가 나오는지는 단순 출력이 아니라 API 계약이다.
Q결과가 어떻게 다른가? (before→after 실측)

가장 대조적인 두 쌍을 본다. 먼저 A(scf.for) — matmul 한 줄이 3중 순차 루프 nest가 된다. K축까지 타일링됐으므로 가장 안쪽 루프가 %c0 to %c256 step %c64(K=256/64=4회)이고, 부분합이 iter_args/scf.yield로 바깥 텐서에 누적된다.

A — tile_using_for [32,32,64] → scf.for ×3
before — payload (matmul 한 줄)
func.func @matmul(%A: tensor<128x256xf32>, %B: tensor<256x64xf32>,
                  %C: tensor<128x64xf32>) -> tensor<128x64xf32> {
  %0 = linalg.matmul ins(%A, %B : tensor<128x256xf32>, tensor<256x64xf32>)
                     outs(%C : tensor<128x64xf32>) -> tensor<128x64xf32>
  return %0 : tensor<128x64xf32>
}
after — out-a-for.mlir
func.func @matmul(%arg0: tensor<128x256xf32>, %arg1: tensor<256x64xf32>, %arg2: tensor<128x64xf32>) -> tensor<128x64xf32> {
  %c256 = arith.constant 256 : index
  %c64 = arith.constant 64 : index
  %c0 = arith.constant 0 : index
  %c128 = arith.constant 128 : index
  %c32 = arith.constant 32 : index
  %0 = scf.for %arg3 = %c0 to %c128 step %c32 iter_args(%arg4 = %arg2) -> (tensor<128x64xf32>) {
    %1 = scf.for %arg5 = %c0 to %c64 step %c32 iter_args(%arg6 = %arg4) -> (tensor<128x64xf32>) {
      %2 = scf.for %arg7 = %c0 to %c256 step %c64 iter_args(%arg8 = %arg6) -> (tensor<128x64xf32>) {
        %extracted_slice = tensor.extract_slice %arg0[%arg3, %arg7] [32, 64] [1, 1] : tensor<128x256xf32> to tensor<32x64xf32>
        %extracted_slice_0 = tensor.extract_slice %arg1[%arg7, %arg5] [64, 32] [1, 1] : tensor<256x64xf32> to tensor<64x32xf32>
        %extracted_slice_1 = tensor.extract_slice %arg8[%arg3, %arg5] [32, 32] [1, 1] : tensor<128x64xf32> to tensor<32x32xf32>
        %3 = linalg.matmul ins(%extracted_slice, %extracted_slice_0 : tensor<32x64xf32>, tensor<64x32xf32>) outs(%extracted_slice_1 : tensor<32x32xf32>) -> tensor<32x32xf32>
        %inserted_slice = tensor.insert_slice %3 into %arg8[%arg3, %arg5] [32, 32] [1, 1] : tensor<32x32xf32> into tensor<128x64xf32>
        scf.yield %inserted_slice : tensor<128x64xf32>
      }
      scf.yield %2 : tensor<128x64xf32>
    }
    scf.yield %1 : tensor<128x64xf32>
  }
  return %0 : tensor<128x64xf32>
}

다음 B(scf.forall) — 같은 payload가 병렬 루프 하나가 된다. 인덱스 공간이 (%arg3, %arg4) in (4, 2)로 평탄화되고, 출력 텐서는 shared_outs로 공유되며 각 타일 쓰기는 scf.forall.in_parallel { tensor.parallel_insert_slice }로 충돌 없이 들어간다. K축이 없다 — A처럼 안쪽 K-루프 대신, inner matmul이 tensor<32x256xf32> × tensor<256x32xf32>(K=256 전체)를 한 번에 계산한다.

B — tile_using_forall num_threads [4,2] → scf.forall ×1
before — A의 scf.for ×3 (참고: for의 모습)
%0 = scf.for %arg3 = %c0 to %c128 step %c32 iter_args(%arg4 = %arg2) -> (tensor<128x64xf32>) {
  %1 = scf.for %arg5 = %c0 to %c64 step %c32 iter_args(%arg6 = %arg4) -> (...) {
    %2 = scf.for %arg7 = %c0 to %c256 step %c64 iter_args(%arg8 = %arg6) -> (...) {
      ... linalg.matmul (32x64 x 64x32) ...
      %inserted_slice = tensor.insert_slice %3 into %arg8 ...
      scf.yield %inserted_slice
    } ... scf.yield ...
  } ... scf.yield ...
}
return %0
after — out-b-forall-numthreads.mlir
#map = affine_map<(d0) -> (d0 * 32)>
func.func @matmul(%arg0: tensor<128x256xf32>, %arg1: tensor<256x64xf32>, %arg2: tensor<128x64xf32>) -> tensor<128x64xf32> {
  %0 = scf.forall (%arg3, %arg4) in (4, 2) shared_outs(%arg5 = %arg2) -> (tensor<128x64xf32>) {
    %1 = affine.apply #map(%arg3)
    %2 = affine.apply #map(%arg4)
    %3 = affine.apply #map(%arg3)
    %4 = affine.apply #map(%arg4)
    %extracted_slice = tensor.extract_slice %arg0[%1, 0] [32, 256] [1, 1] : tensor<128x256xf32> to tensor<32x256xf32>
    %extracted_slice_0 = tensor.extract_slice %arg1[0, %2] [256, 32] [1, 1] : tensor<256x64xf32> to tensor<256x32xf32>
    %extracted_slice_1 = tensor.extract_slice %arg5[%3, %4] [32, 32] [1, 1] : tensor<128x64xf32> to tensor<32x32xf32>
    %5 = linalg.matmul ins(%extracted_slice, %extracted_slice_0 : tensor<32x256xf32>, tensor<256x32xf32>) outs(%extracted_slice_1 : tensor<32x32xf32>) -> tensor<32x32xf32>
    %6 = affine.apply #map(%arg3)
    %7 = affine.apply #map(%arg4)
    scf.forall.in_parallel {
      tensor.parallel_insert_slice %5 into %arg5[%6, %7] [32, 32] [1, 1] : tensor<32x32xf32> into tensor<128x64xf32>
    }
  }
  return %0 : tensor<128x64xf32>
}

그리고 B vs C — num_threads [4,2]와 tile_sizes [32,32]는 이 payload에서 byte-identical한 결과를 낸다(run.sh의 diff가 IDENTICAL 보고). 둘 다 (4, 2) forall에 #map = (d0) -> (d0 * 32) 같은 offset 매핑이다. 차이는 입력 표현이지 출력이 아니다 — dim이 tile로 정확히 나눠떨어지기 때문.

  ...

Q그래서 핵심은?

tile op 선택은 "어떤 루프 구조를 원하나"의 결정이다. 같은 "32×32 타일"이라도 for냐 forall이냐에 따라 후속 lowering·실행 모델이 갈린다.

축tile_using_for (A)tile_using_forall (B/C)
생성 루프scf.for 순차 nest (축마다 1개)scf.forall 1개 (parallel 축 묶음)
실험 결과 nest3중 (M, N, K)단일 (M, N) — K 없음
reduction 축(K)타일링 가능 (안쪽 K-루프 + iter_args 누적)타일링 안 함 (parallel만) → inner가 K 전체
출력 누적iter_args + insert_slice + scf.yieldshared_outs + parallel_insert_slice
result arity(tiled, loop×3)(forall, tiled)
실행 의미순차 (정해진 순서)병렬 (충돌 없는 동시 쓰기)
오프셋 표현루프 induction var 직접 (%arg3 등)affine.apply #map(d0)=(d0*32)

for = 순차 + reduction 친화

K축까지 쪼개 작은 GEMM으로 누적하고 싶을 때. cache blocking·register tiling처럼 reduction을 명시적으로 다루는 schedule에 맞다.

forall = 병렬 매핑의 입구

이후 thread/block/GPU grid에 매핑하는 distribution의 출발점. shared_outs가 동시 쓰기 안전성을 IR 레벨에서 보장.

num_threads ≡ tile_sizes (분할 시)

dim이 tile로 정확히 나눠떨어지면 두 입력은 같은 forall을 낳는다. 표현만 다른 동일 변환 → 어느 쪽으로 schedule을 써도 무방.

Q함정/주의는?
  • forall은 reduction 축을 타일링하지 않는다. num_threads [4, 2]는 인자가 2개라 M·N 두 parallel 축만 받는다. K를 forall로 쪼개려 하면 partial reduction 처리(tile_reduction_using_forall 같은 별도 op)가 필요하다 — 단순 tile_using_forall로는 안 된다.
  • result arity를 타입 시그니처와 정확히 맞춰야 한다. A는 루프 3개라 (!transform.any_op)×4(tiled + loop×3), B/C는 (!transform.any_op)×2(forall + tiled). 개수가 어긋나면 interpreter가 verify 단계에서 거부한다. forall은 루프가 1개뿐임을 잊고 %loops:2 식으로 받으면 실패.
  • "num_threads = tile_sizes" 동치는 균등 분할일 때만. dim이 tile로 나눠떨어지지 않으면 두 방식의 경계 처리(나머지 타일)가 달라질 수 있다. 이 실험은 128/32, 64/32가 떨어지는 깔끔한 케이스라 IDENTICAL이 나온 것. 일반화 금지.
  • --canonicalize가 IR 모습을 바꾼다. 출력의 정리된 형태(중복 affine.apply, 상수 hoist 등)는 canonicalize 후 모습이다. raw 변환 직후 IR을 보려면 canonicalize를 빼고 비교해야 한다.
Q한 줄 + 관련

한 줄: tile은 "타일 크기"만의 문제가 아니라 "루프 종류(for 순차 vs forall 병렬)"의 선택이고, forall에서 num_threads와 tile_sizes는 균등 분할 시 같은 IR을 낳는다.

관련. P1(in-tree pass 해부)의 linalg-tile 패스가 같은 효과를 C++ 패스 옵션으로 강제하던 것과 대조 — 여기선 같은 tiling을 schedule IR로 명시하고 for/forall을 op 선택으로 고른다. affine baseline의 tile(#0013) 산출 IR과도 비교 가능: polyhedral은 ILP로 자동 결정, 여기선 tile op·인자를 손으로 지정.

다음. forall 결과를 받아 thread/block에 매핑하는 distribute, for nest를 받아 inner matmul을 vectorize하는 schedule — 둘 다 이 실험에서 나온 loop handle을 입력으로 받는다. tile은 그 체인의 첫 단계다.
