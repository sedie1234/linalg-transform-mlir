같은 변환을 두 방식으로 — 고정 패스 vs transform script

같은 변환(generalize: named op → linalg.generic)을 ① 고정 패스(-linalg-generalize-named-ops)와 ② transform script(match+generalize)로 각각 적용해, 무엇이 같고 무엇이 다른지 실측으로 본다.

"그냥 패스 고치거나 옵션 주면 되지 않나?" — 이 문서는 언제 script가 의미 있고 언제 패스가 나은지 경계까지 정직하게 짚는다. IR은 전부 mlir-opt 실측(experiments/0012).

입력(payload) — named op 2개

한 함수에 linalg.fill(초기화)과 linalg.matmul(행렬곱)이 같이 있다. "이 둘을 generic으로 풀어라"가 같은 변환의 목표.

func.func @two_ops(%A: tensor<64x128xf32>, %B: tensor<128x64xf32>, %init: tensor<64x64xf32>) -> tensor<64x64xf32> {
%c0 = arith.constant 0.0 : f32
%filled = linalg.fill ins(%c0 : f32) outs(%init : tensor<64x64xf32>) -> tensor<64x64xf32>
%0 = linalg.matmul ins(%A, %B : tensor<64x128xf32>, tensor<128x64xf32>)
                   outs(%filled : tensor<64x64xf32>) -> tensor<64x64xf32>
return %0 : tensor<64x64xf32>
Q같은 'generalize'인데, 두 방식이 뭔데?

① 고정 패스 — mlir-opt -linalg-generalize-named-ops. 네가 0001에서 해부한 그 패스. 모듈을 훑어 모든 named op을 linalg.generic으로 바꾼다. 노브는 없다(타겟 지정 불가).

② transform script — 변환을 IR로 적어 -transform-interpreter로 적용. "matmul만 골라(match) generalize"처럼 대상을 지정한다:

transform.named_sequence @__transform_main(%arg0: !transform.any_op {transform.readonly}) {
  %mm = transform.structured.match ops{["linalg.matmul"]} in %arg0
          : (!transform.any_op) -> !transform.any_op
  %g = transform.structured.generalize %mm : (!transform.any_op) -> !transform.any_op
  transform.yield
}
둘 다 같은 변환 *기계*를 쓴다. script가 generalize를 재구현하는 게 아니라, 패스가 부르는 그 generalizeOp(0001의 그것)를 똑같이 부른다 — 차이는 "어디에 적용하나"의 제어다.
Q그래서 결과가 어떻게 다른데? (실측)

같은 payload에 두 방식을 돌린 결과를 나란히 — "변경 강조"를 켜면 갈라지는 줄이 색칠된다.

같은 payload, 두 방식의 결과
① 고정 패스 → 둘 다 generic (fill·matmul 모두)

#map1 = affine_map<(d0, d1) -> (d0, d1)>
#map2 = affine_map<(d0, d1, d2) -> (d0, d2)>
#map3 = affine_map<(d0, d1, d2) -> (d2, d1)>
#map4 = affine_map<(d0, d1, d2) -> (d0, d1)>
func.func @two_ops(%arg0: tensor<64x128xf32>, %arg1: tensor<128x64xf32>, %arg2: tensor<64x64xf32>) -> tensor<64x64xf32> {
  %cst = arith.constant 0.000000e+00 : f32
  %0 = linalg.generic {indexing_maps = [#map, #map1], iterator_types = ["parallel", "parallel"]} ins(%cst : f32) outs(%arg2 : tensor<64x64xf32>) {
  ^bb0(%in: f32, %out: f32):
    linalg.yield %in : f32
  } -> tensor<64x64xf32>
  %1 = linalg.generic {indexing_maps = [#map2, #map3, #map4], iterator_types = ["parallel", "parallel", "reduction"]} ins(%arg0, %arg1 : tensor<64x128xf32>, tensor<128x64xf32>) outs(%0 : tensor<64x64xf32>) {
  ^bb0(%in: f32, %in_0: f32, %out: f32):
    %2 = arith.mulf %in, %in_0 : f32
    %3 = arith.addf %out, %2 : f32
    linalg.yield %3 : f32
  } -> tensor<64x64xf32>
  return %1 : tensor<64x64xf32>
}

#map1 = affine_map<(d0, d1, d2) -> (d2, d1)>
#map2 = affine_map<(d0, d1, d2) -> (d0, d1)>
func.func @two_ops(%arg0: tensor<64x128xf32>, %arg1: tensor<128x64xf32>, %arg2: tensor<64x64xf32>) -> tensor<64x64xf32> {

  %0 = linalg.fill ins(%cst : f32) outs(%arg2 : tensor<64x64xf32>) -> tensor<64x64xf32>
  %1 = linalg.generic {indexing_maps = [#map, #map1, #map2], iterator_types = ["parallel", "parallel", "reduction"]} ins(%arg0, %arg1 : tensor<64x128xf32>, tensor<128x64xf32>) outs(%0 : tensor<64x64xf32>) {
  ^bb0(%in: f32, %in_0: f32, %out: f32):

    %3 = arith.addf %out, %2 : f32

  } -> tensor<64x64xf32>
  return %1 : tensor<64x64xf32>
}
payload
fill + matmul
-linalg-generalize-named-ops
전부 generic (fill→generic, matmul→generic) — 타겟팅 불가
match matmul → generalize
matmul만 generic, linalg.fill 그대로 — 정밀 타겟

핵심: 변환 자체는 같은데, 고정 패스는 "전부 아니면 전무", script는 "고른 것만"이다.

Q그래서 script가 뭐가 나은데?

① 정밀 타겟팅

Q2 그대로 — 모듈 전체가 아니라 고른 op에만. 큰 모델에서 "이 matmul만 다르게 처리" 가 가능.

② 조합(compose)

한 레시피 안에서 match → tile → generalize를 순서대로 엮는다. 패스 파이프라인은 순서가 암묵 의존성에 묶이고 타겟도 안 되는데, script는 명시적.

③ 재빌드 0회 swap

B(타겟 generalize)와 C(tile+generalize)는 같은 바이너리로, script 텍스트만 바꿔 나온 것. 컴파일러를 안 고친다.

②의 실제 — match matmul → tile [16,16,16] → generalize를 한 script로 (fill은 손 안 댐):

%mm = transform.structured.match ops{["linalg.matmul"]} in %arg0 : ...
%t, %l:3 = transform.structured.tile_using_for %mm tile_sizes [16, 16, 16] : ...
%g = transform.structured.generalize %t : ...   // 한 레시피로 match→tile→generalize
compose: 한 script로 matmul만 tile + generalize
before — payload (fill + matmul)
func.func @two_ops(%A: tensor<64x128xf32>, %B: tensor<128x64xf32>, %init: tensor<64x64xf32>) -> tensor<64x64xf32> {
%c0 = arith.constant 0.0 : f32
%filled = linalg.fill ins(%c0 : f32) outs(%init : tensor<64x64xf32>) -> tensor<64x64xf32>
%0 = linalg.matmul ins(%A, %B : tensor<64x128xf32>, tensor<128x64xf32>)
                   outs(%filled : tensor<64x64xf32>) -> tensor<64x64xf32>
return %0 : tensor<64x64xf32>
after — fill 보존 + matmul은 scf.for 타일 nest 안 generic
#map = affine_map<(d0, d1, d2) -> (d0, d2)>
#map1 = affine_map<(d0, d1, d2) -> (d2, d1)>
#map2 = affine_map<(d0, d1, d2) -> (d0, d1)>
func.func @two_ops(%arg0: tensor<64x128xf32>, %arg1: tensor<128x64xf32>, %arg2: tensor<64x64xf32>) -> tensor<64x64xf32> {
  %cst = arith.constant 0.000000e+00 : f32
  %0 = linalg.fill ins(%cst : f32) outs(%arg2 : tensor<64x64xf32>) -> tensor<64x64xf32>
  %c0 = arith.constant 0 : index
  %c64 = arith.constant 64 : index
  %c16 = arith.constant 16 : index
  %1 = scf.for %arg3 = %c0 to %c64 step %c16 iter_args(%arg4 = %0) -> (tensor<64x64xf32>) {
    %c0_0 = arith.constant 0 : index
    %c64_1 = arith.constant 64 : index
    %c16_2 = arith.constant 16 : index
    %2 = scf.for %arg5 = %c0_0 to %c64_1 step %c16_2 iter_args(%arg6 = %arg4) -> (tensor<64x64xf32>) {
      %c0_3 = arith.constant 0 : index
      %c128 = arith.constant 128 : index
      %c16_4 = arith.constant 16 : index
      %3 = scf.for %arg7 = %c0_3 to %c128 step %c16_4 iter_args(%arg8 = %arg6) -> (tensor<64x64xf32>) {
        %extracted_slice = tensor.extract_slice %arg0[%arg3, %arg7] [16, 16] [1, 1] : tensor<64x128xf32> to tensor<16x16xf32>
        %extracted_slice_5 = tensor.extract_slice %arg1[%arg7, %arg5] [16, 16] [1, 1] : tensor<128x64xf32> to tensor<16x16xf32>
        %extracted_slice_6 = tensor.extract_slice %arg8[%arg3, %arg5] [16, 16] [1, 1] : tensor<64x64xf32> to tensor<16x16xf32>
        %4 = linalg.generic {indexing_maps = [#map, #map1, #map2], iterator_types = ["parallel", "parallel", "reduction"]} ins(%extracted_slice, %extracted_slice_5 : tensor<16x16xf32>, tensor<16x16xf32>) outs(%extracted_slice_6 : tensor<16x16xf32>) {
        ^bb0(%in: f32, %in_7: f32, %out: f32):
          %5 = arith.mulf %in, %in_7 : f32
          %6 = arith.addf %out, %5 : f32
          linalg.yield %6 : f32
        } -> tensor<16x16xf32>
        %inserted_slice = tensor.insert_slice %4 into %arg8[%arg3, %arg5] [16, 16] [1, 1] : tensor<16x16xf32> into tensor<64x64xf32>
        scf.yield %inserted_slice : tensor<64x64xf32>
      }
      scf.yield %3 : tensor<64x64xf32>
    }
    scf.yield %2 : tensor<64x64xf32>
  }
  return %1 : tensor<64x64xf32>
}
같은 바이너리, 다른 script → 다른 결과 (B: 타겟 generalize / C: 타겟 tile+generalize). 이게 "schedule = 데이터"의 실측이다.
Q근데 그냥 패스에 옵션 주면 되지 않나?

정직하게 — 경우에 따라 맞다. 하지만 패스 옵션은 미리 정의된 노브 안에서만 움직인다. 예로 0010 -linalg-block-pack-matmul은 옵션이 8개나 있지만, "이 matmul만 / op마다 다른 블록 / tile 먼저 하고 pack"은 그 8개에 없으면 컴파일러를 고쳐야 한다. script는 그걸 텍스트로 표현한다.

하고 싶은 것고정 패스(+옵션)transform script
모듈 전체에 똑같이 변환간단 — 이게 더 낫다가능하나 과함
특정 op만 / op별 다르게옵션에 없으면 컴파일러 수정match로 지정
여러 변환을 정해진 순서로 엮기패스 순서·암묵 의존한 레시피에 명시
수백 변형 sweep (autotuning)재빌드 반복 불가script 생성·sweep
경계 한 줄: 한 변환을 모듈 전체에 똑같이 = 고정 패스가 낫다. 타겟·조합·탐색이 필요해지는 순간 script. (CUDA 손튜닝 vs 컴파일러/autotuning 논의와 같은 결.)

Q한 줄로?
같은 변환이라도, 고정 패스는 "전부에 똑같이"(타겟·조합·탐색 불가)이고 transform script는 "고른 것에·엮어서·재빌드 없이 갈아끼우며" 적용한다. 변환 *기계*는 공유하고, script는 그 위의 제어·조합·탐색 층이다.
