T10 — full kernel schedule (match→tile→fuse)

하나의 named_sequence로 fc+bias+relu 3-op 커널을 match → tile → fuse → vectorize 단계로 점진적으로 내려, payload 한 함수가 어떻게 단일 scf.forall 타일 루프 + vector.contract 코드젠 직전 형태까지 변형되는지 한 줄로 추적한다.

Q이 실험은 뭘 보여주나?

지금까지의 transform 실험은 단일 op 하나를 tile하거나(T다른 실험들) 하나의 변환을 격리해서 봤다. T10은 여러 transform op을 한 schedule 안에서 엮어 "커널 한 채(full kernel)"를 코드젠 직전까지 몰아가는 흐름을 보여준다.

payload는 작은 신경망 레이어 한 칸이다 — fully-connected(matmul) → bias 더하기(elemwise add) → ReLU(elemwise max with 0). linalg op 3개가 producer→consumer로 사슬을 이룬다:

%matmul = linalg.matmul                          // fc
%biased = linalg.elemwise_binary<add>       // + bias
%relued = linalg.elemwise_binary<max_signed> // relu = max(x, 0)

같은 binary(mlir-opt --transform-interpreter)에 schedule script만 점점 키우며 3개 stage를 찍는다:

A — tile

마지막 relu(max) op 하나만 tile_using_forall [8,32]. matmul·add는 루프 밖에 그대로. fusion 없는 baseline.

B — +fuse

A의 forall 안으로 producer를 끌어들임. fuse_into_containing_op로 add → matmul 순서로 fuse. 세 op이 한 루프 안 tile 단위로 모임.

C — +vectorize

B의 함수 전체를 vectorize_children_and_apply_patterns. tile 위 linalg op이 vector.transfer_read/contract/transfer_write로 내려감 — codegen lowering target.

핵심 관전 포인트: 매 stage가 이전 schedule 위에 한 줄씩만 추가된다는 점, 그리고 그 한 줄이 IR을 얼마나 크게 바꾸는지다.

Q어떻게 했나? (schedule)

3 stage 모두 같은 골격이다 — transform.structured.match로 payload op에 handle을 잡고, split_handle로 두 elemwise op(add/max)을 분리한 뒤 점점 더 많은 변환을 건다.

Stage A (tile만): elemwise 둘을 match → split → max 하나만 forall로 tile.

%ew = transform.structured.match ops{["linalg.elemwise_binary"]} in %arg0
    : (!transform.any_op) -> !transform.any_op
%add, %max = transform.split_handle %ew
    : (!transform.any_op) -> (!transform.any_op, !transform.any_op)
%tiled, %loop = transform.structured.tile_using_forall %max tile_sizes [8, 32]
    : (!transform.any_op) -> (!transform.any_op, !transform.any_op)
transform.yield

split_handle은 "elemwise_binary" 매치로 잡힌 2개 op handle을 IR 등장 순서대로 갈라 %add(먼저 나온 add), %max(나중 max)로 준다. tile은 consumer인 %max에만 건다.

Stage B (tile → fuse): A에 matmul match를 추가하고, tile이 만든 %loop(=scf.forall handle) 안으로 producer를 fuse한다. 순서가 중요 — consumer에 가까운 producer부터(add 먼저, 그 다음 matmul).

%mm = transform.structured.match ops{["linalg.matmul"]} in %arg0 ...
%ew = transform.structured.match ops{["linalg.elemwise_binary"]} in %arg0 ...
%add, %max = transform.split_handle %ew ...

// consumer(relu/max)를 forall로 tile
%tiled, %loop = transform.structured.tile_using_forall %max tile_sizes [8, 32]
    : (!transform.any_op) -> (!transform.any_op, !transform.any_op)

// producer를 루프 안으로 — consumer-before-producer 순서
%add_fused, %loop2 = transform.structured.fuse_into_containing_op %add into %loop
    : (!transform.any_op, !transform.any_op) -> (!transform.any_op, !transform.any_op)
%mm_fused, %loop3 = transform.structured.fuse_into_containing_op %mm into %loop2
    : (!transform.any_op, !transform.any_op) -> (!transform.any_op, !transform.any_op)

fuse_into_containing_op는 매번 새 loop handle(%loop2, %loop3)을 돌려준다 — fusion이 컨테이너 루프를 다시 쓰기 때문에 이전 handle은 무효화되고, 다음 fuse는 갱신된 handle을 받아야 한다.

Stage C (… → vectorize): B 뒤에 func.func를 잡아 children을 vectorize. vectorize_children_and_apply_patterns는 isolated-from-above parent(여기선 func.func) handle을 받아 그 안의 모든 linalg op을 vector dialect로 내린다.

%func = transform.structured.match ops{["func.func"]} in %arg0
    : (!transform.any_op) -> !transform.any_op
%vectorized = transform.structured.vectorize_children_and_apply_patterns %func
    : (!transform.any_op) -> !transform.any_op

실행은 mlir-opt --transform-interpreter --canonicalize. --canonicalize가 fold·dedup를 정리해 읽기 좋은 최종 IR을 만든다.

Q결과가 어떻게 다른가? (before→after 실측)

먼저 변형 전 payload(원본 3-op 사슬). tensor 전체에 op 3개가 차례로 흐른다 — 루프 없음.

func.func @fc_relu(%lhs: tensor<128x128xf32>, %rhs: tensor<128x128xf32>,
                   %bias: tensor<128x128xf32>, %output: tensor<128x128xf32>)
                   -> tensor<128x128xf32> {
  %matmul = linalg.matmul ins(%lhs, %rhs: tensor<128x128xf32>, tensor<128x128xf32>)
                          outs(%output: tensor<128x128xf32>) -> tensor<128x128xf32>
  %biased = linalg.elemwise_binary { fun = #linalg.binary_fn<add> }
    ins(%matmul, %bias : tensor<128x128xf32>, tensor<128x128xf32>)
    outs(%output : tensor<128x128xf32>) -> tensor<128x128xf32>
  %c0f = arith.constant 0.0 : f32
  %relued = linalg.elemwise_binary { fun = #linalg.binary_fn<max_signed> }
    ins(%biased, %c0f : tensor<128x128xf32>, f32)
    outs(%output : tensor<128x128xf32>) -> tensor<128x128xf32>
  func.return %relued : tensor<128x128xf32>
}

대조 1 — A(tile만) vs B(tile→fuse). 가장 핵심적인 변화다. A는 max만 루프 안에 들어가고 matmul·add는 루프 밖에 남는다. B는 fuse로 셋 다 루프 안으로 빨려 들어간다.

A → B : fuse_into_containing_op가 producer를 루프 안으로
before — A (tile만): matmul·add는 forall 밖
%0 = linalg.matmul ins(%arg0, %arg1 : tensor<128x128xf32>, tensor<128x128xf32>) outs(%arg3 : tensor<128x128xf32>) -> tensor<128x128xf32>
%1 = linalg.elemwise_binary {fun = #linalg.binary_fn<add>} ins(%0, %arg2 : tensor<128x128xf32>, tensor<128x128xf32>) outs(%arg3 : tensor<128x128xf32>) -> tensor<128x128xf32>
%2 = scf.forall (%arg4, %arg5) in (16, 4) shared_outs(%arg6 = %arg3) -> (tensor<128x128xf32>) {
  %3 = affine.apply #map(%arg4)
  %4 = affine.apply #map1(%arg5)
  %extracted_slice = tensor.extract_slice %1[%3, %4] [8, 32] [1, 1] : tensor<128x128xf32> to tensor<8x32xf32>
  %extracted_slice_0 = tensor.extract_slice %arg6[%5, %6] [8, 32] [1, 1] : tensor<128x128xf32> to tensor<8x32xf32>
  %7 = linalg.elemwise_binary {fun = #linalg.binary_fn<max_signed>} ins(%extracted_slice, %cst : tensor<8x32xf32>, f32) outs(%extracted_slice_0 : tensor<8x32xf32>) -> tensor<8x32xf32>
  scf.forall.in_parallel {
    tensor.parallel_insert_slice %7 into %arg6[%8, %9] [8, 32] [1, 1] : tensor<8x32xf32> into tensor<128x128xf32>
  }
}
after — B (+fuse): 세 op 모두 forall 안, 8x32 tile 위에서
%0 = scf.forall (%arg4, %arg5) in (16, 4) shared_outs(%arg6 = %arg3) -> (tensor<128x128xf32>) {
  %extracted_slice = tensor.extract_slice %arg0[%7, 0] [8, 128] [1, 1] : tensor<128x128xf32> to tensor<8x128xf32>
  %extracted_slice_0 = tensor.extract_slice %arg1[0, %8] [128, 32] [1, 1] : tensor<128x128xf32> to tensor<128x32xf32>
  %extracted_slice_1 = tensor.extract_slice %arg3[%9, %10] [8, 32] [1, 1] : tensor<128x128xf32> to tensor<8x32xf32>
  %11 = linalg.matmul ins(%extracted_slice, %extracted_slice_0 : tensor<8x128xf32>, tensor<128x32xf32>) outs(%extracted_slice_1 : tensor<8x32xf32>) -> tensor<8x32xf32>
  %extracted_slice_2 = tensor.extract_slice %arg2[%3, %4] [8, 32] [1, 1] : tensor<128x128xf32> to tensor<8x32xf32>
  %12 = linalg.elemwise_binary {fun = #linalg.binary_fn<add>} ins(%11, %extracted_slice_2 : tensor<8x32xf32>, tensor<8x32xf32>) outs(%extracted_slice_3 : tensor<8x32xf32>) -> tensor<8x32xf32>
  %13 = linalg.elemwise_binary {fun = #linalg.binary_fn<max_signed>} ins(%12, %cst : tensor<8x32xf32>, f32) outs(%extracted_slice_4 : tensor<8x32xf32>) -> tensor<8x32xf32>
  scf.forall.in_parallel {
    tensor.parallel_insert_slice %13 into %arg6[%14, %15] [8, 32] [1, 1] : tensor<8x32xf32> into tensor<128x128xf32>
  }
}

B에서 주목할 점: fuse된 matmul은 단순히 같은 8x32 tile이 아니라 reduction 차원 K를 통째로 가진 slice를 본다 — %arg0[%7,0] [8,128] × %arg1[0,%8] [128,32] → tensor<8x32>. tile은 출력 공간(M,N)만 자르고, contraction의 K(128)는 한 tile 안에서 통째로 돈다. add도 %arg2(bias)의 같은 8x32 slice를 끌고 들어와 producer 사슬이 tile 단위로 재구성됐다.

대조 2 — B(linalg) vs C(vector). vectorize가 tile 위 linalg op을 vector dialect로 내린다. linalg.matmul→vector.contract, elemwise add→arith.addf, max_signed→arith.maximumf. linalg op은 0개가 된다(grep linalg. = 0).

B → C : vectorize_children — linalg op이 vector/arith로
before — B: tile 위 linalg op 3종
%11 = linalg.matmul ins(%extracted_slice, %extracted_slice_0 : tensor<8x128xf32>, tensor<128x32xf32>) outs(%extracted_slice_1 : tensor<8x32xf32>) -> tensor<8x32xf32>
%12 = linalg.elemwise_binary {fun = #linalg.binary_fn<add>} ins(%11, %extracted_slice_2 : ...) outs(...) -> tensor<8x32xf32>
%13 = linalg.elemwise_binary {fun = #linalg.binary_fn<max_signed>} ins(%12, %cst : tensor<8x32xf32>, f32) outs(...) -> tensor<8x32xf32>
after — C (+vectorize): transfer_read / contract / addf / maximumf / transfer_write
%4 = vector.transfer_read %arg0[%3, %c0], %cst_0 {in_bounds = [true, true]} : tensor<128x128xf32>, vector<8x128xf32>
%6 = vector.transfer_read %arg1[%c0, %5], %cst_0 {in_bounds = [true, true]} : tensor<128x128xf32>, vector<128x32xf32>
%9 = vector.transfer_read %arg3[%7, %8], %cst_0 {in_bounds = [true, true]} : tensor<128x128xf32>, vector<8x32xf32>
%10 = vector.contract {indexing_maps = [#map2, #map3, #map4], iterator_types = ["parallel", "parallel", "reduction"], kind = #vector.kind<add>} %4, %6, %9 : vector<8x128xf32>, vector<128x32xf32> into vector<8x32xf32>
%13 = vector.transfer_read %arg2[%11, %12], %cst_0 {in_bounds = [true, true]} : tensor<128x128xf32>, vector<8x32xf32>
%14 = arith.addf %10, %13 : vector<8x32xf32>
%15 = arith.maximumf %14, %cst : vector<8x32xf32>
%16 = vector.transfer_write %15, %extracted_slice[%c0, %c0] {in_bounds = [true, true]} : vector<8x32xf32>, tensor<8x32xf32>

C에서 %cst가 dense<0.0> : vector<8x32xf32>로 바뀐 점도 본다 — relu의 스칼라 0이 broadcast된 vector 상수로 hoist돼 arith.maximumf %14, %cst가 곧 ReLU다. in_bounds = [true, true]는 128이 8·32로 딱 나눠떨어져 경계 검사가 필요 없다는 정적 결론.

Q그래서 핵심은?

T10의 통찰은 "transform dialect는 schedule을 합성 가능한(composable) IR 한 채로 쓴다"는 것이다. tile·fuse·vectorize가 각각 독립 op이고, 앞 op이 돌려준 handle을 뒤 op이 입력으로 받아 사슬을 잇는다 — polyhedral의 단일 ILP 결정과 달리, 사람이 단계를 명시적으로 쌓는다.

tile = 출력 공간만 자른다

forall (16, 4) = M/8 × N/32. contraction 차원 K(128)는 tile에 들어가지 않아 matmul fuse 시 [8,128]×[128,32] slice로 K 전체를 한 tile에서 돈다.

fuse는 handle을 갱신한다

fuse_into_containing_op가 매번 새 loop handle(%loop2/%loop3)을 반환. 이전 handle은 무효 — 다음 fuse는 갱신본을 받아야 IR 정합성이 유지된다.

fuse 순서 = consumer→producer

add(consumer에 가까움)를 먼저, matmul(멀리)을 나중에. producer-consumer 사슬을 역순으로 끌어들여야 각 단계가 직전 결과를 본다.

vectorize = codegen 직전 경계

linalg op 0개, 자리에 vector.contract + arith.addf/maximumf. 여기서부터는 vector→LLVM lowering이 받는다. structured op의 끝.

매 stage가 schedule에 딱 필요한 줄만 추가해 IR을 한 단계씩 내리는 모습 자체가, transform dialect가 codegen을 "프로그래밍 가능한 pass 파이프라인"으로 재구성한다는 증거다.

Q함정/주의는?
함정맞는 방법
split_handle 없이 elemwise 둘에 바로 tile"elemwise_binary" match는 add·max 2개를 한 handle로 잡는다. tile은 단일 op을 기대하므로 split_handle로 갈라 %max만 지정해야 한다.
fuse 후에도 옛 %loop handle 재사용fusion이 컨테이너 forall을 rewrite → 옛 handle 무효. fuse_into_containing_op가 돌려준 새 loop handle(%loop2 등)을 다음 fuse 입력으로 써야 한다.
producer를 consumer보다 먼저 fusematmul을 먼저 fuse하면 아직 add가 루프 밖이라 사슬이 끊긴다. add → matmul(소비자에 가까운 쪽부터) 순서를 지킨다.
vectorize에 tile op handle을 넘김vectorize_children_and_apply_patterns는 isolated-from-above parent를 받는다. 여기선 func.func를 다시 match해 넘긴다 — forall이나 개별 linalg handle이 아니다.
--canonicalize 생략없으면 fuse가 만든 중복 affine.apply·dead slice가 그대로 남아 IR이 어수선하다. 실측 out IR이 깔끔한 건 canonicalize 덕.

검증: run.sh 그대로 재실행 시 A는 forall=1·matmul=1(루프 밖), B는 forall=1·matmul-in-loop=1(루프 안), C는 forall=1·vector.contract=1·linalg-left=0. 본 문서 IR과 일치한다.

Q한 줄 + 관련

한 줄: transform dialect는 fc+bias+relu 커널 한 채를 match → tile_using_forall → fuse_into_containing_op×2 → vectorize_children 사슬로, 매 stage 딱 한 줄씩 추가해 단일 scf.forall + vector.contract codegen 직전 형태까지 명시적으로 몰아간다 — handle이 곧 데이터 흐름이고, fuse 순서·handle 갱신·isolated parent가 정합성의 핵심.

관련 (P1 cycle)

T이전 단일 tile/fuse 실험들 — 본 실험은 그것들을 한 schedule로 합성한 종합편. affine-mlir baseline의 tile #0013·fuse 변환과 1:1 대조: polyhedral은 ILP 한 방, transform은 단계 누적.

다음
