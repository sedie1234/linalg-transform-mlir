T06 — pad & pack (tensor.pack/unpack, mmt4d)

하나의 linalg.matmul을 transform.structured.pack으로 block-pack 해서 6D linalg.generic + tensor.pack×3 + tensor.unpack으로 바꾸고, 거기서 파생된 pack_transpose(레이아웃 뒤집기)와 lower_pack(pack→pad+expand_shape+transpose 분해)까지 schedule IR로 명시한다.

▶먼저 — 왜 pack인가? blocked 4D layout이 뭔가?

문제. linalg.matmul은 2D 텐서를 행 우선(row-major)으로 읽는다. 그런데 micro-kernel(SIMD·레지스터로 작은 타일을 곱하는 알맹이 코드)은 한 번에 작은 블록(예: 4×8)을 처리하는데, 2D row-major에서는 한 블록의 원소들이 메모리에 흩어져 있다 — 행이 바뀔 때마다 한 행 크기(stride)만큼 건너뛰어야 한다. 캐시 적중·SIMD 적재 효율이 떨어진다.

pack = 데이터를 다시 깔기. 그래서 연산 전에 2D를 4D blocked layout으로 재배치한다. 각 차원을 outer × inner로 쪼개고 inner를 뒤로 모아, 한 타일이 메모리에 연속으로 놓이게 한다. 값은 그대로, 배치만 바꾼다.

예: 32×32를 inner-tile [4, 8]로 pack하면 8×4×4×8이 된다.

tensor<32 x 32>   --pack inner_tiles=[4,8]-->   tensor<8 x 4 x 4 x 8>
        M    K                                    Mo  Ko  Mi  Ki
   M = 32 = Mo·Mi = 8·4          Mo,Ko = "어느 블록"(8×4개 블록 그리드)
   K = 32 = Ko·Ki = 4·8          Mi,Ki = "블록 안 4×8 타일" = 메모리 연속

즉 [Mo=8][Ko=4][Mi=4][Ki=8] — 바깥 두 축은 블록 좌표, 안쪽 두 축은 그 블록 안의 4×8 타일. 같은 타일의 32개 원소가 이제 메모리에 붙어 있어 micro-kernel이 한 번에 빨아들인다.

그래서 matmul이 6D generic이 된다. 세 operand가 다 4D가 되니 곱셈 루프도 6중([Mo,No,Ko, Mi,Ni,Ki])이 된다. 결과 IR의 iterator_types = [parallel,parallel,reduction, parallel,parallel,reduction]이 정확히 [Mo,No,Ko, Mi,Ni,Ki] — 바깥 3개(블록 단위 M·N·K)와 안쪽 3개(타일 안 m·n·k), 그중 K축(Ko·Ki)이 reduction이다. 이게 mmt4d(0010에서 본 4D micro-kernel matmul) 형태다.

unpack = 되돌리기. 계산이 끝난 4D 결과를 다시 2D로 푼다. pack/unpack은 레이아웃만 바꿀 뿐 matmul의 수학적 의미는 그대로(입·출력 2D 타입 불변). 0010 linalg-block-pack-matmul 패스가 자동으로 하던 그 일을, T06은 schedule IR로 직접 명시·통제한다.

Q이 실험은 뭘 보여주나?

이 실험은 data layout 변환 계열 transform op 묶음을 한 자리에 모은 것이다. 핵심 대상은 tensor.pack / tensor.unpack — 2D matmul operand를 cache·SIMD 친화적인 4D blocked layout으로 재배치하는 연산이다.

세 가지 schedule을 같은 payload(혹은 더 큰 변형 payload)에 적용해 다음을 캡처한다.

  • a-pack: transform.structured.pack 하나로 matmul(M,N,K)을 packed_sizes=[4,8,8]로 block-pack. 결과는 tensor.pack×3(A,B,C) + 6D linalg.generic + tensor.unpack×1. 이는 0010 block-pack 패스가 IR에서 한 일을 schedule IR로 명시한 버전이다.
  • b-pack-transpose: a-pack 위에서 B operand의 pack을 잡아 pack_transpose로 inner/outer 순서를 뒤집는다. KN-packing → NK-packing.
  • c-greedily-lower: pack_greedily로 휴리스틱 block-pack 후 한 tensor.pack을 lower_pack으로 tensor.pad + tensor.expand_shape + linalg.transpose 세 갈래로 분해. 이쪽 payload는 packed size로 안 떨어지는 1023×255 등 odd shape이라 padding이 실제로 발생한다.

왜? mmt4d(4D micro-kernel matmul)·tile·vectorize 파이프라인의 앞단은 거의 항상 "operand를 micro-kernel이 좋아하는 blocked layout으로 pack"하는 단계로 시작한다. 이 실험은 그 pack 단계가 transform dialect에서 어떤 op으로, 어떤 IR을 만들며 표현되는지를 실측 IR로 못 박는다.

Q어떻게 했나? (schedule)

세 schedule 모두 transform.structured.match ops{["linalg.matmul"]}로 payload의 matmul 하나를 handle로 잡는 데서 시작한다. 거기서부터 갈라진다.

a-pack — 기본 pack. matmul의 (M,N,K) 세 축에 inner-tile 크기 [4,8,8]을 주면, 각 operand가 2D→4D로 packed 되고 compute op은 6D generic으로 바뀐다.

%matmul = transform.structured.match ops{["linalg.matmul"]} in %arg0
  : (!transform.any_op) -> !transform.any_op
//                                              M  N  K
%packed = transform.structured.pack %matmul packed_sizes = [4, 8, 8]
  : (!transform.any_op) -> (!transform.op<"linalg.generic">)

b-pack-transpose — pack 후 레이아웃 뒤집기. 먼저 a-pack과 동일하게 pack 한 뒤, get_producer_of_operand %packed[1]로 compute op의 operand 1(=B)을 만든 tensor.pack handle을 집어온다. 그 pack에 pack_transpose를 걸어 outer_perm=[1,0] inner_perm=[1,0]으로 inner/outer 차원 순서를 뒤집는다. 이때 compute op의 indexing_map도 자동으로 함께 갱신된다(레이아웃이 바뀌었으니 일관성 유지).

%packed = transform.structured.pack %matmul packed_sizes = [4, 8, 8]
  : (!transform.any_op) -> (!transform.op<"linalg.generic">)
// operand 1 = B 의 producer pack 을 집어온다.
%pack_b = transform.get_producer_of_operand %packed[1]
  : (!transform.op<"linalg.generic">) -> (!transform.op<"tensor.pack">)
%generic2, %pack2, %unpack2 =
  transform.structured.pack_transpose %pack_b with_compute_op(%packed)
  outer_perm = [1, 0] inner_perm = [1, 0]
  : (!transform.op<"tensor.pack">, !transform.op<"linalg.generic">)
  -> (!transform.op<"linalg.generic">, !transform.op<"tensor.pack">, !transform.any_op)

c-greedily-lower — 자동 pack 후 한 pack을 lower. pack_greedily는 휴리스틱으로 M,N,K inner-tile을 잡아 한 방에 generic으로 pack 한다(여기선 matmul_packed_sizes=[8,16,32], matmul_inner_dims_order=[1,2,0] 지정). 그 다음 operand 0(=A)을 만든 tensor.pack을 lower_pack으로 분해한다. pack은 "(필요하면) pad → expand_shape로 차원 쪼개기 → transpose로 재배열"의 합성이므로, padding이 필요한 경우 결과 타입이 tensor.pad, tensor.expand_shape, linalg.transpose 세 갈래로 나온다.

%matmul = transform.structured.match ops{["linalg.matmul"]} in %arg0
  : (!transform.any_op) -> !transform.op<"linalg.matmul">
// greedily: heuristic으로 M,N,K inner-tile을 잡아 generic으로 pack.
%generic = transform.structured.pack_greedily %matmul
    matmul_packed_sizes = [8, 16, 32] matmul_inner_dims_order = [1, 2, 0]
  : (!transform.op<"linalg.matmul">) -> !transform.op<"linalg.generic">
// 생성된 tensor.pack 들 중 첫 번째(A operand)를 lower.
%pack = transform.get_producer_of_operand %generic[0]
  : (!transform.op<"linalg.generic">) -> (!transform.op<"tensor.pack">)
transform.structured.lower_pack %pack : (!transform.op<"tensor.pack">)
  -> (!transform.op<"tensor.pad">, !transform.op<"tensor.expand_shape">, !transform.op<"linalg.transpose">)

패턴: match → pack(또는 pack_greedily) → get_producer_of_operand[i]로 특정 operand의 pack handle 회수 → 그 handle에 pack_transpose/lower_pack 후처리. operand별 producer pack을 handle로 다시 잡아 정밀 조작한다는 게 이 계열의 공통 골격이다.

Q결과가 어떻게 다른가? (before→after 실측)

(1) a-pack — matmul 하나가 pack 떼거리 + 6D generic + unpack 으로. 입력 payload는 평범한 32×32 matmul 단 한 줄이다. pack 후엔 operand 3개가 모두 4D로 재배치되고(32×32 → 8×4×4×8 등), 연산은 6개 iterator(parallel,parallel,reduction,parallel,parallel,reduction)를 가진 generic이 된다. 마지막에 tensor.unpack이 결과를 다시 2D로 되돌린다.

a-pack — pack [4,8,8]
before — input/payload.mlir
func.func @matmul(%A: tensor<32x32xf32>, %B: tensor<32x32xf32>,
                  %C: tensor<32x32xf32>) -> tensor<32x32xf32> {
  %0 = linalg.matmul ins(%A, %B : tensor<32x32xf32>, tensor<32x32xf32>)
                     outs(%C : tensor<32x32xf32>) -> tensor<32x32xf32>
  return %0 : tensor<32x32xf32>
}
after — output/out-a-pack.mlir

#map1 = affine_map<(d0, d1, d2, d3, d4, d5) -> (d2, d1, d4, d5)>
#map2 = affine_map<(d0, d1, d2, d3, d4, d5) -> (d0, d1, d3, d4)>
func.func @matmul(%arg0: tensor<32x32xf32>, %arg1: tensor<32x32xf32>, %arg2: tensor<32x32xf32>) -> tensor<32x32xf32> {
  %0 = tensor.empty() : tensor<8x4x4x8xf32>
  %pack = tensor.pack %arg0 inner_dims_pos = [0, 1] inner_tiles = [4, 8] into %0 : tensor<32x32xf32> -> tensor<8x4x4x8xf32>
  %1 = tensor.empty() : tensor<4x4x8x8xf32>
  %pack_0 = tensor.pack %arg1 inner_dims_pos = [1, 0] inner_tiles = [8, 8] into %1 : tensor<32x32xf32> -> tensor<4x4x8x8xf32>
  %2 = tensor.empty() : tensor<8x4x4x8xf32>
  %pack_1 = tensor.pack %arg2 inner_dims_pos = [0, 1] inner_tiles = [4, 8] into %2 : tensor<32x32xf32> -> tensor<8x4x4x8xf32>
  %3 = linalg.generic {indexing_maps = [#map, #map1, #map2], iterator_types = ["parallel", "parallel", "reduction", "parallel", "parallel", "reduction"]} ins(%pack, %pack_0 : tensor<8x4x4x8xf32>, tensor<4x4x8x8xf32>) outs(%pack_1 : tensor<8x4x4x8xf32>) {
  ^bb0(%in: f32, %in_2: f32, %out: f32):
    %4 = arith.mulf %in, %in_2 : f32
    %5 = arith.addf %out, %4 : f32
    linalg.yield %5 : f32
  } -> tensor<8x4x4x8xf32>
  %unpack = tensor.unpack %3 inner_dims_pos = [0, 1] inner_tiles = [4, 8] into %arg2 : tensor<8x4x4x8xf32> -> tensor<32x32xf32>
  return %unpack : tensor<32x32xf32>
}

(2) a-pack vs b-pack-transpose — B의 pack 한 줄과 그 indexing_map만 달라진다. a→b의 차이는 국소적이다. B의 pack이 a에선 inner_dims_pos=[1,0] inner_tiles=[8,8](결과 4x4x8x8, perm 없음)였는데, b에선 outer_dims_perm=[1,0] inner_dims_pos=[0,1] inner_tiles=[8,8]로 outer 차원이 뒤집힌 pack이 된다. 동시에 B에 대응하는 #map1이 (d2,d1,d4,d5)→(d1,d2,d5,d4)로 갱신된다 — pack_transpose가 레이아웃과 indexing_map을 한꺼번에 일관되게 돌렸다는 증거다.

(3) c-greedily-lower — pack 한 개가 pad+expand_shape+transpose 로 풀린다. 이쪽 payload는 1023×255 · 255×127 · 1023×127의 odd shape이라 packed size [8,16,32]로 안 떨어진다. pack_greedily로 6D generic을 만든 뒤 A operand의 pack을 lower_pack 하면, A는 더 이상 tensor.pack 한 줄이 아니라 pad(1023×255→1024×256) → expand_shape(차원 쪼개기) → linalg.transpose([0,2,3,1]) 세 단계의 합성으로 풀려 나온다. 반면 lower 안 한 B·C는 여전히 tensor.pack(이쪽은 padding_value를 단 채)으로 남는다 — 대조가 선명하다.

c-greedily-lower — A는 lower, B/C는 pack 잔존
before — input/sched-c payload
func.func @matmul_mk_kn_mn(%A: tensor<1023x255xf32>, %B: tensor<255x127xf32>,
                           %C: tensor<1023x127xf32>) -> tensor<1023x127xf32> {
  %0 = linalg.matmul ins(%A, %B : tensor<1023x255xf32>, tensor<255x127xf32>)
                     outs(%C : tensor<1023x127xf32>) -> tensor<1023x127xf32>
  return %0 : tensor<1023x127xf32>
}
after — output/out-c-greedily-lower.mlir
#map = affine_map<(d0, d1, d2, d3, d4, d5) -> (d1, d0, d3, d4)>
#map1 = affine_map<(d0, d1, d2, d3, d4, d5) -> (d0, d2, d3, d5)>
#map2 = affine_map<(d0, d1, d2, d3, d4, d5) -> (d1, d2, d4, d5)>
func.func @matmul_mk_kn_mn(%arg0: tensor<1023x255xf32>, %arg1: tensor<255x127xf32>, %arg2: tensor<1023x127xf32>) -> tensor<1023x127xf32> {
  %0 = tensor.empty() : tensor<128x8x32x8xf32>
  %cst = arith.constant 0.000000e+00 : f32
  %padded = tensor.pad %arg0 low[0, 0] high[1, 1] {
  ^bb0(%arg3: index, %arg4: index):
    tensor.yield %cst : f32
  } : tensor<1023x255xf32> to tensor<1024x256xf32>
  %expanded = tensor.expand_shape %padded [[0, 1], [2, 3]] output_shape [128, 8, 8, 32] : tensor<1024x256xf32> into tensor<128x8x8x32xf32>
  %transposed = linalg.transpose ins(%expanded : tensor<128x8x8x32xf32>) outs(%0 : tensor<128x8x32x8xf32>) permutation = [0, 2, 3, 1]
  %1 = tensor.empty() : tensor<8x8x32x16xf32>
  %cst_0 = arith.constant 0.000000e+00 : f32
  %pack = tensor.pack %arg1 padding_value(%cst_0 : f32) inner_dims_pos = [0, 1] inner_tiles = [32, 16] into %1 : tensor<255x127xf32> -> tensor<8x8x32x16xf32>
  %2 = tensor.empty() : tensor<128x8x8x16xf32>
  %cst_1 = arith.constant 0.000000e+00 : f32
  %pack_2 = tensor.pack %arg2 padding_value(%cst_1 : f32) inner_dims_pos = [0, 1] inner_tiles = [8, 16] into %2 : tensor<1023x127xf32> -> tensor<128x8x8x16xf32>
  %3 = linalg.generic {indexing_maps = [#map, #map1, #map2], iterator_types = ["reduction", "parallel", "parallel", "reduction", "parallel", "parallel"]} ins(%transposed, %pack : tensor<128x8x32x8xf32>, tensor<8x8x32x16xf32>) outs(%pack_2 : tensor<128x8x8x16xf32>) {
  ^bb0(%in: f32, %in_3: f32, %out: f32):
    %4 = arith.mulf %in, %in_3 : f32
    %5 = arith.addf %out, %4 : f32
    linalg.yield %5 : f32
  } -> tensor<128x8x8x16xf32>
  %unpack = tensor.unpack %3 inner_dims_pos = [0, 1] inner_tiles = [8, 16] into %arg2 : tensor<128x8x8x16xf32> -> tensor<1023x127xf32>
  return %unpack : tensor<1023x127xf32>
}
▶pack_greedily와 lower_pack을 더 풀면?

pack_greedily — "휴리스틱으로 알아서 pack". 평범한 structured.pack은 operand마다 inner_dims_pos·inner_tiles를 일일이 지정해야 한다. pack_greedily는 op이 matmul임을 greedy하게 인식해 세 operand(A·B·C)를 한 번에 일관되게 pack한다 — matmul_packed_sizes=[8,16,32]는 검출된 M·N·K의 inner-tile 크기. operand별 세부를 안 짜도 contraction에 맞춰 자동 packing 한다는 게 "greedily"의 뜻이다.

pack (수동)pack_greedily (자동)
지정operand마다 inner_dims_pos·inner_tilesmatmul_packed_sizes [M,N,K] 한 번
대상 인식사용자가 직접matmul/contraction을 greedy 인식

lower_pack — "추상 pack을 실제 데이터 이동으로 분해". tensor.pack은 "블록으로 재배치하라"는 의도만 담은 한 op이다. lower_pack은 그게 실제로 무슨 일인지를 하위 op 셋으로 푼다(progressive lowering = 추상 → 구체):

분해 결과하는 일 (c-greedily-lower의 A operand, 1023×255 기준)
tensor.padinner tile 배수로 패딩 — 1023→1024, 255→256(high[1,1], 0으로 채움)
tensor.expand_shape각 차원을 outer×inner로 쪼개 reshape — 1024×256 → 128×8×8×32
linalg.transpose축을 [outer…, inner…] 순서로 재배열 — permutation=[0,2,3,1]

즉 tensor.pack 한 줄 = "(필요하면) pad → expand_shape로 차원 분할 → transpose로 재배열"의 합성이다. 실측 C는 A operand의 pack 하나만 lower해서 pad·expand_shape·transpose가 1개씩 나오고, lower 안 한 B·C의 tensor.pack은 그대로 남는다 — 대조가 보이게 일부러 하나만 내렸다. odd shape(1023…)이라 pad가 실제로 등장하고, nice shape(32…)이면 pad 없이 expand+transpose만으로 풀린다.

Q그래서 핵심은?

pack = layout-only, semantic 보존

pack/unpack은 데이터를 4D blocked layout으로 재배치할 뿐 matmul의 수학적 의미를 바꾸지 않는다. matmul → 6D generic + pack×3 + unpack×1으로 바뀌어도 입/출력 2D 타입은 그대로다(unpack이 마지막에 되돌림).

operand별 producer를 다시 handle로

get_producer_of_operand %op[i]로 특정 operand를 만든 pack을 handle로 회수해 정밀 조작한다. pack_transpose도 lower_pack도 모두 이 회수된 pack handle에 거는 후처리다.

pack_transpose는 indexing_map까지 동기화

레이아웃만 뒤집고 compute op은 안 고치면 의미가 깨진다. pack_transpose는 pack의 perm과 generic의 indexing_map(#map1)을 한 번에 일관 갱신한다.

lower_pack = pad+expand_shape+transpose 합성

tensor.pack은 사실 "(필요시) pad → expand_shape로 차원 분할 → transpose로 재배열"의 합성 연산. odd shape이면 pad가 실제로 등장하고(1023→1024), nice shape이면 pad 없이 풀린다.

schedule핵심 oppayload관찰된 IR 산출
a-packstructured.pack [4,8,8]32×32 (nice)pack×3 + 6D generic + unpack×1, padding 없음
b-pack-transposepack_transpose outer/inner=[1,0]32×32 (nice)B pack에 outer_dims_perm=[1,0], #map1 동시 갱신
c-greedily-lowerpack_greedily + lower_pack1023×255 (odd)A: pad(→1024×256)+expand_shape+transpose, B/C: padding_value 단 pack 잔존

관통 정리: pack 계열은 "matmul → blocked 4D micro-kernel (mmt4d) 파이프라인"의 첫 단계다. transform dialect는 이 pack을 (1) 통째로 적용(pack), (2) 특정 operand만 레이아웃 뒤집기(pack_transpose), (3) primitive로 분해(lower_pack)하는 세 수준의 제어를 schedule IR로 명시할 수 있게 한다. 같은 효과를 0010 block-pack 패스는 자동으로 했지만, 여기선 무엇을·어디에·어떻게가 IR 텍스트로 다 드러난다.

Q한 줄 + 관련

한 줄: transform.structured.pack은 matmul을 4D blocked layout(pack×3 + 6D generic + unpack)으로 재배치하고, pack_transpose는 한 operand의 레이아웃을 indexing_map과 함께 뒤집으며, lower_pack은 pack을 pad+expand_shape+transpose primitive로 분해한다 — mmt4d 파이프라인 앞단을 schedule IR로 완전히 명시할 수 있다.

관련:

  • 0010 block-pack 패스 — 같은 pack 효과를 자동 패스로 수행한 비교 기준. 이 실험은 그 패스가 IR에서 한 일을 schedule IR로 명시한 버전이다.
  • T05 / tile-first 계열 — pack 다음 단계인 tile·fuse·vectorize. mmt4d 파이프라인은 pack → tile → micro-kernel 순.
  • affine baseline — polyhedral 쪽에는 pack에 직접 대응하는 자동 변환이 없다(layout은 보통 data-layout 패스/수동). transform dialect는 pack을 1급 schedule op으로 제공한다는 게 대조점.

다음: lower_pack에 이어 tensor.pack/unpack을 loop·vector로 더 내려보내거나, mmt4d micro-kernel로 연결해 실제 코드 생성까지 잇는 단계.
