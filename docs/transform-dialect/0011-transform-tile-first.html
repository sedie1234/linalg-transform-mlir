변환을 컴파일러 *밖*에서 — transform dialect

변환 규칙(어느 op을·어떻게·어떤 순서로 바꿀지)을 C++ 패스에 박지 않고, 텍스트 MLIR script로 외부에서 쓰고 -transform-interpreter로 적용한다.

"사용자가 변환규칙을 컴파일러 외부에서 정의하고 적용한다는 게 어떤 이점을 주는지 이해가 안 된다" — 이 문서는 그 질문을 정면으로 푼다. 아래 IR은 전부 mlir-opt -transform-interpreter 실제 출력(T01).
payload IR

변환 대상. 예: linalg.matmul. "무엇을 계산하나".

transform IR

변환을 기술한 script. transform op들의 나열. "어떻게 최적화하나".

handle

transform IR의 SSA 값이 payload의 op/value를 가리킴. %mm = "그 matmul".

interpreter

-transform-interpreter가 script를 읽어 payload에 적용. 진입점 @__transform_main.

Q이게 대체 뭔데?

MLIR의 보통 변환은 C++로 짠 패스다(네가 P1에서 해부한 linalg-* 패스들). transform dialect는 다르다 — 변환 자체를 IR로 표현한다. 한 모듈 안에 변환 대상(payload)과 변환 지시서(transform IR)가 같이 있고, interpreter가 지시서를 해석해 대상을 바꾼다.

linalg.matmul ins(%A,%B) outs(%C)

►
적용

transform IR — 어떻게

변환 지시서(schedule). 텍스트로 갈아끼움.

match → tile → fuse → vectorize ...
핵심 한 줄: "무엇을 계산하나(payload)"와 "어떻게 최적화하나(schedule)"를 분리하고, 후자를 데이터로 만든 것. (Halide/TVM의 compute–schedule 분리를 MLIR로.)
Q그래서 어떻게 쓰는데? (사용하는 입장에서)

모듈에 transform.with_named_sequence 속성을 달고, 진입점 @__transform_main 안에 ① 대상을 지목(match) → ② 변환(tile 등)을 쓴 뒤 mlir-opt -transform-interpreter로 돌린다.

module attributes {transform.with_named_sequence} {
  func.func @matmul(%A: tensor<128x256xf32>, %B: tensor<256x64xf32>,
                    %C: tensor<128x64xf32>) -> tensor<128x64xf32> {
    %0 = linalg.matmul ins(%A, %B : ...) outs(%C : ...) -> tensor<128x64xf32>
    return %0 : tensor<128x64xf32>
  }
  // ── 여기부터가 schedule (payload 가 아니라 '변환 지시서') ──
  transform.named_sequence @__transform_main(%arg0: !transform.any_op {transform.readonly}) {
    %mm = transform.structured.match ops{["linalg.matmul"]} in %arg0
            : (!transform.any_op) -> !transform.any_op
    %t, %loops:3 = transform.structured.tile_using_for %mm tile_sizes [32, 32, 64]
            : (!transform.any_op) -> (!transform.any_op, !transform.any_op, !transform.any_op, !transform.any_op)
    transform.yield
  }
}

실행 한 줄: mlir-opt -transform-interpreter sched.mlir — interpreter가 schedule을 payload에 적용하고, schedule(transform IR)은 소비된다.

match는 변환이 아니다. %mm = transform.structured.match는 payload에서 matmul을 찾아 handle로 잡을 뿐, IR을 안 바꾼다. 실제 변환은 그 handle을 받는 tile_using_for가 한다.
Q그래서 어떤 효과가 나는데? (before → after, 실측)

위 schedule을 linalg.matmul(128×256×64)에 적용한 실제 결과. matmul 한 줄이 3중 scf.for 타일 nest + 32×32 inner matmul로 바뀐다. "변경 강조"로 바뀐 부분을 본다.

transform.structured.tile_using_for [32,32,64] 적용
before — payload (matmul 한 줄)
func.func @matmul(%A: tensor<128x256xf32>, %B: tensor<256x64xf32>,
                %C: tensor<128x64xf32>) -> tensor<128x64xf32> {
%0 = linalg.matmul ins(%A, %B : tensor<128x256xf32>, tensor<256x64xf32>)
                   outs(%C : tensor<128x64xf32>) -> tensor<128x64xf32>
return %0 : tensor<128x64xf32>
after — scf.for ×3 타일 nest
func.func @matmul(%arg0: tensor<128x256xf32>, %arg1: tensor<256x64xf32>, %arg2: tensor<128x64xf32>) -> tensor<128x64xf32> {
  %c256 = arith.constant 256 : index
  %c64 = arith.constant 64 : index
  %c0 = arith.constant 0 : index
  %c128 = arith.constant 128 : index
  %c32 = arith.constant 32 : index
  %0 = scf.for %arg3 = %c0 to %c128 step %c32 iter_args(%arg4 = %arg2) -> (tensor<128x64xf32>) {
    %1 = scf.for %arg5 = %c0 to %c64 step %c32 iter_args(%arg6 = %arg4) -> (tensor<128x64xf32>) {
      %2 = scf.for %arg7 = %c0 to %c256 step %c64 iter_args(%arg8 = %arg6) -> (tensor<128x64xf32>) {
        %extracted_slice = tensor.extract_slice %arg0[%arg3, %arg7] [32, 64] [1, 1] : tensor<128x256xf32> to tensor<32x64xf32>
        %extracted_slice_0 = tensor.extract_slice %arg1[%arg7, %arg5] [64, 32] [1, 1] : tensor<256x64xf32> to tensor<64x32xf32>
        %extracted_slice_1 = tensor.extract_slice %arg8[%arg3, %arg5] [32, 32] [1, 1] : tensor<128x64xf32> to tensor<32x32xf32>
        %3 = linalg.matmul ins(%extracted_slice, %extracted_slice_0 : tensor<32x64xf32>, tensor<64x32xf32>) outs(%extracted_slice_1 : tensor<32x32xf32>) -> tensor<32x32xf32>
        %inserted_slice = tensor.insert_slice %3 into %arg8[%arg3, %arg5] [32, 32] [1, 1] : tensor<32x32xf32> into tensor<128x64xf32>
        scf.yield %inserted_slice : tensor<128x64xf32>
      }
      scf.yield %2 : tensor<128x64xf32>
    }
    scf.yield %1 : tensor<128x64xf32>
  }
  return %0 : tensor<128x64xf32>
}

아래 애니메이션은 schedule이 payload를 지목(match)하고 타일하는 과정을 단계별로 보인다. schedule을 바꾸면 같은 payload가 다른 구조가 된다(A: 순차 scf.for / C: 병렬 scf.forall).

네 핵심 질문이다. 답: schedule이 컴파일러 코드가 아니라 데이터라서. 같은 payload, 같은 mlir-opt 바이너리에 schedule 텍스트만 바꿔 전혀 다른 IR을 얻는다 — 재빌드 0회. 아래 셋은 실제로 그렇게 나온 결과다.

func.func @matmul(%arg0: tensor<128x256xf32>, %arg1: tensor<256x64xf32>, %arg2: tensor<128x64xf32>) -> tensor<128x64xf32> {
  %c0 = arith.constant 0 : index
  %c128 = arith.constant 128 : index
  %c64 = arith.constant 64 : index
  %0 = scf.for %arg3 = %c0 to %c128 step %c64 iter_args(%arg4 = %arg2) -> (tensor<128x64xf32>) {
    %extracted_slice = tensor.extract_slice %arg0[%arg3, 0] [64, 256] [1, 1] : tensor<128x256xf32> to tensor<64x256xf32>
    %extracted_slice_0 = tensor.extract_slice %arg4[%arg3, 0] [64, 64] [1, 1] : tensor<128x64xf32> to tensor<64x64xf32>
    %1 = linalg.matmul ins(%extracted_slice, %arg1 : tensor<64x256xf32>, tensor<256x64xf32>) outs(%extracted_slice_0 : tensor<64x64xf32>) -> tensor<64x64xf32>
    %inserted_slice = tensor.insert_slice %1 into %arg4[%arg3, 0] [64, 64] [1, 1] : tensor<64x64xf32> into tensor<128x64xf32>
    scf.yield %inserted_slice : tensor<128x64xf32>
  }
  return %0 : tensor<128x64xf32>
}

#map = affine_map<(d0) -> (d0 * 64)>
#map1 = affine_map<(d0) -> (d0 * 32)>
func.func @matmul(%arg0: tensor<128x256xf32>, %arg1: tensor<256x64xf32>, %arg2: tensor<128x64xf32>) -> tensor<128x64xf32> {
  %0 = scf.forall (%arg3, %arg4) in (2, 2) shared_outs(%arg5 = %arg2) -> (tensor<128x64xf32>) {
    %1 = affine.apply #map(%arg3)
    %2 = affine.apply #map1(%arg4)
    %3 = affine.apply #map(%arg3)
    %4 = affine.apply #map1(%arg4)
    %extracted_slice = tensor.extract_slice %arg0[%1, 0] [64, 256] [1, 1] : tensor<128x256xf32> to tensor<64x256xf32>
    %extracted_slice_0 = tensor.extract_slice %arg1[0, %2] [256, 32] [1, 1] : tensor<256x64xf32> to tensor<256x32xf32>
    %extracted_slice_1 = tensor.extract_slice %arg5[%3, %4] [64, 32] [1, 1] : tensor<128x64xf32> to tensor<64x32xf32>
    %5 = linalg.matmul ins(%extracted_slice, %extracted_slice_0 : tensor<64x256xf32>, tensor<256x32xf32>) outs(%extracted_slice_1 : tensor<64x32xf32>) -> tensor<64x32xf32>
    %6 = affine.apply #map(%arg3)
    %7 = affine.apply #map1(%arg4)
    scf.forall.in_parallel {
      tensor.parallel_insert_slice %5 into %arg5[%6, %7] [64, 32] [1, 1] : tensor<64x32xf32> into tensor<128x64xf32>
    }
  }
  return %0 : tensor<128x64xf32>
}

이게 가져다주는 이점 4가지

① 재빌드 없는 schedule 교체 → autotuning

tile 크기·순서·병렬화를 바꾸려고 컴파일러를 고쳐 재빌드할 필요가 없다. script만 바꿔 수백~수천 변형을 생성→측정→최적 선택. schedule이 데이터라 가능. (NPU·가속기 커널 튜닝의 핵심.)

② 정책과 메커니즘 분리

tiling 알고리즘은 컴파일러에, "이 matmul을 32로 타일하라"는 결정은 외부 schedule에. 하드웨어를 아는 성능 엔지니어가 컴파일러 내부를 안 건드리고 튜닝.

③ 명시적·재현가능한 레시피

greedy 패스 파이프라인의 동작은 순서·benefit에서 창발하지만, schedule은 "이 handle에 tile→그 producer를 fuse→vectorize"가 IR로 1급 명시. 저장·diff·버전·공유·재생.

④ schedule이 작은 프로그램

match/foreach/alternatives(되는 것 선택)/실패(silenceable·definite)로 schedule에 제어흐름을 넣는다.

네가 아는 것으로 비유하면

  • 비유
    • Halide/TVM: Func가 무엇을 계산할지(compute), .tile().vectorize().parallel()가 schedule. transform dialect = MLIR판 compute–schedule 분리.
    • DB 쿼리: 옵티마이저에 맡기는 대신 explicit query plan/hint를 주는 것. schedule = 명시·override 가능한 "최적화 계획".
    • C++: 알고리즘을 하드코딩 vs 스크립팅 인터페이스로 노출. 빌드 스크립트처럼 "무엇을"은 두고 "어떻게"를 외부 텍스트로 빼 재컴파일 없이 갈아끼움.

Q속에선 무슨 일이 일어나는데? (메커니즘)

-transform-interpreter는 진입점 @__transform_main을 찾아 그 안의 transform op을 위에서 아래로 실행한다. 각 transform op은 입력 handle이 가리키는 payload op들에 자기 변환을 적용하고, 결과(새 op·loop)를 가리키는 출력 handle을 만든다.

  • 핵심 동작
    • handle ↔ payload 매핑: interpreter가 "이 SSA handle = 이 payload op들"이라는 표를 유지하며 갱신.
    • consume / invalidate: 변환이 payload op을 없애거나 바꾸면 그걸 가리키던 handle은 무효가 된다. 소비된 handle을 또 쓰면 interpreter가 에러.
    • 1:N handle: 한 handle이 여러 payload op을 가리킬 수 있다(match가 여러 개 잡음). split_handle/foreach로 다룸.
    • 실패 종류: silenceable(복구 가능 — alternatives에서 다음 시도) vs definite(즉시 중단).
P1의 패스가 greedy driver나 dialect conversion으로 pattern을 적용했다면(0008 참고), transform은 interpreter가 schedule을 순차 해석해 적용한다. 변환 기계(tiling 알고리즘)는 P1과 같은 것을 재사용 — transform op이 그 위의 얇은 래퍼다. (내부 해부는 P2 후반 cycle에서.)
Q안 되는 경우는? (no-op·실패)
상황무슨 일
match가 아무것도 못 찾음빈 handle. 이후 변환은 0개 대상에 적용(대개 무해) — 또는 matchFailure로 schedule 실패하게 설정 가능
이미 소비된 handle 재사용interpreter 에러 ("handle invalidated") — 변환이 payload를 바꾸면 옛 handle 무효
적용 불가한 변환(전제 위반)silenceable failure → alternatives면 다음 schedule 시도, 아니면 definite로 중단
payload가 변환의 가정과 다름예: buffer semantics 요구하는 변환에 tensor — 해당 변환이 실패(P1의 no-op 가드와 같은 결).

요지: transform은 실패를 1급 개념으로 다뤄, "되는 schedule을 고르는" 로직(alternatives·matcher)을 schedule 안에 짤 수 있다.

Q한 줄로 정리하면?
transform dialect = 변환을 IR(데이터)로 외부에서 프로그래밍하는 것. payload(무엇을)와 schedule(어떻게)을 분리하고, schedule을 텍스트로 갈아끼워 재빌드 없이 tile/fuse/vectorize/병렬화를 자유롭게 조합·튜닝한다. 그래서 autotuning·하드웨어별 최적화·재현가능한 레시피가 가능해진다 — 이게 "변환을 컴파일러 밖에서 정의·적용"의 이점이다.

다음(P2 cycle): T02 같은 변환 두 방식 대조 → match/handle → tile → fuse → pad·pack → vectorize → bufferize → 풀 kernel schedule(autotuning 스케치) → 제어흐름 → interpreter 해부 → 커스텀 transform op. payload는 네가 아는 linalg op 그대로.
