T11 — schedule sweep (autotuning 스케치)

payload(@matmul) 한 개와 바이너리(mlir-opt) 한 개를 고정한 채, transform script의 tile size·loop construct만 갈아끼워 4가지 IR을 뽑는다. schedule을 payload에서 분리해 외부 데이터로 다룰 수 있다는 것 — 즉 autotuning search의 가장 단순한 골격을 보여준다.

Q이 실험은 뭘 보여주나?

핵심 주장 하나: schedule(어떻게 변환할지)이 payload(무엇을 계산할지)에서 완전히 떨어져 나온다. transform dialect에서는 변환 절차 자체가 IR로 쓰여 있고(transform.named_sequence @__transform_main), mlir-opt --transform-interpreter가 그것을 해석해 payload를 고친다. 따라서 payload 파일과 바이너리를 한 글자도 안 바꾸고, transform script만 바꿔치우면 매번 다른 결과 IR이 나온다.

이 실험은 그 점을 네 개의 schedule을 같은 matmul에 sweep해서 실증한다. matmul payload는 다음 한 줄이다:

func.func @matmul(%A: tensor<128x256xf32>, %B: tensor<256x64xf32>,
                  %C: tensor<128x64xf32>) -> tensor<128x64xf32> {
  %0 = linalg.matmul ins(%A, %B : tensor<128x256xf32>, tensor<256x64xf32>)
                     outs(%C : tensor<128x64xf32>) -> tensor<128x64xf32>
  return %0 : tensor<128x64xf32>
}

iteration space는 (M, N, K) = (128, 64, 256)이다. 네 schedule:

변형transform optile sizes나오는 loop
a-8tile_using_for[8, 8, 8]scf.for × 3 (step 8)
b-32tile_using_for[32, 32, 32]scf.for × 3 (step 32, 구조 동일)
c-64x64x0tile_using_for[64, 64, 0]scf.for × 1 (K 미타일 → loop 축소)
d-foralltile_using_forall[64, 32]scf.forall (2,2) 병렬

왜 중요하냐면, 이것이 곧 autotuning의 골격이기 때문이다. autotuner는 결국 "schedule 후보를 생성 → 적용 → 측정 → 더 좋은 걸 채택"의 search loop인데, 여기서 "schedule 후보를 적용"하는 단계가 바로 이 실험이다. tile size를 외부 데이터(파일·파라미터)로 빼면 search space를 코드 수정 없이 훑을 수 있다.

Q어떻게 했나? (schedule)

각 schedule 파일은 같은 골격을 가진다: payload(@matmul)를 같은 module 안에 두고, @__transform_main이라는 named sequence가 그 payload를 변환한다. 흐름은 항상 match → tile 두 단계다.

a-8 / b-32 — tile_using_for (tile size만 다름):

transform.named_sequence @__transform_main(%arg0: !transform.any_op {transform.readonly}) {
  %mm = transform.structured.match ops{["linalg.matmul"]} in %arg0
        : (!transform.any_op) -> !transform.any_op
  %t, %l:3 = transform.structured.tile_using_for %mm tile_sizes [8, 8, 8]
        : (!transform.any_op) -> (!transform.any_op, !transform.any_op, !transform.any_op, !transform.any_op)
  transform.yield
}

읽는 법: match ops{["linalg.matmul"]}가 payload 안에서 linalg.matmul op을 찾아 transform handle %mm에 바인딩한다(이름이 아니라 op 종류로 잡는 게 핵심 — payload가 어떻게 생겼든 matmul이면 잡힌다). 그 handle을 tile_using_for에 넘기면 tiling이 적용된다. 결과로 tiled op handle %t와 생성된 loop handle %l:3(3개)이 돌아온다. b-32는 단 한 곳, tile_sizes [8,8,8] → [32,32,32]만 다르다.

c-64x64x0 — tile size 0 = 그 차원은 안 타일:

  // tile size 0 on K => that dim is not tiled, so only 2 loops are produced.
  %t, %l:2 = transform.structured.tile_using_for %mm tile_sizes [64, 64, 0]
        : (!transform.any_op) -> (!transform.any_op, !transform.any_op, !transform.any_op)

K(reduction) 차원에 0을 주면 그 축은 타일링하지 않는다. 따라서 loop이 3개가 아니라 2개만 생기고, 결과 handle도 %l:2로 받아야 한다(개수가 안 맞으면 verifier가 거부). 게다가 M(=128)을 64로 타일하면 quotient가 정확히 2이고 N(=64)을 64로 타일하면 quotient가 1이라, 뒤에서 보듯 canonicalize가 두 loop 중 trivial한 쪽을 더 정리해 결국 loop이 1개로 보일 수도 있다.

d-forall — tile_using_forall (loop construct 자체가 다름):

  %t, %f = transform.structured.tile_using_forall %mm tile_sizes [64, 32]
        : (!transform.any_op) -> (!transform.any_op, !transform.any_op)

tile_using_for가 순차 scf.for 중첩을 만드는 것과 달리, tile_using_forall은 단일 scf.forall(병렬 의미)을 만든다. tile size [64, 32]는 (M, N) 두 parallel 차원에만 주어진다(reduction K는 forall로 병렬화하지 않음). 반환 handle도 loop 여러 개가 아니라 forall op 하나(%f)다.

실행은 run.sh가 4개를 루프 돈다:

for s in a-8 b-32 c-64x64x0 d-forall; do
  "$MO" --transform-interpreter --canonicalize input/sched-$s.mlir > output/full-$s.mlir
  sed '/transform.named_sequence/,/^  }$/d' output/full-$s.mlir > output/out-$s.mlir
done

--transform-interpreter가 named sequence를 찾아 실행하고, --canonicalize가 정리한다. sed는 결과에서 transform script 부분을 떼어내 payload-only IR(out-*.mlir)만 남긴다.

Q결과가 어떻게 다른가? (before→after 실측)

가장 대조적인 두 쌍을 본다. 먼저 a-8 (3중 scf.for) — 표준 tiling 결과:

a-8 · tile_using_for [8,8,8]
before — payload (linalg.matmul 한 줄)
func.func @matmul(%arg0: tensor<128x256xf32>, %arg1: tensor<256x64xf32>,
                  %arg2: tensor<128x64xf32>) -> tensor<128x64xf32> {
  %0 = linalg.matmul
       ins(%arg0, %arg1 : tensor<128x256xf32>, tensor<256x64xf32>)
       outs(%arg2 : tensor<128x64xf32>) -> tensor<128x64xf32>
  return %0 : tensor<128x64xf32>
}
after — out-a-8.mlir (scf.for × 3, step 8)
%c256 = arith.constant 256 : index
%c64 = arith.constant 64 : index
%c0 = arith.constant 0 : index
%c128 = arith.constant 128 : index
%c8 = arith.constant 8 : index
%0 = scf.for %arg3 = %c0 to %c128 step %c8 iter_args(%arg4 = %arg2) -> (tensor<128x64xf32>) {
  %1 = scf.for %arg5 = %c0 to %c64 step %c8 iter_args(%arg6 = %arg4) -> (tensor<128x64xf32>) {
    %2 = scf.for %arg7 = %c0 to %c256 step %c8 iter_args(%arg8 = %arg6) -> (tensor<128x64xf32>) {
      %extracted_slice = tensor.extract_slice %arg0[%arg3, %arg7] [8, 8] [1, 1]
            : tensor<128x256xf32> to tensor<8x8xf32>
      %extracted_slice_0 = tensor.extract_slice %arg1[%arg7, %arg5] [8, 8] [1, 1]
            : tensor<256x64xf32> to tensor<8x8xf32>
      %extracted_slice_1 = tensor.extract_slice %arg8[%arg3, %arg5] [8, 8] [1, 1]
            : tensor<128x64xf32> to tensor<8x8xf32>
      %3 = linalg.matmul ins(%extracted_slice, %extracted_slice_0 : tensor<8x8xf32>, tensor<8x8xf32>)
            outs(%extracted_slice_1 : tensor<8x8xf32>) -> tensor<8x8xf32>
      %inserted_slice = tensor.insert_slice %3 into %arg8[%arg3, %arg5] [8, 8] [1, 1]
            : tensor<8x8xf32> into tensor<128x64xf32>
      scf.yield %inserted_slice : tensor<128x64xf32>
    }
    scf.yield %2 : tensor<128x64xf32>
  }
  scf.yield %1 : tensor<128x64xf32>
}
return %0 : tensor<128x64xf32>

3개 loop(M, N, K 순서로 step 8/8/8), 각 iteration에서 8×8 타일을 extract_slice로 잘라 linalg.matmul(8×8)을 수행하고 insert_slice로 되돌린다. iter_args로 출력 tensor가 loop를 통과한다(tensor SSA value의 in-place 갱신을 표현). b-32는 이 IR과 구조가 한 글자도 다르지 않고, 오직 step %c8이 step %c32로, slice 크기 [8,8]이 [32,32]로 바뀐다. 즉 a-8↔b-32 차이는 순수 파라미터 차이다.

다음은 가장 다른 형태 — d-forall (병렬 scf.forall):

d-forall · tile_using_forall [64,32]
before — payload (동일한 linalg.matmul)
%0 = linalg.matmul
     ins(%arg0, %arg1 : tensor<128x256xf32>, tensor<256x64xf32>)
     outs(%arg2 : tensor<128x64xf32>) -> tensor<128x64xf32>
after — out-d-forall.mlir (scf.forall, 2×2 병렬)
#map = affine_map<(d0) -> (d0 * 64)>
#map1 = affine_map<(d0) -> (d0 * 32)>
...
%0 = scf.forall (%arg3, %arg4) in (2, 2) shared_outs(%arg5 = %arg2) -> (tensor<128x64xf32>) {
  %1 = affine.apply #map(%arg3)
  %2 = affine.apply #map1(%arg4)
  %3 = affine.apply #map(%arg3)
  %4 = affine.apply #map1(%arg4)
  %extracted_slice = tensor.extract_slice %arg0[%1, 0] [64, 256] [1, 1]
        : tensor<128x256xf32> to tensor<64x256xf32>
  %extracted_slice_0 = tensor.extract_slice %arg1[0, %2] [256, 32] [1, 1]
        : tensor<256x64xf32> to tensor<256x32xf32>
  %extracted_slice_1 = tensor.extract_slice %arg5[%3, %4] [64, 32] [1, 1]
        : tensor<128x64xf32> to tensor<64x32xf32>
  %5 = linalg.matmul ins(%extracted_slice, %extracted_slice_0 : tensor<64x256xf32>, tensor<256x32xf32>)
        outs(%extracted_slice_1 : tensor<64x32xf32>) -> tensor<64x32xf32>
  %6 = affine.apply #map(%arg3)
  %7 = affine.apply #map1(%arg4)
  scf.forall.in_parallel {
    tensor.parallel_insert_slice %5 into %arg5[%6, %7] [64, 32] [1, 1]
          : tensor<64x32xf32> into tensor<128x64xf32>
  }
}
return %0 : tensor<128x64xf32>

완전히 다른 골격이다. scf.for 중첩 대신 단일 scf.forall (%arg3, %arg4) in (2, 2)가 2×2 = 4개의 독립 타일을 병렬로 표현한다(M/64 = 2, N/32 = 2). loop induction variable이 곧 타일 인덱스라, 실제 offset은 affine.apply #map(%arg3)(×64), #map1(%arg4)(×32)로 곱해 만든다. 출력은 iter_args(순차적 통과)가 아니라 shared_outs + tensor.parallel_insert_slice(서로 겹치지 않는 타일을 병렬로 써넣음, race 없음)로 표현된다. K 축은 forall에 없으므로 안쪽 matmul이 K 전체(256)를 한 번에 처리한다.

참고로 c-64x64x0은 tile size 0의 효과를 보여준다 — K를 안 타일하므로 안쪽 loop이 사라지고, N(=64)을 64로 타일하면 1조각이라 그 loop도 정리되어, 결국 M 방향 scf.for 1개만 남는다:

%0 = scf.for %arg3 = %c0 to %c128 step %c64 iter_args(%arg4 = %arg2) -> (tensor<128x64xf32>) {
  %extracted_slice = tensor.extract_slice %arg0[%arg3, 0] [64, 256] [1, 1] : tensor<128x256xf32> to tensor<64x256xf32>
  %extracted_slice_0 = tensor.extract_slice %arg4[%arg3, 0] [64, 64] [1, 1] : tensor<128x64xf32> to tensor<64x64xf32>
  %1 = linalg.matmul ins(%extracted_slice, %arg1 : tensor<64x256xf32>, tensor<256x64xf32>) outs(%extracted_slice_0 : tensor<64x64xf32>) -> tensor<64x64xf32>
  %inserted_slice = tensor.insert_slice %1 into %arg4[%arg3, 0] [64, 64] [1, 1] : tensor<64x64xf32> into tensor<128x64xf32>
  scf.yield %inserted_slice : tensor<128x64xf32>
}

B(%arg1)는 slice되지 않고 통째로(tensor<256x64xf32>) 안쪽 matmul에 들어가는 것을 보라 — K도 N도 타일되지 않았다는 증거다.

Q그래서 핵심은?

한 줄: schedule은 payload에서 분리된 외부 데이터이고, 그래서 search 대상이 된다. 네 변형이 각각 다른 결을 비춘다.

schedule = 데이터

payload·바이너리 고정, transform script만 교체 → 매번 다른 IR. tile size를 파라미터로 빼면 코드 수정 없이 search space를 훑는다. 이것이 autotuning의 최소 골격.

파라미터 vs 구조

a-8 ↔ b-32는 step/slice 크기만 다른 파라미터 변형(IR 골격 동일). c·d는 loop 개수·construct가 바뀌는 구조 변형. autotuning에서 둘 다 후보가 된다.

tile size 0 = skip

차원에 0을 주면 그 축은 타일하지 않는다 → loop 개수가 줄고, 결과 handle 개수(%l:N)도 따라 줄여 받아야 한다(verifier가 강제).

for vs forall

같은 tiling이라도 tile_using_for는 순차 scf.for+iter_args, tile_using_forall은 병렬 scf.forall+shared_outs/parallel_insert_slice. construct 선택이 곧 병렬화 전략 선택.

결정적으로, 이 모든 선택이 payload 코드 밖, 명시적 schedule IR에 적혀 있다. polyhedral(affine) 쪽이 dependence·legality를 ILP로 풀어 schedule을 자동 도출하는 것과 정반대 축이다 — transform dialect는 schedule을 사람(또는 autotuner)이 명시하고, 도구는 그걸 충실히 적용만 한다.

Q함정/주의는?
함정올바른 처리
tile size 0인데 loop handle을 3개로 받음0이 든 차원만큼 loop이 줄므로 %l:2처럼 개수를 맞춰 받는다 (개수 불일치 시 verifier 거부)
tile_using_for/forall handle 개수를 헷갈림for는 (tiled_op, loop×N), forall은 (tiled_op, forall 1개). result type 시그니처를 그에 맞춤
--canonicalize가 IR을 더 정리해버림c-64x64x0처럼 trivial loop(1조각)이 사라져 "기대보다 loop 적음"이 정상. canonicalize 효과를 결과 판독에 반영
forall 결과를 scf.for처럼 읽으려 함iter_args/scf.yield가 아니라 shared_outs/scf.forall.in_parallel+parallel_insert_slice로 출력이 표현됨
script가 payload를 이름으로 잡을 거라 가정match ops{["linalg.matmul"]}는 op 종류로 잡는다 — payload 함수명·변수명과 무관

실측 sanity check: run.sh 끝의 grep 카운트가 a-8 / b-32는 scf.for 3개씩, c-64x64x0은 1개, d-forall은 scf.forall 1개임을 확인한다. 이 숫자가 위 IR 판독과 일치한다.

Q한 줄 + 관련

한 줄: payload·바이너리를 고정하고 transform script만 sweep하면 같은 matmul에서 step만 다른 변형(a-8/b-32), loop이 줄어든 변형(c, tile 0), 병렬 construct로 바뀐 변형(d, forall)이 줄줄이 나온다 — schedule이 외부 데이터라는 것, 곧 autotuning search의 최소 골격이다.

관련 / 다음:

  • T11 ← T(0011) transform-tile-first: tiling을 transform script로 처음 적용한 실험. 본 실험은 그 schedule을 변수로 다뤄 여러 개를 sweep한 확장판.
  • 0012 pass-vs-transform-script: 같은 변환을 고정 pass로 거는 길과 명시 schedule IR로 거는 길의 대비. 본 실험은 "명시 schedule" 쪽을 데이터화한 셈.
  • affine-mlir baseline 대조: polyhedral은 dependence/legality를 ILP로 풀어 schedule을 자동 도출. transform dialect는 schedule을 명시·sweep. autotuner는 후자의 search loop 위에 측정·선택을 얹은 것 — 다음 단계로 결과 IR을 vectorize/bufferize까지 내려 실제 cost를 측정하는 파이프라인을 붙이면 search loop이 닫힌다.
