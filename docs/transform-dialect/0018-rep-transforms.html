T08 — 표현 변환 (generalize·specialize·interchange·decompose)

데이터 흐름(semantic)은 그대로 두고 같은 연산을 어떤 op로 표기하느냐만 바꾸는 4가지 transform: named ↔ generic 상호 변환, generic의 iterator 순서 재작성, aggregate op의 기본 연산 시퀀스 해체.

Q이 실험은 뭘 보여주나?

tile/fuse/vectorize 같은 변환은 loop nest의 구조(반복 횟수·메모리 접근)를 바꾼다. 반면 이 실험의 4가지는 연산의 표현(representation) 자체만 바꾼다 — 계산하는 값은 1바이트도 달라지지 않는다. linalg는 같은 연산을 추상도가 다른 여러 형태로 적을 수 있기 때문에, 그 사이를 오가는 transform op들이 따로 있다.

transform op방향한 일
generalizenamed → genericlinalg.matmul → 동등한 linalg.generic (indexing_maps + 본문 명시)
specializegeneric → namedtrivial-copy linalg.generic → linalg.copy (역방향, 패턴 인식)
interchangegeneric → genericiterator 순서 [0,1] → [1,0], indexing_maps 재작성 (generic 전용)
decompose_interfaceaggregate → 기본 op 시퀀스linalg.softmax → fill/max/exp/sum/div generic 5개

payload(input/payload.mlir)에는 각 변환의 대상 함수가 하나씩 들어 있고, schedule 4개를 각각 같은 mlir-opt --transform-interpreter 바이너리로 돌려 4가지 결과를 얻었다. 핵심 질문은 "이 변환들이 IR을 정확히 어떻게 다시 쓰는가, 그리고 어느 방향이 자동/수동인가"다.

Q어떻게 했나? (schedule)

네 schedule 모두 동일한 골격이다: transform.structured.match로 대상 op를 op-name으로 잡고(!transform.any_op handle), 그 handle을 표현 변환 op에 넘긴다. payload op이 함수별로 하나씩이라 match가 곧바로 단일 핸들을 만든다.

generalize — matmul handle을 generalize에 넘긴다. 결과 handle은 새로 생긴 generic을 가리킨다.

%mm = transform.structured.match ops{["linalg.matmul"]} in %arg0 : (!transform.any_op) -> !transform.any_op
%g  = transform.structured.generalize %mm : (!transform.any_op) -> !transform.any_op

specialize — generalize의 역방향. generic handle을 specialize에 넘기면, 그 generic이 알려진 named op 패턴(여기선 trivial copy)과 일치할 때 named op로 되돌린다.

%g = transform.structured.match ops{["linalg.generic"]} in %arg0 : (!transform.any_op) -> !transform.any_op
%s = transform.structured.specialize %g : (!transform.any_op) -> !transform.any_op

interchange — generic handle에 iterator_interchange = [1, 0] attribute를 함께 준다. 결과를 capture하지 않는 형태로 호출했다(부수효과만 사용). interchange는 generic 전용이라, named op에 적용하면 wrong-op-kind 에러가 난다.

%g = transform.structured.match ops{["linalg.generic"]} in %arg0 : (!transform.any_op) -> !transform.any_op
transform.structured.interchange %g iterator_interchange = [1, 0] : (!transform.any_op) -> !transform.any_op

decompose_interface — softmax handle을 넘긴다. softmax는 AggregatedOpInterface를 구현하므로, 인터페이스가 정의한 기본 연산 시퀀스로 풀린다.

%sm = transform.structured.match ops{["linalg.softmax"]} in %arg0 : (!transform.any_op) -> !transform.any_op
%d  = transform.structured.decompose_interface %sm : (!transform.any_op) -> !transform.any_op
Q결과가 어떻게 다른가? (before→after 실측)

가장 대조적인 두 변환을 실측 IR로 본다 — named ↔ generic을 가장 명확히 보여주는 generalize와, aggregate op이 어떻게 풀리는지 보여주는 decompose.

generalize — named op이 generic으로 펼쳐진다
before — linalg.matmul (named, 본문 암묵)
func.func @mm(%A: tensor<64x128xf32>, %B: tensor<128x64xf32>, %init: tensor<64x64xf32>) -> tensor<64x64xf32> {
  %0 = linalg.matmul ins(%A, %B : tensor<64x128xf32>, tensor<128x64xf32>)
                     outs(%init : tensor<64x64xf32>) -> tensor<64x64xf32>
  return %0 : tensor<64x64xf32>
}
after — linalg.generic (maps·iterator·본문 명시)
#map  = affine_map<(d0, d1, d2) -> (d0, d2)>
#map1 = affine_map<(d0, d1, d2) -> (d2, d1)>
#map2 = affine_map<(d0, d1, d2) -> (d0, d1)>
func.func @mm(%arg0: tensor<64x128xf32>, %arg1: tensor<128x64xf32>, %arg2: tensor<64x64xf32>) -> tensor<64x64xf32> {
  %0 = linalg.generic {indexing_maps = [#map, #map1, #map2], iterator_types = ["parallel", "parallel", "reduction"]} ins(%arg0, %arg1 : tensor<64x128xf32>, tensor<128x64xf32>) outs(%arg2 : tensor<64x64xf32>) {
  ^bb0(%in: f32, %in_0: f32, %out: f32):
    %1 = arith.mulf %in, %in_0 : f32
    %2 = arith.addf %out, %1 : f32
    linalg.yield %2 : f32
  } -> tensor<64x64xf32>
  return %0 : tensor<64x64xf32>
}

matmul 한 줄에 압축돼 있던 정보가 모두 표면화됐다: 3개의 indexing_map((d0,d2)=A, (d2,d1)=B, (d0,d1)=C), iterator 3개(parallel, parallel, reduction — 마지막이 contraction 축 d2), 그리고 본문의 mulf+addf (multiply-accumulate). 즉 generalize는 "named op이 무엇을 의미했는지"를 빠짐없이 적어 내려 쓴 것이다.

decompose — softmax 한 줄이 fill/max/exp/sum/div 시퀀스로
before — linalg.softmax (aggregate)
func.func @sm(%arg0: tensor<2x16x32xf32>, %dst: tensor<2x16x32xf32>) -> tensor<2x16x32xf32> {
  %1 = linalg.softmax dimension(2) ins(%arg0 : tensor<2x16x32xf32>) outs(%dst : tensor<2x16x32xf32>) -> tensor<2x16x32xf32>
  return %1 : tensor<2x16x32xf32>
}
after — fill + 4 generic (max·exp·sum·div)
#map  = affine_map<(d0, d1, d2) -> (d0, d1, d2)>
#map1 = affine_map<(d0, d1, d2) -> (d0, d1)>
func.func @sm(%arg0: tensor<2x16x32xf32>, %arg1: tensor<2x16x32xf32>) -> tensor<2x16x32xf32> {
  %0 = tensor.empty() : tensor<2x16xf32>
  %cst = arith.constant -3.40282347E+38 : f32
  %1 = linalg.fill ins(%cst : f32) outs(%0 : tensor<2x16xf32>) -> tensor<2x16xf32>
  %2 = linalg.generic {indexing_maps = [#map, #map1], iterator_types = ["parallel", "parallel", "reduction"]} ins(%arg0 : tensor<2x16x32xf32>) outs(%1 : tensor<2x16xf32>) {
  ^bb0(%in: f32, %out: f32):
    %7 = arith.maxnumf %in, %out : f32
    linalg.yield %7 : f32
  } -> tensor<2x16xf32>
  %3 = linalg.generic {indexing_maps = [#map, #map1, #map], iterator_types = ["parallel", "parallel", "parallel"]} ins(%arg0, %2 : tensor<2x16x32xf32>, tensor<2x16xf32>) outs(%arg1 : tensor<2x16x32xf32>) {
  ^bb0(%in: f32, %in_1: f32, %out: f32):
    %7 = arith.subf %in, %in_1 : f32
    %8 = math.exp %7 : f32
    linalg.yield %8 : f32
  } -> tensor<2x16x32xf32>
  %cst_0 = arith.constant 0.000000e+00 : f32
  %4 = linalg.fill ins(%cst_0 : f32) outs(%0 : tensor<2x16xf32>) -> tensor<2x16xf32>
  %5 = linalg.generic {indexing_maps = [#map, #map1], iterator_types = ["parallel", "parallel", "reduction"]} ins(%3 : tensor<2x16x32xf32>) outs(%4 : tensor<2x16xf32>) {
  ^bb0(%in: f32, %out: f32):
    %7 = arith.addf %in, %out : f32
    linalg.yield %7 : f32
  } -> tensor<2x16xf32>
  %6 = linalg.generic {indexing_maps = [#map, #map1, #map], iterator_types = ["parallel", "parallel", "parallel"]} ins(%3, %5 : tensor<2x16x32xf32>, tensor<2x16xf32>) outs(%arg1 : tensor<2x16x32xf32>) {
  ^bb0(%in: f32, %in_1: f32, %out: f32):
    %7 = arith.divf %in, %in_1 : f32
    linalg.yield %7 : f32
  } -> tensor<2x16x32xf32>
  return %6 : tensor<2x16x32xf32>
}

softmax의 수치 안정 알고리즘이 그대로 IR로 펼쳐졌다: (1) fill -∞ → (2) dim 2 따라 row-max 구하기(maxnumf + reduction) → (3) x - max 후 exp → (4) fill 0 → dim 2 따라 합(addf + reduction) → (5) exp/sum 나눗셈. reduction 결과는 tensor<2x16xf32>로 dim 2가 사라진 shape이고, broadcast는 #map1 = (d0,d1,d2)->(d0,d1)로 표현된다. 나머지 두 변환(specialize·interchange)은 Q5 표로 요약한다.

Q그래서 핵심은?

표현 변환은 "같은 계산, 다른 추상도" 사이의 이동이다. 추상도가 높을수록(named·aggregate) 후속 분석/매칭이 쉽고, 낮을수록(generic·기본 op) 더 일반적인 변환(tile/fuse/vectorize)을 걸 수 있다. 4가지를 두 축으로 정리하면:

generalize ↔ specialize (가역 쌍)

named ↔ generic을 오간다. generalize는 항상 가능(named의 의미를 적어 내리면 됨). specialize는 패턴 인식 의존 — generic이 알려진 named op과 구조적으로 일치할 때만 성공한다. trivial copy generic → linalg.copy처럼.

interchange (generic 내부 재작성)

iterator 순서를 바꾸되 indexing_maps에 그 순열을 흡수시켜 데이터 흐름은 불변. (d0,d1)->(d0,d1)이 (d0,d1)->(d1,d0)으로. loop 구조를 정하기 전 단계의 논리적 순서 재배열이다. generic 전용.

decompose_interface (aggregate 해체)

AggregatedOpInterface가 정의한 "기본 연산으로의 분해 레시피"를 실행. softmax 1개 → fill 2 + generic 4(max·exp·sum·div). 한 번 풀리면 각 generic을 독립적으로 tile/fuse 가능.

변환op 개수 변화 (payload)가역성 / 조건
generalizematmul 1 → generic 1항상 가능 (의미를 명시화)
specializegeneric 1 → copy 1named 패턴 일치 시에만
interchangegeneric 1 → generic 1 (개수 불변)maps만 재작성, 가역
decomposesoftmax 1 → fill 2 + generic 4인터페이스 레시피, 단방향

실측 확인: run.sh의 grep 카운트가 이를 검증한다 — generalize 후 payload에 generic 1개(matmul 소멸), specialize 후 copy 1개(generic 소멸), interchange 후 transposed map (d0, d1) -> (d1, d0) 등장, decompose 후 generic 4개 + fill 2개(softmax 소멸).

Q함정/주의는?
함정주의
interchange를 named op(matmul 등)에 적용wrong-op-kind 에러. interchange는 linalg.generic 전용 — named op은 먼저 generalize한 뒤 걸어야 한다.
specialize가 "항상 generic→named"라고 가정패턴 인식 의존. generic이 알려진 named op과 구조적으로 일치할 때만 변환된다. 임의의 generic은 그대로 남는다 (실패가 아니라 no-op일 수 있음).
decompose가 generic을 만든다고 generalize와 헷갈림decompose는 aggregate → 여러 기본 op(op 개수 증가)이고, generalize는 1 op → 1 generic(개수 불변, 의미 명시화). 입력 인터페이스(AggregatedOpInterface vs LinalgOp)가 다르다.
interchange가 메모리 접근 순서를 바꾼다고 오해indexing_maps에 순열을 흡수하므로 계산 결과·접근 패턴은 불변. iterator 라벨의 논리적 순서만 재배열된다. 실제 loop 순서는 이후 tile/loop 변환에서 결정된다.
Q한 줄 + 관련

한 줄: 표현 변환 4종은 데이터 흐름을 보존한 채 추상도만 옮긴다 — generalize/specialize는 named↔generic 가역 쌍, interchange는 generic 내부 maps 재작성, decompose_interface는 aggregate op(softmax)을 기본 op 시퀀스로 해체한다.
