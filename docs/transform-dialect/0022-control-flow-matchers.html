T12 — 제어흐름 · 매처 (foreach·alternatives)

하나의 match가 잡은 op 들을 foreach로 하나씩 순회하고, alternatives로 "조건 검사 실패 시 다른 변환 경로"를 표현한다 — schedule IR 안에서 제어흐름을 명시적으로 쓰는 법.

Q이 실험은 뭘 보여주나?

지금까지의 schedule 은 "match 한 번 → transform op 한 번" 의 직선 흐름이었다. 그런데 실제 schedule 은 여러 op 에 같은 변환을 반복하거나, 조건에 따라 분기해야 할 때가 있다. transform dialect 는 이를 위한 두 개의 제어흐름 op 를 제공한다.

transform.foreach

match 가 잡은 핸들(여러 payload op 의 묶음)을 받아, 각 op 마다 region body 를 한 번씩 실행한다. C++ 의 for (auto op : matched) 에 해당. body 안에서 핸들이 다음 op 로 흘러가는 데이터플로우도 그대로 표현된다.

transform.alternatives

여러 변환 분기를 순서대로 시도하다가 처음으로 성공한 분기를 채택한다. 어떤 분기가 silenceable failure 를 내면 그 분기의 모든 부작용을 롤백하고 다음 분기로 폴백한다. C++ 의 try/catch + fallback 사슬에 가깝다.

payload 는 linalg.fill 한 개와 linalg.matmul 두 개(서로 다른 K 차원: 128, 96)가 섞인 함수다. 이 혼합 payload 위에서 match 의 선택 범위(전체 linalg vs matmul 만)와 foreach/alternatives 의 제어흐름이 결과 IR 에 어떻게 드러나는지 4개 schedule (a/b/c/d)로 대조한다.

payload — input/payload.mlir
fill 1개 + matmul 2개 (혼합)
func.func @multi(%A1: tensor<64x128xf32>, %B1: tensor<128x64xf32>,
                 %A2: tensor<64x96xf32>,  %B2: tensor<96x64xf32>,
                 %init: tensor<64x64xf32>) -> (tensor<64x64xf32>, tensor<64x64xf32>) {
  %c0 = arith.constant 0.0 : f32
  %filled = linalg.fill ins(%c0 : f32) outs(%init : tensor<64x64xf32>) -> tensor<64x64xf32>
  %0 = linalg.matmul ins(%A1, %B1 : tensor<64x128xf32>, tensor<128x64xf32>)
                     outs(%filled : tensor<64x64xf32>) -> tensor<64x64xf32>
  %1 = linalg.matmul ins(%A2, %B2 : tensor<64x96xf32>, tensor<96x64xf32>)
                     outs(%filled : tensor<64x64xf32>) -> tensor<64x64xf32>
  return %0, %1 : tensor<64x64xf32>, tensor<64x64xf32>
}
4개 schedule 이 노리는 변환
// a: match(all linalg) -> foreach { generalize }
//    fill+matmul x2 = 3개 모두 linalg.generic 으로

// b: match(matmul)     -> foreach { tile }
//    각 matmul 이 자기만의 scf.for 3중첩, fill 무관

// c: match(matmul)     -> foreach { tile; generalize }
//    foreach body 안에서 핸들 연쇄: tile -> 안쪽 generic

// d: match(func) -> alternatives { Alt1 실패 -> Alt2 }
//    Alt1 조건검사 실패 -> Alt2(foreach generalize) 폴백
Q어떻게 했나? (schedule)

schedule a — match(all linalg) → foreach { generalize }. interface{LinalgOp} 로 fill+matmul 모두(3개)를 한 핸들에 묶어 잡고, transform.foreach 가 그 묶음을 풀어 op 하나씩 body 에 넘긴다. body 는 매 op 마다 generalize 를 한 번 실행한다.

transform.named_sequence @__transform_main(%arg0: !transform.any_op {transform.readonly}) {
  // 모든 linalg structured op 매치 (fill + matmul x2 = 3개).
  %all = transform.structured.match interface{LinalgOp} in %arg0
    : (!transform.any_op) -> !transform.any_op
  // 매치된 각 op 에 대해 body 를 한 번씩 실행.
  transform.foreach %all : !transform.any_op {
  ^bb0(%op: !transform.any_op):
    transform.structured.generalize %op : (!transform.any_op) -> !transform.any_op
  }
  transform.yield
}

schedule b — match(matmul) → foreach { tile }. match 의 선택자를 ops{["linalg.matmul"]} 로 좁혀 matmul 2개만 잡는다. fill 은 핸들에 들어오지 않으므로 손대지 않는다. body 는 각 matmul 을 [32,32,32] 로 독립 tile 한다.

  %mms = transform.structured.match ops{["linalg.matmul"]} in %arg0
    : (!transform.any_op) -> !transform.any_op
  transform.foreach %mms : !transform.any_op {
  ^bb0(%mm: !transform.any_op):
    %tiled, %loops:3 = transform.structured.tile_using_for %mm tile_sizes [32, 32, 32]
      : (!transform.any_op) -> (!transform.any_op, !transform.any_op, !transform.any_op, !transform.any_op)
  }

schedule c — foreach body 안에서 변환 연쇄. body 가 두 step 으로 늘어난다: tile 이 돌려주는 핸들 %tiled (안쪽 32x32 matmul)를 같은 body 안에서 generalize 가 받는다. foreach body 내부의 SSA 데이터플로우가 그대로 동작함을 보인다.

  transform.foreach %mms : !transform.any_op {
  ^bb0(%mm: !transform.any_op):
    %tiled, %loops:3 = transform.structured.tile_using_for %mm tile_sizes [32, 32, 32]
      : (!transform.any_op) -> (!transform.any_op, !transform.any_op, !transform.any_op, !transform.any_op)
    // tile 된 안쪽 matmul(%tiled)을 generic 으로 generalize.
    transform.structured.generalize %tiled : (!transform.any_op) -> !transform.any_op
  }

schedule d — alternatives. transform.alternatives 는 IsolatedFromAbove scope(외부 핸들이 region 안으로 새지 않는 격리 영역)만 받으므로, scope 로 쓸 func.func 를 먼저 match 해서 operand 로 준다. 각 분기 region 은 %scope 인자로 그 함수를 다시 받아 자기 안에서 새로 매칭한다.

  • Alt1: fill 을 잡은 뒤 transform.match.operation_name %fill ["linalg.matmul"] 로 "matmul 인가?" 검사 → fill 이므로 silenceable failure → Alt1 전체가 롤백되고 그 뒤 generalize 는 실행되지 않는다.
  • Alt2: matmul 2개를 잡아 foreach 로 각각 generalize → 성공 → 이 분기가 채택된다.
  // alternatives 의 scope 로 쓸 func.func 를 매치.
  %func = transform.structured.match ops{["func.func"]} in %arg0
    : (!transform.any_op) -> !transform.any_op
  transform.alternatives %func : !transform.any_op {
  ^bb1(%scope: !transform.any_op):
    // Alt1: fill 을 잡아 "matmul 인가?" 검사 -> 거짓 -> silenceable fail -> 폴백.
    %fill = transform.structured.match ops{["linalg.fill"]} in %scope
      : (!transform.any_op) -> !transform.any_op
    transform.match.operation_name %fill ["linalg.matmul"] : !transform.any_op
    // (도달 불가) 만약 통과했다면 fill 을 generalize 했을 것.
    transform.structured.generalize %fill : (!transform.any_op) -> !transform.any_op
  }, {
  ^bb1(%scope: !transform.any_op):
    // Alt2: matmul 2개를 잡아 foreach 로 각각 generalize -> 성공.
    %mms = transform.structured.match ops{["linalg.matmul"]} in %scope
      : (!transform.any_op) -> !transform.any_op
    transform.foreach %mms : !transform.any_op {
    ^bb2(%mm: !transform.any_op):
      transform.structured.generalize %mm : (!transform.any_op) -> !transform.any_op
    }
  }

모두 mlir-opt --transform-interpreter 로 실제 적용한다. b/c 는 --canonicalize 를 함께 걸어 tile 후 잔여 IR 을 정리한다. run.sh 가 변환 후 transform.named_sequence 블록을 sed 로 지워 payload-only(out-*.mlir) 버전을 따로 남긴다 — 아래 after 는 그 payload-only 결과다.

Q결과가 어떻게 다른가? (before→after 실측)

schedule a — match 범위가 fill 까지 포함. interface{LinalgOp} 가 fill 도 잡으므로 foreach 는 op 3개를 순회한다. 결과는 linalg.generic 3개 — fill 은 (d0,d1)->() 스칼라 broadcast generic, matmul 2개는 reduction generic 으로 풀린다.

a — match(all linalg) → foreach { generalize }
before — payload (fill + matmul x2)
%filled = linalg.fill ins(%c0 : f32)
            outs(%init : tensor<64x64xf32>) -> tensor<64x64xf32>
%0 = linalg.matmul ins(%A1, %B1 : ...) outs(%filled : ...) -> ...
%1 = linalg.matmul ins(%A2, %B2 : ...) outs(%filled : ...) -> ...
after — out-a (generic 3개)
%0 = linalg.generic {indexing_maps = [#map, #map1], iterator_types = ["parallel", "parallel"]} ins(%cst : f32) outs(%arg4 : tensor<64x64xf32>) {
^bb0(%in: f32, %out: f32):
  linalg.yield %in : f32
} -> tensor<64x64xf32>
%1 = linalg.generic {indexing_maps = [#map2, #map3, #map4], iterator_types = ["parallel", "parallel", "reduction"]} ins(%arg0, %arg1 : tensor<64x128xf32>, tensor<128x64xf32>) outs(%0 : tensor<64x64xf32>) {
^bb0(%in: f32, %in_0: f32, %out: f32):
  %3 = arith.mulf %in, %in_0 : f32
  %4 = arith.addf %out, %3 : f32
  linalg.yield %4 : f32
} -> tensor<64x64xf32>
%2 = linalg.generic {indexing_maps = [#map2, #map3, #map4], iterator_types = ["parallel", "parallel", "reduction"]} ins(%arg2, %arg3 : tensor<64x96xf32>, tensor<96x64xf32>) outs(%0 : tensor<64x64xf32>) {
^bb0(%in: f32, %in_0: f32, %out: f32):
  %3 = arith.mulf %in, %in_0 : f32
  %4 = arith.addf %out, %3 : f32
  linalg.yield %4 : f32
} -> tensor<64x64xf32>

schedule b — match 범위가 matmul 만. foreach 가 matmul 2개만 순회하므로 linalg.fill 은 그대로 살아남고, 각 matmul 이 독립적으로 scf.for 3중첩(M·N·K = 64/32, 64/32, 128|96/32)으로 tile 된다. 두 번째 matmul 의 K 루프 상한이 %c96 으로 다른 점이 입력 차원 차이를 그대로 반영한다.

b — match(matmul) → foreach { tile [32,32,32] }
before — payload
%filled = linalg.fill ... -> tensor<64x64xf32>
%0 = linalg.matmul ins(%A1, %B1 : ...K=128...) outs(%filled) -> ...
%1 = linalg.matmul ins(%A2, %B2 : ...K=96...)  outs(%filled) -> ...
after — out-b (fill 보존 + scf.for 3중첩 x2)
%0 = linalg.fill ins(%cst : f32) outs(%arg4 : tensor<64x64xf32>) -> tensor<64x64xf32>
%1 = scf.for %arg5 = %c0 to %c64 step %c32 iter_args(%arg6 = %0) -> (tensor<64x64xf32>) {
  %3 = scf.for %arg7 = %c0 to %c64 step %c32 iter_args(%arg8 = %arg6) -> (tensor<64x64xf32>) {
    %4 = scf.for %arg9 = %c0 to %c128 step %c32 iter_args(%arg10 = %arg8) -> (tensor<64x64xf32>) {
      %extracted_slice = tensor.extract_slice %arg0[%arg5, %arg9] [32, 32] [1, 1] : tensor<64x128xf32> to tensor<32x32xf32>
      %extracted_slice_0 = tensor.extract_slice %arg1[%arg9, %arg7] [32, 32] [1, 1] : tensor<128x64xf32> to tensor<32x32xf32>
      %extracted_slice_1 = tensor.extract_slice %arg10[%arg5, %arg7] [32, 32] [1, 1] : tensor<64x64xf32> to tensor<32x32xf32>
      %5 = linalg.matmul ins(%extracted_slice, %extracted_slice_0 : tensor<32x32xf32>, tensor<32x32xf32>) outs(%extracted_slice_1 : tensor<32x32xf32>) -> tensor<32x32xf32>
      %inserted_slice = tensor.insert_slice %5 into %arg10[%arg5, %arg7] [32, 32] [1, 1] : tensor<32x32xf32> into tensor<64x64xf32>
      scf.yield %inserted_slice : tensor<64x64xf32>
    }
    scf.yield %4 : tensor<64x64xf32>
  }
  scf.yield %3 : tensor<64x64xf32>
}
// %2 = scf.for ... %c96 ... (두 번째 matmul, K=96) 동일 구조

schedule c 는 b 의 결과에 더해 안쪽 tensor<32x32xf32> matmul 까지 linalg.generic 으로 풀린다 (foreach body 의 핸들 연쇄 tile→generalize 가 동작한 증거). 즉 out-c = "scf.for 3중첩 x2 + 그 안쪽 matmul 2개가 generic". schedule d 의 결과는 out-a 와 거의 동일(matmul 2개만 generic, fill 보존)하다 — Alt1 이 실제로 실행됐다면 fill 도 건드렸을 텐데 fill 이 linalg.fill 로 그대로 남은 것이 Alt1 폴백의 직접 증거다.

d — match(func) → alternatives { Alt1 실패 → Alt2 } (out-d, payload-only)
before — payload (fill + matmul x2)
%filled = linalg.fill ... -> tensor<64x64xf32>
%0 = linalg.matmul ins(%A1, %B1 : ...) outs(%filled) -> ...
%1 = linalg.matmul ins(%A2, %B2 : ...) outs(%filled) -> ...
after — out-d (fill 보존! matmul 2개만 generic)
%0 = linalg.fill ins(%cst : f32) outs(%arg4 : tensor<64x64xf32>) -> tensor<64x64xf32>
%1 = linalg.generic {indexing_maps = [#map, #map1, #map2], iterator_types = ["parallel", "parallel", "reduction"]} ins(%arg0, %arg1 : tensor<64x128xf32>, tensor<128x64xf32>) outs(%0 : tensor<64x64xf32>) {
^bb0(%in: f32, %in_0: f32, %out: f32):
  %3 = arith.mulf %in, %in_0 : f32
  %4 = arith.addf %out, %3 : f32
  linalg.yield %4 : f32
} -> tensor<64x64xf32>
%2 = linalg.generic {indexing_maps = [#map, #map1, #map2], iterator_types = ["parallel", "parallel", "reduction"]} ins(%arg2, %arg3 : tensor<64x96xf32>, tensor<96x64xf32>) outs(%0 : tensor<64x64xf32>) {
^bb0(%in: f32, %in_0: f32, %out: f32):
  %3 = arith.mulf %in, %in_0 : f32
  %4 = arith.addf %out, %3 : f32
  linalg.yield %4 : f32
} -> tensor<64x64xf32>
Q그래서 핵심은?

제어흐름 op 의 동작과, 결과 IR 에서 그것을 어떻게 읽어내는지가 핵심이다.

match 범위 = 변환 대상

interface{LinalgOp}(a/d-Alt1) 는 fill 까지 잡고, ops{["linalg.matmul"]}(b/c/d-Alt2) 는 matmul 만 잡는다. foreach 는 잡힌 것만 순회하므로, fill 의 보존/변환 여부가 곧 match 선택자의 결과다.

foreach = 핸들 묶음 순회

한 핸들에 묶인 여러 payload op 를 op 단위로 풀어 body 를 반복 실행. body 안의 SSA(tile→generalize) 데이터플로우도 그대로 흐른다(schedule c).

alternatives = 폴백 사슬

분기를 순서대로 시도, 첫 성공을 채택. silenceable failure 가 난 분기는 부작용까지 롤백 — fill 이 그대로 남은 것이 Alt1 미적용의 증거.

IsolatedFromAbove scope

alternatives region 은 외부 핸들을 못 받는다. func.func(또는 각 matmul)를 operand 로 주고 region 인자 %scope 로 받아 내부에서 다시 match 해야 한다.

schedulematch 범위제어흐름결과 IR (out-*)

bops{matmul} (2개)foreach { tile }fill 보존 + scf.for 3중첩 x2
cops{matmul} (2개)foreach { tile; generalize }scf.for 3중첩 x2 + 안쪽 matmul 2개 generic
dfunc → Alt1 fill / Alt2 matmulalternatives + foreachfill 보존 + matmul 2개 generic (Alt2 채택)
Q함정/주의는?
함정옳은 방법
alternatives region 에 외부 핸들 직접 사용scope 를 operand 로 넘기고 %scope region 인자로 받아 내부 재매칭 (IsolatedFromAbove)
Alt1 의 generalize 가 부작용을 남길까 걱정match.operation_name 이 먼저 silenceable fail → 그 뒤 generalize 는 도달 불가, 분기 전체 롤백
foreach body 안에서 핸들이 안 흐를 것이라 가정tile 이 돌려준 %tiled 를 같은 body 의 generalize 가 받는 SSA 데이터플로우는 정상 동작 (schedule c)
match 선택자를 헷갈려 fill 을 의도치 않게 변환interface{LinalgOp} 는 fill 포함, ops{["linalg.matmul"]} 는 matmul 전용 — 대상 범위를 선택자로 명시
tile 결과 IR 이 지저분해 비교 어려움b/c 는 --canonicalize 동반 실행으로 잔여 정리 후 비교

silenceable failure 는 transform interpreter 가 "이 분기는 실패했지만 치명적이지 않다 — 폴백 가능"으로 처리하는 등급이다. 만약 어떤 분기가 definite failure 를 냈다면 alternatives 전체가 중단된다. d 에서 Alt1 이 폴백으로 흡수된 것은 match.operation_name 불일치가 silenceable 등급이기 때문이다.

Q한 줄 + 관련

한 줄: transform schedule 은 직선이 아니다 — foreach 로 match 묶음을 op 단위 반복하고, alternatives 로 조건 분기·폴백을 명시하며, 결과 IR 의 "무엇이 변하고 무엇이 보존됐나"(특히 fill 보존)가 곧 제어흐름의 실행 흔적이다.

관련: P1 cycle 의 T01 — tile_using_for 기본(여기서 본 scf.for 3중첩의 단일 버전)과 T02 — pass vs transform script(같은 효과를 pass 와 script 로 표현하는 대조)가 선행 맥락이다. polyhedral baseline(affine #0013 tile)에서는 tiling 이 ILP 로 완전 자동 결정됐던 것과 달리, 여기서는 어떤 op 를(match) 어떤 순서로(foreach) 어떤 조건에서(alternatives) 변환할지가 schedule IR 에 명시된다.

다음: alternatives 의 조건을 match.structured 계열(차원·연산 패턴 매칭)로 정교화하거나, foreach 안에서 tile→fuse→vectorize 사슬을 묶어 한 op 당 full pipeline 을 적용하는 schedule 로 확장.
