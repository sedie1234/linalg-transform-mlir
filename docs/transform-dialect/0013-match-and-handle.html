T03 — match & handle (핸들로 정밀 지목·탐색)

transform script가 payload op을 어떻게 "잡고(match)" 그 핸들을 어떻게 "탐색(navigate)"하는가 — op 이름으로 잡고, producer·parent·split로 정밀하게 좁혀, 정확히 그 op에만 변환을 거는 과정을 실측 IR로 증명한다.

Q이 실험은 뭘 보여주나?

transform dialect에서 변환을 거는 첫 단계는 항상 "어느 payload op에 걸 것인가"를 지목하는 것이다. 그 지목의 결과물이 handle(!transform.any_op 같은 SSA value)인데, handle은 IR 자체가 아니라 "payload op들의 집합을 가리키는 포인터"다. 이 실험은 핸들을 만들고(match) 다루는(navigate) 네 가지 핵심 연산을 한 자리에서 검증한다.

match

transform.structured.match ops{[...]} — op 이름으로 payload를 잡아 핸들 생성. 한 핸들이 여러 payload op을 가리킬 수 있다.

get_producer_of_operand

핸들이 가리키는 op의 N번째 피연산자를 생산하는 op으로 핸들을 옮긴다 (data-flow를 거슬러 올라감).

split_handle

여러 payload를 가리키는 핸들 하나를 IR 순서대로 singleton 핸들 여러 개로 쪼갠다.

get_parent_op

leaf op 핸들에서 그것을 감싸는 enclosing op(예: func.func)으로 핸들을 끌어올린다.

핵심 질문은 항상 "이 핸들은 지금 정확히 어떤 payload op을 가리키는가?"이고, 이 실험은 그것을 두 가지 방식으로 증명한다: (1) transform.print로 핸들이 가리키는 op을 그대로 덤프하고, (2) 그 핸들에만 후속 변환(generalize)을 걸어 오직 그 op만 바뀌는 것을 결과 IR로 확인한다.

대상 payload는 두 함수다. @mm은 linalg.fill(producer) → linalg.matmul(consumer) 체인 — matmul의 outs 피연산자(#2)가 fill의 출력이다. @two_gen은 linalg.generic 두 개(add, mul)로 split_handle 데모용이다.

Q어떻게 했나? (schedule)

세 개의 transform script(A·B·C)를 각각 mlir-opt --transform-interpreter로 돌렸다. 모두 transform.named_sequence @__transform_main이 진입점이고, 첫 인자 %arg0이 payload root(module)를 가리키는 핸들이다.

A — match + print: op 이름으로 fill과 matmul을 따로 잡아 각 핸들을 그대로 출력한다. payload는 건드리지 않는다(순수 지목 증명).

transform.named_sequence @__transform_main(%arg0: !transform.any_op {transform.readonly}) {
  %fill = transform.structured.match ops{["linalg.fill"]} in %arg0 : (!transform.any_op) -> !transform.any_op
  %mm   = transform.structured.match ops{["linalg.matmul"]} in %arg0 : (!transform.any_op) -> !transform.any_op
  transform.print %fill {name = "FILL handle"} : !transform.any_op
  transform.print %mm   {name = "MATMUL handle"} : !transform.any_op
  transform.yield
}

B — producer 탐색: matmul만 잡은 뒤, get_producer_of_operand %mm[2]로 matmul의 outs 피연산자(#2 = %filled)를 생산하는 op으로 핸들을 옮긴다. 그 producer 핸들에만 generalize를 건다 — 핸들이 정말 fill을 가리킨다면 fill만 linalg.generic으로 바뀌고 matmul은 named op 그대로여야 한다.

transform.named_sequence @__transform_main(%arg0: !transform.any_op {transform.readonly}) {
  %mm = transform.structured.match ops{["linalg.matmul"]} in %arg0 : (!transform.any_op) -> !transform.any_op
  // operand #2 of linalg.matmul is the outs tensor, produced by linalg.fill.
  %producer = transform.get_producer_of_operand %mm[2]
    : (!transform.any_op) -> !transform.any_op
  transform.print %producer {name = "producer of matmul outs (expect linalg.fill)"} : !transform.any_op
  %g = transform.structured.generalize %producer : (!transform.any_op) -> !transform.any_op
  transform.yield
}

C — split + parent: 한 번의 match로 두 개의 generic을 모두 잡은 핸들을 split_handle로 IR 순서대로 두 singleton 핸들(#0 = add, #1 = mul)로 쪼갠다. 그 후 split #1(mul)에서 get_parent_op {op_name = "func.func"}로 감싸는 함수까지 끌어올린다. 각 단계마다 print로 핸들이 가리키는 대상을 확인한다.

transform.named_sequence @__transform_main(%arg0: !transform.any_op {transform.readonly}) {
  // one handle -> two payload generics
  %gens = transform.structured.match ops{["linalg.generic"]} in %arg0 : (!transform.any_op) -> !transform.any_op
  transform.print %gens {name = "both generics (single handle, 2 payloads)"} : !transform.any_op
  // split into two singleton handles, in IR order: #0 = add, #1 = mul
  %first, %second = transform.split_handle %gens
    : (!transform.any_op) -> (!transform.any_op, !transform.any_op)
  transform.print %first  {name = "split #0 (expect addf generic)"} : !transform.any_op
  transform.print %second {name = "split #1 (expect mulf generic)"} : !transform.any_op
  // climb from a leaf op up to its enclosing func.func
  %parent = transform.get_parent_op %second {op_name = "func.func"} : (!transform.any_op) -> !transform.any_op
  transform.print %parent {name = "get_parent_op of split #1 (expect func.func @two_gen)"} : !transform.any_op
  transform.yield
}

흐름은 항상 match → (navigate) → print/transform이다. match는 핸들의 출발점이고, navigate op들이 그 핸들을 데이터플로우(producer)·구조(parent)·집합(split)을 따라 이동시키며, print 또는 후속 변환이 그 위치를 검증한다.

Q결과가 어떻게 다른가? (before→after 실측)

A (print only): payload는 한 글자도 바뀌지 않는다 — match·print는 순수 조회다. 다만 interpreter가 핸들이 가리키는 op을 stderr로 덤프한다(아래 print 증거). before/after가 동일하므로 여기서는 print 덤프만 본다.

[[[ IR printer: FILL handle ]]]
%0 = linalg.fill ins(%cst : f32) outs(%arg2 : tensor<64x64xf32>) -> tensor<64x64xf32>
[[[ IR printer: MATMUL handle ]]]
%1 = linalg.matmul ins(%arg0, %arg1 : tensor<64x128xf32>, tensor<128x64xf32>) outs(%0 : tensor<64x64xf32>) -> tensor<64x64xf32>

두 핸들이 각각 fill 한 개, matmul 한 개를 정확히 가리킨다.

B — producer 핸들에만 generalize
before — @mm payload (fill + matmul 체인)
func.func @mm(%arg0: tensor<64x128xf32>, %arg1: tensor<128x64xf32>, %arg2: tensor<64x64xf32>) -> tensor<64x64xf32> {
  %cst = arith.constant 0.000000e+00 : f32
  %0 = linalg.fill ins(%cst : f32) outs(%arg2 : tensor<64x64xf32>) -> tensor<64x64xf32>
  %1 = linalg.matmul ins(%arg0, %arg1 : tensor<64x128xf32>, tensor<128x64xf32>) outs(%0 : tensor<64x64xf32>) -> tensor<64x64xf32>
  return %1 : tensor<64x64xf32>
}
after — fill만 generic으로, matmul은 named 그대로
func.func @mm(%arg0: tensor<64x128xf32>, %arg1: tensor<128x64xf32>, %arg2: tensor<64x64xf32>) -> tensor<64x64xf32> {
  %cst = arith.constant 0.000000e+00 : f32
  %0 = linalg.generic {indexing_maps = [affine_map<(d0, d1) -> ()>, affine_map<(d0, d1) -> (d0, d1)>], iterator_types = ["parallel", "parallel"]} ins(%cst : f32) outs(%arg2 : tensor<64x64xf32>) {
  ^bb0(%in: f32, %out: f32):
    linalg.yield %in : f32
  } -> tensor<64x64xf32>
  %1 = linalg.matmul ins(%arg0, %arg1 : tensor<64x128xf32>, tensor<128x64xf32>) outs(%0 : tensor<64x64xf32>) -> tensor<64x64xf32>
  return %1 : tensor<64x64xf32>
}

핸들은 matmul에서 출발했지만 [2]의 producer로 이동했기 때문에 generalize가 fill에만 적용됐다. matmul은 손대지 않은 채 named op으로 남았다 — 이것이 "핸들이 producer를 가리킨다"는 byte-level 증거다. print 덤프도 같은 결론을 준다:

[[[ IR printer: producer of matmul outs (expect linalg.fill) ]]]
%0 = linalg.fill ins(%cst : f32) outs(%arg2 : tensor<64x64xf32>) -> tensor<64x64xf32>
C — split_handle 가 가리키는 대상 (print 증거)
before — 한 핸들 %gens 가 두 generic 모두 가리킴
[[[ IR printer: both generics (single handle, 2 payloads) ]]]
%1 = linalg.generic {...} ins(%arg0, %arg1 ...) ... {
^bb0(%in: f32, %in_0: f32, %out: f32):
  %4 = arith.addf %in, %in_0 : f32
  linalg.yield %4 : f32
} -> tensor<32xf32>
%3 = linalg.generic {...} ins(%1, %arg1 ...) ... {
^bb0(%in: f32, %in_0: f32, %out: f32):
  %4 = arith.mulf %in, %in_0 : f32
  linalg.yield %4 : f32
} -> tensor<32xf32>
after — split 후 #0=add, #1=mul, parent=func.func
[[[ IR printer: split #0 (expect addf generic) ]]]
%1 = linalg.generic {...} ins(%arg0, %arg1 ...) ... {
  %4 = arith.addf %in, %in_0 : f32
  ...
}
[[[ IR printer: split #1 (expect mulf generic) ]]]
%3 = linalg.generic {...} ins(%1, %arg1 ...) ... {
  %4 = arith.mulf %in, %in_0 : f32
  ...
}
[[[ IR printer: get_parent_op of split #1 (expect func.func @two_gen) ]]]
func.func @two_gen(%arg0: tensor<32xf32>, %arg1: tensor<32xf32>) -> ... {
  ... (전체 함수 본문) ...
}

한 match 핸들이 두 payload를 가리키다가(2-payload), split로 정확히 add·mul singleton으로 갈라지고, mul의 parent가 함수 전체로 끌어올려졌다. C의 payload IR 자체는 후속 변환을 걸지 않았으므로 out-c-split-parent.mlir은 입력과 동일하다 — split/parent는 핸들 그래프만 움직일 뿐 payload를 건드리지 않는다.

Q그래서 핵심은?

handle은 "transform IR이 들고 다니는 payload op으로의 포인터"다. match로 만들고, navigate op으로 옮기며, 변환은 핸들이 현재 가리키는 곳에만 적용된다. 그래서 "정밀 지목 → 후속 변환"이 transform dialect의 기본 작업 단위다.

match는 집합을 만든다

op 이름 하나로 잡으면 그 이름의 payload op이 여러 개면 핸들도 여러 개를 가리킨다. C의 %gens가 generic 2개를 동시에 가리킨 것이 그 증거.

변환은 핸들이 가리키는 곳에만

B에서 핸들을 producer로 옮긴 뒤 generalize → matmul은 손대지 않고 fill만 generic화. 핸들 위치 = 변환 범위.

navigate = 데이터플로우·구조 탐색

producer는 SSA def-use를 거슬러(operand→정의 op), parent는 region 중첩을 거슬러(op→enclosing op) 핸들을 옮긴다. polyhedral의 "access를 따라가는" 것과 달리 IR 그래프를 직접 항해한다.

print는 부작용 없는 디버거

transform.print는 payload를 바꾸지 않고 핸들이 가리키는 op을 stderr로 덤프 — schedule 디버깅의 printf.

navigate op핸들 이동 방향본 실험 결과
get_producer_of_operand %h[N]op → N번째 operand를 생산하는 op (data-flow 역방향)matmul[2] → linalg.fill
split_handle %h1 핸들(K payload) → K 핸들(각 1 payload), IR 순서generic 2개 → #0 add, #1 mul
get_parent_op %h {op_name}op → 그것을 감싸는 enclosing op (구조 상방)mul generic → func.func @two_gen
Q함정/주의는?

operand 인덱스는 0-based 전체 operand

matmul의 outs를 가리키는 [2]는 ins 2개(A=#0, B=#1) 다음의 outs(#2)다. ins/outs 합쳐 센 절대 인덱스이므로, 잘못 세면 엉뚱한 producer로 간다.

split_handle은 IR 순서에 의존

#0/#1 배정은 payload op이 함수 안에 나타나는 순서다. 함수 본문 순서가 바뀌면 split 인덱스 의미도 바뀐다 — 위치가 아니라 op 속성으로 잡으려면 별도 match filter가 필요.

print는 stderr·interpreter 출력에 섞인다

이 실험의 run.sh는 awk/sed로 transform interpreter 덤프(상단 print 블록)와 self-contained payload(out-*.mlir)를 분리한다. --mlir-print-local-scope로 #map alias를 인라인해 payload만 떼어냈다.

match가 0개를 잡으면 핸들은 비어 있다

op 이름 오타·존재하지 않는 op면 빈 핸들이 되고, 후속 변환은 (op에 따라) 조용히 no-op이거나 에러. print로 잡힌 개수를 먼저 확인하는 습관이 안전하다.

Q한 줄 + 관련

한 줄: transform dialect의 변환은 항상 handle = payload op 포인터 위에서 일어나며, match로 잡고 producer·parent·split로 정밀하게 옮긴 뒤 그 핸들에만 변환을 건다 — B에서 matmul[2]→fill로 옮긴 핸들에 generalize를 걸어 fill만 generic이 된 것이 그 핵심 증거다.

관련: 같은 transform dialect cycle의 T01 — transform.structured.tile (첫 schedule), T02 — pass vs transform script. T01/T02가 "변환을 어떻게 표현하나"였다면 T03은 그 변환이 걸릴 대상을 어떻게 정밀 지목하나를 채운다.

다음: 잡은 핸들에 tile/fuse/promote 같은 실제 변환을 연쇄로 거는 schedule, 그리고 match를 op 이름이 아니라 속성·indexing_map·iterator type으로 필터링하는 정밀 match 패턴.
