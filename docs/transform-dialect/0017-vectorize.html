T07 — vectorize (linalg → vector dialect)

static-shape linalg op 을 vector dialect 로 lowering 한다. 같은 payload 에 vectorizer 만 바꿔(structured.vectorize vs vectorize_children_and_apply_patterns) multi_reduction 형태와 vector.contract 형태가 어떻게 갈리는지를 IR 레벨로 대조한다.

Q이 실험은 뭘 보여주나?

vectorize 는 structured op(linalg.matmul, linalg.generic …)의 high-level 반복 구조를 고정 크기 vector 연산으로 바꾸는 transform op 이다. iteration domain 이 정적이어야(static shape) 컴파일 타임에 vector 폭이 정해지므로 — payload 는 모두 static shape(tensor<8x16xf32> 등)이다.

핵심 대조 포인트는 "vectorizer 가 두 종류 있고, 같은 matmul 을 서로 다른 vector op 으로 떨어뜨린다" 는 점이다:

A — structured.vectorize

정밀 vectorizer. op 을 직접 매치해 in-place 로 vectorize. matmul 은 mulf + vector.multi_reduction 의 elementwise+reduction 분해형으로 떨어진다.

B — vectorize_children_and_apply_patterns

일괄 vectorizer + canonical vector pattern 적용. matmul 은 곧바로 vector.contract 라는 단일 contraction op으로 떨어진다.

C — generic(elementwise)

contraction 이 없는 순수 elementwise. A 와 같은 op 을 써도 transfer_read + arith.addf <vector> + transfer_write 의 단순 형태.

즉 이 실험은 (1) linalg → vector 의 일반적 lowering 골격(transfer_read … transfer_write)을 보여주고, (2) 같은 connection 을 contraction-분해 vs contraction-유지 로 표현하는 두 vectorizer 의 차이를 박아둔다.

Q어떻게 했나? (schedule)

세 schedule 모두 transform.structured.match 로 target op 을 잡는 데서 시작한다. 핵심 차이는 잡은 뒤 무엇에 vectorize 를 거는가 다.

A — structured.vectorize 를 op 에 직접: matmul 을 매치해 그 op 자체에 vectorize. result type 이 없다(in-place 변환, handle 소비 X).

transform.named_sequence @__transform_main(%arg0: !transform.any_op {transform.readonly}) {
  %mm = transform.structured.match ops{["linalg.matmul"]} in %arg0 : (!transform.any_op) -> !transform.any_op
  transform.structured.vectorize %mm : !transform.any_op
  transform.yield
}

B — 부모 func 을 잡아 children 일괄 vectorize: matmul 을 매치한 뒤 get_parent_op {isolated_from_above} 로 그 op 을 감싼 func.func 를 얻는다(이 vectorizer 는 isolated-from-above target 이 필요). 그 func 안의 모든 linalg child 를 vectorize 하고 canonical vector pattern 까지 적용한다. 이쪽은 새 handle 을 반환한다(-> !transform.any_op).

transform.named_sequence @__transform_main(%arg0: !transform.any_op {transform.readonly}) {
  %mm = transform.structured.match ops{["linalg.matmul"]} in %arg0 : (!transform.any_op) -> !transform.any_op
  %func = transform.get_parent_op %mm {isolated_from_above} : (!transform.any_op) -> !transform.any_op
  %v = transform.structured.vectorize_children_and_apply_patterns %func : (!transform.any_op) -> !transform.any_op
  transform.yield
}

C — generic 을 직접 vectorize: A 와 동일한 op(structured.vectorize)을 linalg.generic(elementwise)에 건다. contraction 이 없으니 분해할 reduction 도 없다.

transform.named_sequence @__transform_main(%arg0: !transform.any_op {transform.readonly}) {
  %g = transform.structured.match ops{["linalg.generic"]} in %arg0 : (!transform.any_op) -> !transform.any_op
  transform.structured.vectorize %g : !transform.any_op
  transform.yield
}

실행은 셋 다 mlir-opt --transform-interpreter --canonicalize 로 동일 — 차이는 오직 script(vectorizer 선택)뿐이다.

Q결과가 어떻게 다른가? (before→after 실측)

가장 대조적인 한 쌍 — 같은 linalg.matmul 을 A vs B 로 vectorize. 입력 payload 는 동일하다:

func.func @matmul(%A: tensor<8x16xf32>, %B: tensor<16x4xf32>,
                  %C: tensor<8x4xf32>) -> tensor<8x4xf32> {
  %0 = linalg.matmul ins(%A, %B : tensor<8x16xf32>, tensor<16x4xf32>)
                     outs(%C : tensor<8x4xf32>) -> tensor<8x4xf32>
  return %0 : tensor<8x4xf32>
}
matmul: A (structured.vectorize) vs B (vectorize_children_and_apply_patterns)
A — multi_reduction 분해형 (out-a-vectorize.mlir)
#map = affine_map<(d0, d1) -> (d0, 0, d1)>
#map1 = affine_map<(d0, d1) -> (0, d1, d0)>
func.func @matmul(%arg0: tensor<8x16xf32>, %arg1: tensor<16x4xf32>, %arg2: tensor<8x4xf32>) -> tensor<8x4xf32> {
  %c0 = arith.constant 0 : index
  %cst = arith.constant 0.000000e+00 : f32
  %0 = vector.transfer_read %arg0[%c0, %c0], %cst {in_bounds = [true, true, true], permutation_map = #map} : tensor<8x16xf32>, vector<8x4x16xf32>
  %1 = vector.transfer_read %arg1[%c0, %c0], %cst {in_bounds = [true, true, true], permutation_map = #map1} : tensor<16x4xf32>, vector<8x4x16xf32>
  %2 = vector.transfer_read %arg2[%c0, %c0], %cst {in_bounds = [true, true]} : tensor<8x4xf32>, vector<8x4xf32>
  %3 = arith.mulf %0, %1 : vector<8x4x16xf32>
  %4 = vector.multi_reduction <add>, %3, %2 [2] : vector<8x4x16xf32> to vector<8x4xf32>
  %5 = vector.transfer_write %4, %arg2[%c0, %c0] {in_bounds = [true, true]} : vector<8x4xf32>, tensor<8x4xf32>
  return %5 : tensor<8x4xf32>
}
B — vector.contract 단일형 (out-b-children-patterns.mlir)
#map = affine_map<(d0, d1, d2) -> (d0, d2)>
#map1 = affine_map<(d0, d1, d2) -> (d2, d1)>
#map2 = affine_map<(d0, d1, d2) -> (d0, d1)>
func.func @matmul(%arg0: tensor<8x16xf32>, %arg1: tensor<16x4xf32>, %arg2: tensor<8x4xf32>) -> tensor<8x4xf32> {
  %c0 = arith.constant 0 : index
  %cst = arith.constant 0.000000e+00 : f32
  %0 = vector.transfer_read %arg0[%c0, %c0], %cst {in_bounds = [true, true]} : tensor<8x16xf32>, vector<8x16xf32>
  %1 = vector.transfer_read %arg1[%c0, %c0], %cst {in_bounds = [true, true]} : tensor<16x4xf32>, vector<16x4xf32>
  %2 = vector.transfer_read %arg2[%c0, %c0], %cst {in_bounds = [true, true]} : tensor<8x4xf32>, vector<8x4xf32>
  %3 = vector.contract {indexing_maps = [#map, #map1, #map2], iterator_types = ["parallel", "parallel", "reduction"], kind = #vector.kind<add>} %0, %1, %2 : vector<8x16xf32>, vector<16x4xf32> into vector<8x4xf32>
  %4 = vector.transfer_write %3, %arg2[%c0, %c0] {in_bounds = [true, true]} : vector<8x4xf32>, tensor<8x4xf32>
  return %4 : tensor<8x4xf32>
}

두 결과의 차이를 정리하면:

A — structured.vectorizeB — children + patterns
핵심 oparith.mulf + vector.multi_reduction <add>vector.contract (단일)
read 형태A,B 를 permutation_map 으로 vector<8x4x16xf32> 로 broadcast-readA,B 를 그대로 2D vector<8x16>·vector<16x4> read
reduction 표현3D 곱 후 축 [2] 를 명시적 reduceiterator_types 의 "reduction" 으로 contract 내부에 은닉
추상화 레벨낮음 — 곱·합으로 풀어헤침높음 — contraction 의미를 한 op 으로 유지

대조용으로 C — elementwise generic 도 같이 본다. contraction 이 없으니 read–compute–write 가 가장 깔끔하게 떨어진다:

add_elemwise: linalg.generic → vector
before — payload (linalg.generic)
#map = affine_map<(d0, d1) -> (d0, d1)>
func.func @add_elemwise(%A: tensor<8x16xf32>, %B: tensor<8x16xf32>,
                        %C: tensor<8x16xf32>) -> tensor<8x16xf32> {
  %0 = linalg.generic {
         indexing_maps = [#map, #map, #map],
         iterator_types = ["parallel", "parallel"]
       } ins(%A, %B : tensor<8x16xf32>, tensor<8x16xf32>)
         outs(%C : tensor<8x16xf32>) {
    ^bb0(%a: f32, %b: f32, %out: f32):
      %s = arith.addf %a, %b : f32
      linalg.yield %s : f32
  } -> tensor<8x16xf32>
  return %0 : tensor<8x16xf32>
}
after — C (out-c-elementwise.mlir)
func.func @add_elemwise(%arg0: tensor<8x16xf32>, %arg1: tensor<8x16xf32>, %arg2: tensor<8x16xf32>) -> tensor<8x16xf32> {
  %c0 = arith.constant 0 : index
  %cst = arith.constant 0.000000e+00 : f32
  %0 = vector.transfer_read %arg0[%c0, %c0], %cst {in_bounds = [true, true]} : tensor<8x16xf32>, vector<8x16xf32>
  %1 = vector.transfer_read %arg1[%c0, %c0], %cst {in_bounds = [true, true]} : tensor<8x16xf32>, vector<8x16xf32>
  %2 = arith.addf %0, %1 : vector<8x16xf32>
  %3 = vector.transfer_write %2, %arg2[%c0, %c0] {in_bounds = [true, true]} : vector<8x16xf32>, tensor<8x16xf32>
  return %3 : tensor<8x16xf32>
}

스칼라 region(^bb0 … arith.addf %a, %b : f32)이 사라지고 그 자리에 arith.addf %0, %1 : vector<8x16xf32> 한 줄이 들어간다 — 같은 arith.addf 가 scalar 에서 vector 로 승격된 것. iterator 가 전부 parallel 이라 reduction 표현이 아예 안 생긴다는 게 matmul 과의 본질적 차이다.

Q그래서 핵심은?

vectorize 는 단일 결과가 아니라 vectorizer 선택에 따라 도착 IR 이 갈리는 변환이다. 같은 matmul 을 두고:

분해형 (A) vs contraction형 (B)

structured.vectorize 는 mulf+multi_reduction 으로 풀어헤치고, vectorize_children_and_apply_patterns 는 canonical pattern 까지 돌려 vector.contract 라는 고수준 단일 op으로 모은다. 후속 lowering(예: outerproduct/매트릭스 unit) 전략이 다르므로 어느 쪽을 원하는지에 따라 vectorizer 를 고른다.

isolated-from-above 요건

B 의 vectorize_children_and_apply_patterns 는 op 자체가 아니라 isolated-from-above 영역(여기선 func.func)을 target 으로 받는다. 그래서 get_parent_op {isolated_from_above} 로 부모 func 을 먼저 끌어와야 한다. A 처럼 op 에 바로 못 건다.

handle 소비 차이

A 의 structured.vectorize 는 result 가 없는 in-place 변환(handle 안 만듦). B 는 새 !transform.any_op handle 을 반환해 이후 chaining 에 쓸 수 있다.

parallel-only 면 reduction 안 생김 (C)

elementwise generic 은 iterator 가 전부 parallel → contraction/reduction 표현 없이 transfer_read·arith.addf <vector>·transfer_write 만. matmul 의 복잡함은 reduction 축에서 온다는 게 대비로 드러난다.

공통 골격은 셋 다 동일하다: tensor 를 vector.transfer_read 로 읽고 → vector 연산으로 계산 → vector.transfer_write 로 다시 tensor 에 쓴다. 이 transfer 쌍이 linalg→vector lowering 의 표준 입출구다.

Q함정/주의는?
피해야 할 것해야 할 것
dynamic shape 에 vectorize 기대vectorize 는 보통 static shape 필요 — vector 폭이 컴파일 타임에 정해져야 한다. 그래서 payload 가 전부 tensor<8x16xf32> 같은 정적 크기다.
vectorize_children_and_apply_patterns 를 op 에 직접 매치이건 isolated-from-above target 이 필요. get_parent_op {isolated_from_above} 로 func.func 를 먼저 얻어 거기에 건다.
"vectorize = 한 가지 결과" 라고 가정vectorizer 마다 도착 IR 이 다르다(A: multi_reduction, B: vector.contract). 후속 패스가 무엇을 기대하는지에 맞춰 선택.
A 의 handle 을 이후 op 에 재사용A(structured.vectorize)는 result 가 없어 chaining 할 handle 이 없다. chaining 이 필요하면 handle 을 반환하는 B 형태를 쓴다.

또 하나: A 의 transfer_read 에 붙는 permutation_map = #map(예: (d0,d1) -> (d0,0,d1))은 2D tensor 를 3D vector <8x4x16> 로 broadcast 하며 읽는 표현이다. matmul 의 contraction 축(16)을 곱셈용 3D 공간으로 펼친 뒤 multi_reduction [2] 로 그 축을 접는 구조 — 읽을 때 차원이 늘어나는 게 의도된 동작이다.

Q한 줄 + 관련

한 줄: vectorize 는 static-shape linalg op 을 transfer_read–vector연산–transfer_write 골격의 vector dialect 로 내리며, structured.vectorize(분해형 multi_reduction) vs vectorize_children_and_apply_patterns(vector.contract 단일형) 두 vectorizer 가 같은 matmul 을 서로 다른 추상화로 떨어뜨린다.

관련 transform op

transform.structured.match(target 잡기) · transform.get_parent_op {isolated_from_above}(부모 func 끌어오기) · transform.structured.vectorize · transform.structured.vectorize_children_and_apply_patterns.

다음 (lowering 후속)

여기서 나온 vector.contract/multi_reduction 을 더 내리는 단계 — vector.outerproduct·vector.fma 로의 unrolling, 그리고 transfer_read/write → 실제 load/store 로의 lowering 이 자연스러운 후속.

대조 (polyhedral baseline)

affine 쪽 vectorization(자동 dependence 분석 기반)과 달리, 여기선 vector 폭·형태가 vectorizer 선택과 op 의미로 결정된다 — schedule IR 이 명시적이라 "어떤 vector op 으로 갈지"를 사람이 고른다.
