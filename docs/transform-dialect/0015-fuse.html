T05 — fuse (producer를 consumer loop nest로)

consumer를 tiling하면서 그 결과 loop nest 안으로 producer의 tile slice를 끌어들인다 — 중간 tensor를 materialize하지 않고 producer·consumer를 한 nest에서 함께 계산하는 transform.structured.fuse / fuse_into_containing_op.

Q이 실험은 뭐를 보여주나?

producer→consumer chain의 fusion을 transform dialect로 표현한다. payload에는 fusion 대상 chain 두 개가 있다.

  • @mm_chain — linalg.elemwise_binary(producer, AA = A + A2) → linalg.matmul(consumer/root, AA * B). producer 결과 %AA가 matmul의 ins로 들어간다.
  • @elem_chain — linalg.elemwise_unary(producer) → linalg.elemwise_binary(consumer/root). 512×512 static shape.

핵심 메커니즘: root(consumer)를 tiling하면 그 결과로 loop nest와 tensor.extract_slice가 생긴다. 그 slice가 요구하는 producer의 입력 영역만큼만 producer를 tiling해서 loop body 안에 복제하면, 중간 full tensor를 만들지 않고 tile 단위로 producer→consumer가 흐른다. 이게 "tile-and-fuse"다.

이 실험은 같은 fusion 효과를 세 가지 schedule로 표현해 대조한다 — (A) one-shot fuse로 matmul root, (B) 저수준 2-step(tile_using_forall + fuse_into_containing_op)으로 같은 matmul, (C) fuse를 elemwise→elemwise chain에. binary는 하나(--transform-interpreter)이고 schedule만 갈아끼운다.

Q어떻게 했나? (schedule)

Variant A — one-shot transform.structured.fuse. matmul을 root로 match하고 tile_sizes = [32, 32]로 tiling하면서 tileable producer(elemwise_binary)를 greedy하게 fuse한다. handle은 fused op + tile loop 2개(%loops:2)를 돌려준다.

transform.named_sequence @__transform_main(%arg0: !transform.any_op {transform.readonly}) {
  %mm = transform.structured.match ops{["linalg.matmul"]} in %arg0
      : (!transform.any_op) -> !transform.any_op
  // tile the two parallel dims (M, N); K dim tile = 0 -> not tiled here.
  %fused, %loops:2 = transform.structured.fuse %mm
      {tile_sizes = [32, 32], tile_interchange = [0, 1]}
      : (!transform.any_op) -> (!transform.any_op, !transform.any_op, !transform.any_op)
  transform.yield
}

Variant B — 저수준 2-step (scf.forall container). fuse가 한 방에 하는 일을 두 op으로 풀어쓴다. (1) tile_using_forall로 matmul root를 scf.forall container로 tiling(M=32, N=16). (2) fuse_into_containing_op로 producer를 그 container 안으로 끌어들인다. container가 scf.for가 아니라 parallel scf.forall인 게 A와의 결정적 차이다.

transform.named_sequence @__transform_main(%arg0: !transform.any_op {transform.readonly}) {
  %prod = transform.structured.match ops{["linalg.elemwise_binary"]} in %arg0
      : (!transform.any_op) -> !transform.any_op
  %root = transform.structured.match ops{["linalg.matmul"]} in %arg0
      : (!transform.any_op) -> !transform.any_op
  // 1) tile root into an scf.forall container (tile M=32, N=16).
  %tiled, %forall = transform.structured.tile_using_forall %root tile_sizes [32, 16]
      : (!transform.any_op) -> (!transform.any_op, !transform.any_op)
  // 2) fuse the elemwise_binary producer into that container.
  %fused, %new_container = transform.structured.fuse_into_containing_op %prod into %forall
      : (!transform.any_op, !transform.any_op) -> (!transform.any_op, !transform.any_op)
  transform.yield
}

Variant C — fuse를 elemwise→elemwise에. consumer(elemwise_binary)를 root로 match해 tile_sizes = [64, 64]로 tiling하고 producer(elemwise_unary)를 fuse한다. upstream @fuse_unary 정규 테스트를 static shape로 옮긴 형태라 IR이 깔끔하다.

transform.named_sequence @__transform_main(%arg0: !transform.any_op {transform.readonly}) {
  %root = transform.structured.match ops{["linalg.elemwise_binary"]} in %arg0
      : (!transform.any_op) -> !transform.any_op
  %fused, %loops:2 = transform.structured.fuse %root
      {tile_sizes = [64, 64], tile_interchange = [0, 1]}
      : (!transform.any_op) -> (!transform.any_op, !transform.any_op, !transform.any_op)
  transform.yield
}

흐름은 모두 match → (root 선정) → tile + producer fuse. A·C는 fuse 한 op이 tiling과 fusion을 동시에 처리하고, B는 그 둘을 tile_using_forall·fuse_into_containing_op 두 handle로 분리한다. 실행: mlir-opt --transform-interpreter --canonicalize.

Q결과가 어떻게 다른가? (before→after 실측)

Variant A — flat한 producer→matmul이 2중 scf.for nest(step 32)로 바뀐다. 주목: loop body 안에 producer(elemwise_binary)가 tile 크기(32×256)로 복제되어 들어가 있고, 바로 그 결과 %2가 같은 nest 안의 matmul ins로 흘러간다. 중간 full tensor<128x256xf32>는 사라졌다 — tile 단위로만 존재한다.

Variant A — fuse (matmul root, scf.for)
before — payload (@mm_chain)
func.func @mm_chain(%A: tensor<128x256xf32>, %A2: tensor<128x256xf32>,
                    %B: tensor<256x64xf32>, %C: tensor<128x64xf32>)
    -> tensor<128x64xf32> {
  // producer: elementwise add  AA = A + A2
  %AA = linalg.elemwise_binary
      ins(%A, %A2 : tensor<128x256xf32>, tensor<128x256xf32>)
      outs(%A : tensor<128x256xf32>) -> tensor<128x256xf32>
  // consumer / root: matmul AA * B
  %0 = linalg.matmul
      ins(%AA, %B : tensor<128x256xf32>, tensor<256x64xf32>)
      outs(%C : tensor<128x64xf32>) -> tensor<128x64xf32>
  return %0 : tensor<128x64xf32>
}
after — out-a-fuse.mlir
func.func @mm_chain(%arg0: tensor<128x256xf32>, %arg1: tensor<128x256xf32>, %arg2: tensor<256x64xf32>, %arg3: tensor<128x64xf32>) -> tensor<128x64xf32> {
  %c64 = arith.constant 64 : index
  %c32 = arith.constant 32 : index
  %c128 = arith.constant 128 : index
  %c0 = arith.constant 0 : index
  %0 = scf.for %arg4 = %c0 to %c128 step %c32 iter_args(%arg5 = %arg3) -> (tensor<128x64xf32>) {
    %1 = scf.for %arg6 = %c0 to %c64 step %c32 iter_args(%arg7 = %arg5) -> (tensor<128x64xf32>) {
      %extracted_slice = tensor.extract_slice %arg0[%arg4, 0] [32, 256] [1, 1] : tensor<128x256xf32> to tensor<32x256xf32>
      %extracted_slice_0 = tensor.extract_slice %arg1[%arg4, 0] [32, 256] [1, 1] : tensor<128x256xf32> to tensor<32x256xf32>
      %extracted_slice_1 = tensor.extract_slice %arg0[%arg4, 0] [32, 256] [1, 1] : tensor<128x256xf32> to tensor<32x256xf32>
      %2 = linalg.elemwise_binary ins(%extracted_slice, %extracted_slice_0 : tensor<32x256xf32>, tensor<32x256xf32>) outs(%extracted_slice_1 : tensor<32x256xf32>) -> tensor<32x256xf32>
      %extracted_slice_2 = tensor.extract_slice %arg2[0, %arg6] [256, 32] [1, 1] : tensor<256x64xf32> to tensor<256x32xf32>
      %extracted_slice_3 = tensor.extract_slice %arg7[%arg4, %arg6] [32, 32] [1, 1] : tensor<128x64xf32> to tensor<32x32xf32>
      %3 = linalg.matmul ins(%2, %extracted_slice_2 : tensor<32x256xf32>, tensor<256x32xf32>) outs(%extracted_slice_3 : tensor<32x32xf32>) -> tensor<32x32xf32>
      %inserted_slice = tensor.insert_slice %3 into %arg7[%arg4, %arg6] [32, 32] [1, 1] : tensor<32x32xf32> into tensor<128x64xf32>
      scf.yield %inserted_slice : tensor<128x64xf32>
    }
    scf.yield %1 : tensor<128x64xf32>
  }
  return %0 : tensor<128x64xf32>
}

Variant B — 같은 chain, 같은 fusion이지만 container가 scf.forall (%arg4, %arg5) in (4, 4)다. tile 좌표는 affine.apply map(d0*16, d0*32)으로 계산되고, 결과 write-back은 tensor.insert_slice + scf.yield가 아니라 scf.forall.in_parallel { tensor.parallel_insert_slice ... }로 나간다. producer는 A와 똑같이 32×256 tile로 body 안에 복제된다 — fusion 결과는 동일하고 container의 병렬 의미만 다르다.

Variant C — elemwise_unary → elemwise_binary chain. 2중 scf.for(step 64) 안에 producer(unary)가 64×64 tile로 복제되고 그 결과 %2가 같은 body의 consumer(binary) ins로 들어간다. matmul처럼 reduction dim이 없어 두 차원 모두 깔끔히 tiling된다.

Variant C — fuse (elemwise→elemwise, scf.for)
before — payload (@elem_chain)
func.func @elem_chain(%arg0: tensor<512x512xf32>, %arg1: tensor<512x512xf32>)
    -> tensor<512x512xf32> {
  // producer: elementwise unary
  %0 = linalg.elemwise_unary ins(%arg0 : tensor<512x512xf32>)
                             outs(%arg1 : tensor<512x512xf32>) -> tensor<512x512xf32>
  // consumer / root: elementwise binary consuming the producer result
  %1 = linalg.elemwise_binary ins(%0, %arg0 : tensor<512x512xf32>, tensor<512x512xf32>)
                              outs(%arg1 : tensor<512x512xf32>) -> tensor<512x512xf32>
  return %1 : tensor<512x512xf32>
}
after — out-c-fuse-elemwise.mlir
func.func @elem_chain(%arg0: tensor<512x512xf32>, %arg1: tensor<512x512xf32>) -> tensor<512x512xf32> {
  %c64 = arith.constant 64 : index
  %c512 = arith.constant 512 : index
  %c0 = arith.constant 0 : index
  %0 = scf.for %arg2 = %c0 to %c512 step %c64 iter_args(%arg3 = %arg1) -> (tensor<512x512xf32>) {
    %1 = scf.for %arg4 = %c0 to %c512 step %c64 iter_args(%arg5 = %arg3) -> (tensor<512x512xf32>) {
      %extracted_slice = tensor.extract_slice %arg0[%arg2, %arg4] [64, 64] [1, 1] : tensor<512x512xf32> to tensor<64x64xf32>
      %extracted_slice_0 = tensor.extract_slice %arg1[%arg2, %arg4] [64, 64] [1, 1] : tensor<512x512xf32> to tensor<64x64xf32>
      %2 = linalg.elemwise_unary ins(%extracted_slice : tensor<64x64xf32>) outs(%extracted_slice_0 : tensor<64x64xf32>) -> tensor<64x64xf32>
      %extracted_slice_1 = tensor.extract_slice %arg0[%arg2, %arg4] [64, 64] [1, 1] : tensor<512x512xf32> to tensor<64x64xf32>
      %extracted_slice_2 = tensor.extract_slice %arg5[%arg2, %arg4] [64, 64] [1, 1] : tensor<512x512xf32> to tensor<64x64xf32>

    scf.yield %1 : tensor<512x512xf32>
  }
  return %0 : tensor<512x512xf32>

Q그래서 핵심은?

fusion은 "consumer를 tiling → 그 slice가 필요로 하는 만큼만 producer를 tiling해 loop body 안에 복제" 다. 어느 schedule이든 중간 full tensor가 tile slice로 쪼개져 nest 안에 들어간다는 본질은 같고, 표현 방식과 container만 다르다.

fuse = tile + producer-fuse 한 방

transform.structured.fuse(A·C)는 root를 tiling하면서 tileable producer를 greedy하게 끌어들인다. handle은 fused op + tile loop들. one-shot이라 schedule이 짧다.

2-step은 분해형

B는 tile_using_forall(container 생성) + fuse_into_containing_op(producer 주입)으로 같은 일을 둘로 나눈다. 어느 producer를 언제 어느 container로 넣을지 명시 제어가 필요할 때.

scf.for vs scf.forall

A는 sequential scf.for + iter_args/insert_slice. B는 parallel scf.forall ... in (4,4) + shared_outs/parallel_insert_slice. fusion 결과 op은 동일, 병렬 의미만 다르다.

producer는 복제된다

fusion은 producer를 "옮기는" 게 아니라 tile 단위로 다시 계산한다. body 안 elemwise는 tile shape(32×256 / 64×64)로 새로 찍힌 인스턴스다 — recompute 비용을 locality와 맞바꾼다.

Q함정/주의는?

handle 개수 = 결과 개수

fuse는 %fused, %loops:2 처럼 결과 type을 정확히 맞춰야 한다. tile_sizes 0이 아닌 차원 수만큼 loop handle이 나온다 — type list 개수가 어긋나면 verify 실패.

matmul의 K dim은 tiling 안 함

A의 tile_sizes = [32, 32]는 M·N(parallel)만 자른다. matmul의 K(reduction, 256)는 0이라 통째로 남는다 — 그래서 body의 producer slice가 32x256(K 전체)으로 나온다. fuse가 reduction을 자동으로 자르지는 않는다.

B의 2-step은 순서가 강제

fuse_into_containing_op은 먼저 만들어진 %forall container handle을 받아야 한다. tile_using_forall → fuse_into 순서가 뒤집히면 넣을 container가 없다. handle 의존성이 곧 schedule 순서.

producer recompute 인지

B의 IR에 affine.apply가 중복(#map1(%arg4) 여러 번)으로 보이는 건 fusion이 producer slice 좌표를 다시 계산하기 때문. --canonicalize가 일부 정리하지만 의미상 producer는 tile마다 새로 도는 op임을 잊지 말 것.

Q한 줄 + 관련

한 줄 — fusion = "consumer를 tiling하고, 그 tile slice가 요구하는 만큼만 producer를 loop body 안에 복제(recompute)" 다. fuse는 이걸 한 방에(scf.for), tile_using_forall+fuse_into_containing_op는 두 step으로(scf.forall) 표현한다 — 효과는 같고 container의 병렬 의미가 다르다.

관련 — T03/0011 tile_first(fuse의 전제인 tiling 자체), T04/0012 pass vs transform script(같은 효과를 pass와 schedule로). affine 쪽 baseline은 polyhedral loop fusion(dependence·legality 자동 판정) — 거기선 ILP가 fusion 가능성을 결정하지만, 여기선 schedule IR에 무엇을·어느 container로 명시한다.

다음 — fuse 결과를 --canonicalize 너머로 더 내려(scf.forall → parallel lowering, vectorize) full pipeline에서의 효과 관찰. producer recompute가 실제 비용/이득으로 어떻게 나타나는지.
