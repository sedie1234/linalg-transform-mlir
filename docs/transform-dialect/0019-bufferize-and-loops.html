T09 — bufferize & to-loops

하나의 matmul을 두고 세 schedule을 돌려, tensor→memref bufferize와 buffer-semantic matmul→scf.for nest로의 lowering을 실측한다. 두 변환의 전제·경계 효과를 분리해서 본다.

Q이 실험은 뭘 보여주나?

linalg를 더 낮은 추상으로 내릴 때 거쳐야 하는 두 단계 — bufferization(tensor → memref)과 to-loops(structured op → 명시적 scf.for nest) — 를 transform dialect op으로 직접 호출해 그 산출 IR을 본다.

왜 두 단계를 함께 보나? to-loops가 bufferization에 의존하기 때문이다. transform.structured.convert_to_loops는 TilingInterface의 generateScalarImplementation hook으로 scalar 본문(load/mulf/addf/store)을 찍는데, 이 hook은 피연산자가 buffer(memref) semantic일 때만 동작한다. tensor SSA 값에는 memref.load를 찍을 위치가 없다. 그래서:

A — bufferize만

bufferize_to_allocation로 accumulator(outs)를 명시적 memref.alloc으로 내린다. matmul은 그대로 tensor. loop 없음.

B — to-loops만

이미 memref-semantic인 matmul에 convert_to_loops. 3중 scf.for nest + scalar 본문. bufferize 불필요(이미 buffer).

C — A + bufferize + B

tensor matmul에 alloc → one_shot_bufferize(나머지 tensor를 전부 memref로) → convert_to_loops. 전체 파이프라인.

payload(tensor matmul, 128x256 · 256x64 → 128x64):

func.func @matmul(%A: tensor<128x256xf32>, %B: tensor<256x64xf32>,
                  %C: tensor<128x64xf32>) -> tensor<128x64xf32> {
  %0 = linalg.matmul ins(%A, %B : tensor<128x256xf32>, tensor<256x64xf32>)
                     outs(%C : tensor<128x64xf32>) -> tensor<128x64xf32>
  return %0 : tensor<128x64xf32>
}

B만은 tensor matmul로는 lowering이 안 되므로, memref-semantic 변형(matmul_buf, 결과 없고 outs가 in-place memref)을 payload로 따로 쓴다. payload.mlir의 주석이 이 점을 명시한다.

Q어떻게 했나? (schedule)

세 schedule 모두 match ops{["linalg.matmul"]}로 handle을 잡고 그 위에 op을 얹는다. 핵심만 escape해서 본다.

A — bufferize_to_allocation (accumulator만 alloc, matmul은 tensor 유지):

transform.named_sequence @__transform_main(%arg0: !transform.any_op {transform.readonly}) {
  %mm = transform.structured.match ops{["linalg.matmul"]} in %arg0 : (!transform.any_op) -> !transform.any_op
  %buf, %new = transform.structured.bufferize_to_allocation %mm
      {memory_space = 0, bufferize_destination_only, emit_dealloc} : !transform.any_op
  transform.yield
}

bufferize_destination_only = outs(destination) operand만 buffer로 내리고 ins/matmul은 건드리지 않는다. emit_dealloc = 짝이 되는 memref.dealloc도 같이 찍는다. memory_space = 0 = 기본 메모리 공간.

B — convert_to_loops (memref matmul → scf.for nest). handle은 readonly, op은 단 하나:

  %mm = transform.structured.match ops{["linalg.matmul"]} in %arg0 : (!transform.any_op) -> !transform.any_op
  %loops = transform.structured.convert_to_loops %mm : (!transform.any_op) -> (!transform.any_op)

C — A + one_shot_bufferize + B. 여기서 %arg0가 transform.consumed인 점이 중요하다 — one_shot_bufferize가 module을 in-place로 다시 쓰기 때문에 입력 handle이 소모된다:

transform.named_sequence @__transform_main(%arg0: !transform.any_op {transform.consumed}) {
  %mm = transform.structured.match ops{["linalg.matmul"]} in %arg0 : (!transform.any_op) -> !transform.any_op
  %buf, %new = transform.structured.bufferize_to_allocation %mm
      {memory_space = 0, bufferize_destination_only, emit_dealloc} : !transform.any_op
  %bufmod = transform.bufferization.one_shot_bufferize %arg0 : (!transform.any_op) -> !transform.any_op
  %mm2 = transform.structured.match ops{["linalg.matmul"]} in %bufmod : (!transform.any_op) -> !transform.any_op
  %loops = transform.structured.convert_to_loops %mm2 : (!transform.any_op) -> (!transform.any_op)
  transform.yield
}

C의 흐름: match → alloc(dest) → one_shot_bufferize(module) → re-match(bufferize 후의 새 matmul을 다시 잡아야 함, 이전 handle은 stale) → convert_to_loops. one_shot_bufferize가 IR을 갈아엎으므로 그 뒤엔 반드시 다시 match 한다.

Q결과가 어떻게 다른가? (before→after 실측)

A — matmul은 그대로 tensor지만, outs operand만 명시적 alloc으로 내려가고 materialize_in_destination / to_tensor / dealloc이 둘러싼다. loop은 안 생긴다.

A — bufferize_to_allocation (destination only)
before — tensor matmul
func.func @matmul(%A: tensor<128x256xf32>, %B: tensor<256x64xf32>,
                  %C: tensor<128x64xf32>) -> tensor<128x64xf32> {
  %0 = linalg.matmul ins(%A, %B : tensor<128x256xf32>, tensor<256x64xf32>)
                     outs(%C : tensor<128x64xf32>) -> tensor<128x64xf32>
  return %0 : tensor<128x64xf32>
}
after — outs만 memref.alloc, matmul은 tensor
func.func @matmul(%arg0: tensor<128x256xf32>, %arg1: tensor<256x64xf32>, %arg2: tensor<128x64xf32>) -> tensor<128x64xf32> {
  %alloc = memref.alloc() : memref<128x64xf32>
  bufferization.materialize_in_destination %arg2 in writable %alloc : (tensor<128x64xf32>, memref<128x64xf32>) -> ()
  %0 = bufferization.to_tensor %alloc restrict writable : memref<128x64xf32>
  %1 = linalg.matmul ins(%arg0, %arg1 : tensor<128x256xf32>, tensor<256x64xf32>) outs(%0 : tensor<128x64xf32>) -> tensor<128x64xf32>
  memref.dealloc %alloc : memref<128x64xf32>
  return %1 : tensor<128x64xf32>
}

B — memref matmul 하나가 3중 scf.for nest(d0=128, d1=64, d2=256)로 풀린다. 본문은 affine.apply로 index를 뽑고 memref.load → mulf → addf → memref.store. #map/#map1/#map2는 (d0,d1,d2) → (d0)/(d2)/(d1) projection — matmul의 A[i,k]·B[k,j]·C[i,j] 접근을 그대로 표현한다.

B — convert_to_loops (memref matmul → scf.for nest)
before — memref-semantic matmul (결과 없음, in-place)
func.func @matmul_buf(%A: memref<128x256xf32>, %B: memref<256x64xf32>, %C: memref<128x64xf32>) {
  linalg.matmul ins(%A, %B : memref<128x256xf32>, memref<256x64xf32>) outs(%C : memref<128x64xf32>)
  return
}
after — 3중 scf.for + scalar 본문
#map = affine_map<(d0, d1, d2) -> (d0)>
#map1 = affine_map<(d0, d1, d2) -> (d2)>
#map2 = affine_map<(d0, d1, d2) -> (d1)>
func.func @matmul_buf(%arg0: memref<128x256xf32>, %arg1: memref<256x64xf32>, %arg2: memref<128x64xf32>) {
  %c0 = arith.constant 0 : index
  %c128 = arith.constant 128 : index
  %c1 = arith.constant 1 : index
  scf.for %arg3 = %c0 to %c128 step %c1 {
    %c0_0 = arith.constant 0 : index
    %c64 = arith.constant 64 : index
    %c1_1 = arith.constant 1 : index
    scf.for %arg4 = %c0_0 to %c64 step %c1_1 {
      %c0_2 = arith.constant 0 : index
      %c256 = arith.constant 256 : index
      %c1_3 = arith.constant 1 : index
      scf.for %arg5 = %c0_2 to %c256 step %c1_3 {
        %0 = affine.apply #map(%arg3, %arg4, %arg5)
        %1 = affine.apply #map1(%arg3, %arg4, %arg5)
        %2 = memref.load %arg0[%0, %1] : memref<128x256xf32>
        %3 = affine.apply #map1(%arg3, %arg4, %arg5)
        %4 = affine.apply #map2(%arg3, %arg4, %arg5)
        %5 = memref.load %arg1[%3, %4] : memref<256x64xf32>
        %6 = affine.apply #map(%arg3, %arg4, %arg5)
        %7 = affine.apply #map2(%arg3, %arg4, %arg5)
        %8 = memref.load %arg2[%6, %7] : memref<128x64xf32>
        %9 = arith.mulf %2, %5 : f32
        %10 = arith.addf %8, %9 : f32
        %11 = affine.apply #map(%arg3, %arg4, %arg5)
        %12 = affine.apply #map2(%arg3, %arg4, %arg5)
        memref.store %10, %arg2[%11, %12] : memref<128x64xf32>
      }
    }
  }
  return
}

C — tensor 입력에서 시작해 A의 alloc + one_shot_bufferize(나머지 operand를 to_memref로 내림) + B의 loop nest가 한 번에 나온다. 경계엔 to_memref(입구) / to_tensor + dealloc(출구)이 남아 tensor 인터페이스를 유지한다. accumulator는 alloc + memref.copy로 초기화되고, loop 본문은 alloc된 buffer에 in-place로 누적한다.

C — full pipeline (tensor → memref → loops)
before — tensor matmul (A와 동일 payload)
func.func @matmul(%A: tensor<128x256xf32>, %B: tensor<256x64xf32>,
                  %C: tensor<128x64xf32>) -> tensor<128x64xf32> {
  %0 = linalg.matmul ins(%A, %B : tensor<128x256xf32>, tensor<256x64xf32>)
                     outs(%C : tensor<128x64xf32>) -> tensor<128x64xf32>
  return %0 : tensor<128x64xf32>
}
after — to_memref + alloc/copy + scf.for nest + to_tensor/dealloc
#map = affine_map<(d0, d1, d2) -> (d0)>
#map1 = affine_map<(d0, d1, d2) -> (d2)>
#map2 = affine_map<(d0, d1, d2) -> (d1)>
func.func @matmul(%arg0: tensor<128x256xf32>, %arg1: tensor<256x64xf32>, %arg2: tensor<128x64xf32>) -> tensor<128x64xf32> {
  %0 = bufferization.to_memref %arg1 : memref<256x64xf32, strided<[?, ?], offset: ?>>
  %1 = bufferization.to_memref %arg0 : memref<128x256xf32, strided<[?, ?], offset: ?>>
  %2 = bufferization.to_memref %arg2 : memref<128x64xf32, strided<[?, ?], offset: ?>>
  %alloc = memref.alloc() : memref<128x64xf32>
  memref.copy %2, %alloc : memref<128x64xf32, strided<[?, ?], offset: ?>> to memref<128x64xf32>
  %c0 = arith.constant 0 : index
  %c128 = arith.constant 128 : index
  %c1 = arith.constant 1 : index
  scf.for %arg3 = %c0 to %c128 step %c1 {
    %c0_0 = arith.constant 0 : index
    %c64 = arith.constant 64 : index
    %c1_1 = arith.constant 1 : index
    scf.for %arg4 = %c0_0 to %c64 step %c1_1 {
      %c0_2 = arith.constant 0 : index
      %c256 = arith.constant 256 : index
      %c1_3 = arith.constant 1 : index
      scf.for %arg5 = %c0_2 to %c256 step %c1_3 {
        %4 = affine.apply #map(%arg3, %arg4, %arg5)
        %5 = affine.apply #map1(%arg3, %arg4, %arg5)
        %6 = memref.load %1[%4, %5] : memref<128x256xf32, strided<[?, ?], offset: ?>>
        %7 = affine.apply #map1(%arg3, %arg4, %arg5)
        %8 = affine.apply #map2(%arg3, %arg4, %arg5)
        %9 = memref.load %0[%7, %8] : memref<256x64xf32, strided<[?, ?], offset: ?>>
        %10 = affine.apply #map(%arg3, %arg4, %arg5)
        %11 = affine.apply #map2(%arg3, %arg4, %arg5)
        %12 = memref.load %alloc[%10, %11] : memref<128x64xf32>
        %13 = arith.mulf %6, %9 : f32
        %14 = arith.addf %12, %13 : f32
        %15 = affine.apply #map(%arg3, %arg4, %arg5)
        %16 = affine.apply #map2(%arg3, %arg4, %arg5)
        memref.store %14, %alloc[%15, %16] : memref<128x64xf32>
      }
    }
  }
  %3 = bufferization.to_tensor %alloc : memref<128x64xf32>
  memref.dealloc %alloc : memref<128x64xf32>
  return %3 : tensor<128x64xf32>
}

run.sh 검증 출력: A memref.alloc=1 B scf.for=3 C scf.for=3 memref.alloc=1 — A는 alloc만(loop 0), B/C는 3중 nest, C는 alloc까지 동반.

Q그래서 핵심은?

두 변환은 직교(orthogonal)지만 방향 의존이 있다 — to-loops는 bufferize를 전제로 한다.

bufferize ⟂ to-loops

bufferize는 데이터 표현(tensor↔memref)을 바꾸고, to-loops는 제어 구조(structured op→explicit loop)를 바꾼다. A는 전자만, B는 후자만 한 것.

convert_to_loops는 memref 필수

generateScalarImplementation hook이 memref.load/store를 찍으므로 피연산자가 buffer여야 한다. tensor matmul엔 직접 못 쓴다(그래서 B는 memref payload, C는 먼저 bufferize).

destination_only의 범위

A의 bufferize_destination_only는 outs operand 하나만 alloc하고 matmul은 tensor로 둔다 — 부분 bufferize. 전체를 내리려면 C처럼 one_shot_bufferize를 더한다.

consumed handle + re-match

C에서 one_shot_bufferize가 module을 in-place로 갈아엎어 %arg0가 consumed. 그 뒤 matmul handle은 stale이므로 새 module에서 다시 match해야 to-loops를 걸 수 있다.

scalar 본문이 affine.apply #map(...)로 index를 매번 다시 계산하는 점도 눈여겨볼 만하다 — #map/#map1/#map2는 각각 (d0,d1,d2)에서 d0/d2/d1을 뽑는 projection으로, matmul의 표준 indexing map(A는 (m,k), B는 (k,n), C는 (m,n))을 그대로 풀어 쓴 것이다. 이 redundant한 apply들은 이후 canonicalize/CSE에서 정리된다.

Q함정/주의는?
함정대응
tensor matmul에 convert_to_loops 직접 호출먼저 bufferize (B는 memref payload로, C는 one_shot_bufferize 후 호출)
one_shot_bufferize 후 옛 matmul handle 재사용bufferize 결과 module에서 다시 match (C의 %mm2)
%arg0를 readonly로 둔 채 one_shot_bufferizemodule을 in-place 재작성하므로 handle을 transform.consumed로 선언
C 결과의 to_memref가 strided<[?,?],offset:?> 타입정상 — 함수 인자 tensor를 buffer view로 본 dynamic stride. loop은 그 위에서 load/store

B와 C의 loop 본문은 거의 동일하지만, B는 input/output이 모두 정적 memref<...>인 반면 C는 입력이 strided<[?,?],offset:?> view다(함수 boundary tensor에서 온 것). accumulator만 C에서 fresh memref.alloc(정적)이라 in-place 누적된다.

Q한 줄 + 관련

한 줄: bufferize는 tensor→memref(데이터 표현)를, convert_to_loops는 structured op→scf.for nest(제어 구조)를 내린다 — 후자는 memref 피연산자를 전제로 하며, one_shot_bufferize로 module을 갈아엎은 뒤엔 handle을 consumed로 선언하고 re-match 해야 한다.

관련 (P1 cycle)

affine #0023 linalg ad-hoc lowering 산출 IR과의 대조 — polyhedral 측은 affine.for를 직접 생성하지만, 여기선 transform op이 scf.for + affine.apply index로 내린다.
