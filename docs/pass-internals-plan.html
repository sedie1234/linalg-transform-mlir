Linalg in-tree pass 내부구조 학습 계획

LLVM 19.1.7의 linalg in-tree pass 12종 — pass 내부 코드가 어떻게 구성되어 있고(pass → pattern → 핵심 함수 체인), 적용하면 MLIR이 어떻게 변하는지를 해부한다. 각 pass의 핵심 함수를 out-of-tree에서 가져와 재현해, 개인 컴파일러에 pass를 이식할 수 있는 수준의 이해를 목표로 한다.

날짜: 2026-06-12 (재시작) 대상: mlir/include/mlir/Dialect/Linalg/Passes.td의 pass 12종 전부 cycle: 10 (Loops.cpp 3 pass = 1 cycle) 드라이버: out-of-tree/build/bin/my-mlir-opt (LLVM 19.1.7 재사용, green)

1. 목적

추후 개인 컴파일러에서 linalg pass를 적절하게 활용하기 위해, pass 단위로 두 가지를 파악한다:

  • 내부 코드 구성 — Passes.td 정의 → tablegen이 생성하는 base class → 구현 .cpp의 pass class → runOnOperation() → 어떤 populate*Patterns를 채우는가 → 각 pattern의 matchAndRewrite가 부르는 핵심 함수 — 이 호출 체인을 파일·라인 단위로 문서화. driver 종류(greedy / dialect conversion / walk)와 옵션이 코드 어디로 흘러가는지 포함.
  • IR 변화 — 대표 입력 2~3개에 pass를 적용한 before/after IR과, 내부 코드의 어느 단계가 어떤 IR 변화를 만드는지의 매핑.

핵심 제약

본인 pass는 in-tree pass가 쓰는 그 함수들을 out-of-tree에서 가져와(populate*/핵심 transform 함수) 호출해야 한다 — 알고리즘 재구현 금지. in-tree pass와의 byte-diff가 "내 컴파일러에 그대로 이식 가능"의 증명이 된다.

3. pass 전수 목록 & cycle 배정 (Passes.td 진실 소스)

19.1.7 Linalg/Passes.td에 정의된 pass는 정확히 12종. 구현 파일을 공유하는 lowering 3종은 한 cycle로 묶는다 (같은 내부 기계의 세 instantiation).

cyclepass (CLI 이름)구현 파일해부 포인트 (예상)
0001linalg-generalize-named-opsGeneralization.cpp가장 단순한 패턴 pass: LinalgGeneralizationPattern 1개 + greedy driver → generalizeNamedOp. pattern pass의 최소 해부 표본
0002linalg-specialize-generic-opsSpecialize.cpp역방향(raise). idiom 인식 분기(isaCopyOpInterface 등) → specializeGenericOp
0003linalg-inline-scalar-operandsInlineScalarOperands.cppscalar(rank-0/1) operand를 body 안 상수 접근으로 끌어들이는 단일 pattern. indexing_maps 재작성 표본
0004linalg-named-op-conversionNamedOpConversions.cppnamed↔named 정규화 (예: depthwise conv multiplier=1 단순화). named op 간 변환 패턴 구조
0005convert-elementwise-to-linalgElementwiseToLinalg.cpparith/math elementwise → linalg.generic *진입* 변환. ConversionTarget+applyPartialConversion인지 greedy인지 — driver 차이 해부
0006linalg-fuse-elementwise-opsElementwiseOpFusion.cpp대형 pass: populateElementwiseOpsFusionPatterns의 패턴 군(fusion + reshape 전파 + 상수 fold), controlFn 훅, fuseElementwiseOps 본체
0007linalg-fold-unit-extent-dimsDropUnitDims.cpprank 축소. populateFoldUnitExtentDimsPatterns, reshapes vs slices 두 전략 옵션이 패턴 선택을 어떻게 가르는지
0008linalg-detensorizeDetensorize.cpp유일한 InterfacePass + type conversion 기계( DialectConversion, cost model로 어느 generic을 scalar로 풀지) — 이전 cycle에서 다루지 않은 신규
0009convert-linalg-to-loops / -affine-loops / -parallel-loopsLoops.cpp (공유)한 template 기계(linalgOpToLoopsImpl<LoopTy>)의 세 instantiation. indexing_maps→load/store index, payload→body의 lowering 본체
0010linalg-block-pack-matmulBlockPackMatmul.cpp최다 옵션(7개) pass: 옵션→ControlBlockPackMatmulFn→blockPackMatmul 흐름, data-layout 변환의 패턴 구성

이전 cycle의 sweep(11종)에 없던 linalg-detensorize 포함 — Passes.td 전수 기준으로 누락 0.
